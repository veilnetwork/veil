# Как работает veil-сеть

Обзорная экскурсия по тому, как Veil устроена изнутри, — для инженеров,
которым нужна общая картина системы, прежде чем открывать исходники.

Хотите потом копнуть глубже? На это есть три справочника:
[ARCHITECTURE_FULL.md](ARCHITECTURE_FULL.md) проходит всю систему
целиком, [NETWORK.md](NETWORK.md) разбирает, как движутся данные, а
[WIRE_PROTOCOL.md](WIRE_PROTOCOL.md) расписывает точный формат байтов на
проводе. English: [HOW_IT_WORKS.md](../en/HOW_IT_WORKS.md).

---

## 1. Что это

Veil — это одноранговая сеть: каждый участник равноправен, никакого
центрального сервера. Вместе узлы образуют **DHT** — общий распределённый
справочник адресов (здесь — на основе Kademlia), по которому они находят
друг друга. Узлы обмениваются шифрованными сообщениями, сами пробивают
домашние роутеры и кладут сообщение в **почтовый ящик** (mailbox), если
получатель не в сети, — там оно дожидается, пока тот не вернётся.
Шифрование работает «от двери до двери» и устойчиво к квантовому
компьютеру: прочитать сообщение могут только две его стороны, и
математика выдержит даже будущий квантовый компьютер (ML-KEM-768 + AEAD).
Узлы бывают всего двух видов:

- **Leaf** — телефоны, IoT, лёгкие клиенты. Лист не держит DHT, ничего
  не ретранслирует и не хранит почтовый ящик. Он просто подключается к
  одному или нескольким Core-узлам — либо через соседей, перечисленных в
  конфигурации, либо через другие узлы, найденные в локальной сети.
- **Core** — полноценный участник. Держит «корзину» Kademlia на K=20
  узлов (свою часть справочника адресов), ретранслирует чужой трафик,
  хранит почтовые ящики и может работать шлюзом, который держит записи,
  указывающие обратно на прикреплённые листы.

```
              ┌──────────────────────────────────────┐
              │           CORE VEIL                  │
              │                                      │
              │   Core ─── Core ─── Core ─── Core    │
              │     │      ╱  ╲      │       │       │
              │     │    ╱     ╲     │       │       │
              │   Core ─ Core ─ Core ─ Core          │
              │    DHT (Kademlia, K=20)              │
              └────┬──────────────────────────┬──────┘
                   │                          │
              ┌────┴────┐                ┌────┴────┐
              │  Leaf   │                │  Leaf   │
              │ (phone) │                │ (phone) │
              └─────────┘                └─────────┘
                  │                          │
              ┌───┴───┐                  ┌───┴───┐
              │  App  │                  │  App  │
              └───────┘                  └───────┘
```

Когда лист прикрепляется к Core-узлу, эта связь записывается через
Discovery (`AttachmentPayload`), чтобы остальная сеть знала, как
проложить ответ обратно к листу.

## 2. Стек: слои внутри узла

```
┌──────────────────────────────────────────────────────┐
│   APP                                                │
│   ├─ IPC client (Unix / NamedPipe / TCP loopback)    │
│   └─ Veil client library (veilclient)                │
└────────────────────────┬─────────────────────────────┘
                         │ IPC frames
┌────────────────────────┴─────────────────────────────┐
│   APPLICATION LAYER                                  │
│   ├─ AppEndpointRegistry  (endpoint mailbox channels)│
│   ├─ AppStreamTable       (stream FSM, windowing)    │
│   └─ IPC server           (auth, capability gates)   │
├──────────────────────────────────────────────────────┤
│   DISPATCH LAYER                                     │
│   FrameDispatcher — pure-sync family switch:         │
│   Session, Control, Discovery, Delivery, Routing,    │
│   App, Mesh, PeerExchange, Tunnel, RelayChain, Diag  │
├──────────────────────────────────────────────────────┤
│   SESSION LAYER                                      │
│   ├─ SessionRunner (одна на peer; AEAD + WRR sched)  │
│   ├─ Handshake FSM (Hello→Identity→Caps→KEX→Confirm) │
│   ├─ Keepalive / Rekey / Hot-standby                 │
│   └─ Session TX registry (lock-free fan-out)         │
├──────────────────────────────────────────────────────┤
│   ROUTING / DHT                                      │
│   ├─ KademliaService (K=20, iterative lookup)        │
│   ├─ RouteCache (TTL, multi-path scoring)            │
│   ├─ Discovery (Attachment, AppEndpoint, MailboxRef) │
│   └─ MeshForwarder (UDP beacon, gateway bridge)      │
├──────────────────────────────────────────────────────┤
│   TRANSPORT                                          │
│   TCP / TLS / QUIC / WebSocket (ws,wss) / Unix       │
│   ─ pluggable; per-listener + per-peer overrides ─   │
└──────────────────────────────────────────────────────┘
```

Одно правило делает этот стек простым для понимания: каждый слой
работает синхронно, если только он явно не помечен как асинхронный.
Диспетчер (часть, которая смотрит на кадр и решает, что с ним делать;
по-английски dispatcher) просто возвращает `DispatchResult` — одно из
значений `Response`, `NoResponse`, `Violation` или `RateLimited` — и
ничего не дожидается. Весь реальный ввод-вывод происходит в задачах
`tokio` (это лёгкие фоновые потоки) — слоем выше или ниже.

## 3. Identity

```
keygen(Ed25519 | Falcon-512)  →  (public_key, private_key)
mine(pk, difficulty=24 bits)  →  nonce
node_id                       =  BLAKE3(public_key)
identity_proof                =  (pk, nonce, sign(pk, nonce))
```

- `node_id` — плоское 256-битное имя. Здесь нет PKI (инфраструктуры
  открытых ключей) и нет доменных имён: личность узла — это просто его
  ключ, ничего не нужно регистрировать и некому доверять.
- Узел подписывает одним из двух алгоритмов: **Ed25519** (по умолчанию,
  быстрый) или **Falcon-512** (пост-квантовый, ключи больше). Выбор
  делается отдельно для каждого узла через `[identity] algo`. В любом
  случае BLAKE3 (хеш-функция) сворачивает открытый ключ к одному и тому
  же 32-байтному node_id.
- Доказательство работы (Proof of Work) — небольшая задачка, из-за
  которой штамповать поддельные личности стоит реального процессорного
  времени. Сложность стартует с 24 бит (16 в отладочных сборках) и
  растёт вместе с сетью: `24 + ceil(log2(N / 100K))`, где N — число
  узлов, которое DHT отслеживает за эпоху.

Личность может быть и **суверенной**: один главный ключ Falcon-512
подписывает повседневные ключи Ed25519 ваших отдельных устройств.
Именно это позволяет мессенджеру работать сразу на нескольких
устройствах и отзывать любое из них по отдельности. См.
[identity-model.md](identity-model.md).

## 4. Сессии: рукопожатие → AEAD-кадры

Прежде чем два узла смогут общаться, они обмениваются рукопожатием.
Рукопожатие (handshake) — это короткий обмен репликами, в котором они
подтверждают, кто они такие, и договариваются о ключах для всего
дальнейшего. Рукопожатие Veil занимает шесть оборотов (запрос-ответ
туда и обратно), и каждый шаг едет внутри кадра OVL1:

```
   Client                                    Server
     │                                         │
     │ ──Hello(OVL1, v1, node_id)──→           │
     │           ←──Hello──                    │
     │ ──Identity(algo, pk, nonce, mlkem_ek?)→ │
     │           ←──Identity──                 │
     │ ──Capabilities(role_bits, flags)──→     │
     │           ←──Capabilities──             │
     │ ──KeyAgreement(X25519 ephemeral pk)──→  │
     │           ←──KeyAgreement──             │
     │   [HKDF-SHA256 → tx_key, rx_key,        │
     │                  session_id]            │
     │ ──SessionConfirm(session_id, HMAC)──→   │
     │           ←──SessionConfirm──           │
     │                                         │
     │  ... все последующие кадры AEAD'd с     │
     │      ChaCha20-Poly1305                  │
```

Как только проходит `SessionConfirm`, сессия установлена, и всё после
неё шифруется:
- Каждый кадр уходит как `header || ciphertext`, где `ciphertext =
  ChaCha20-Poly1305(key, nonce=session_id||counter, plaintext,
  AAD=header)`. (AEAD просто означает, что шифр одновременно прячет
  содержимое и замечает любую подмену.)
- Ключи живут не вечно. Смена ключей запускается после **128 GiB**
  кадров, через **32 дня** или когда счётчик AEAD переполнится — что
  наступит раньше.
- Кадры-заполнители (`SessionMsg::Padding`) добивают каждую запись на
  проводе до размера MTU, чтобы тот, кто просто наблюдает за трафиком,
  не мог считать с него длины сообщений.

### Hot-standby

Сессия может на ходу переехать с одного транспорта на другой — с TCP на
TLS, с IPv4 на IPv6, с порта на порт — не повторяя рукопожатие.
Состояние AEAD переносится, а пишущая задача просто подменяет сокет на
границе кадров, так что подмены никто не замечает. См.
[hot-standby.md](hot-standby.md).

## 5. Маршрутизация: как сообщения находят получателя

Маршрутизация — это задача доставить сообщение нужному узлу, когда вы
знаете только его адрес, но не то, где он находится. Veil опирается на
три механизма, которые работают вместе: быстрый локальный кеш, поиск по
DHT на весь мир, когда в кеше промах, и путь, заданный самим
отправителем, когда дорога уже известна заранее:

### 5.1 Кеш маршрутов (локальная молва)

```
A анонсирует route к D → B (TTL=2) → C (TTL=1, re-announce только
                                        к прямо-подключенным) → STOP
```

`ROUTE_ANNOUNCE` отправляется с TTL (временем жизни), равным 2, поэтому
популярный маршрут расходится ровно на два перехода и не дальше. Записи
в кеше устаревают сами (по умолчанию через 60 с). Каждый маршрут
получает оценку по тому, как он себя ведёт, — время до получателя и
обратно, дрожание задержки, перегрузка и заряд батареи у соседа, — а вес
каждого фактора можно настроить в `[routing]`. Veil хранит к каждому
получателю не один, а несколько лучших путей, чтобы распределять между
ними нагрузку и переключаться на запасной, если один отвалится.

### 5.2 Kademlia DHT (промах кеша)

```
Sender A хочет дотянуться до D, нет cached route:

   A находит N3 closest к node_id(D) в своём bucket'е
   A отправляет RecursiveRelay(dst=D, payload) → N3

   N3: есть ли у меня прямая session к D?
       да → forward через session; готово
       нет → найти closest к D от N3, forward → N3'
       ...
   После ≤ 16 hop'ов, либо mailbox fallback если D offline.
```

Каждый переход примерно вдвое сокращает расстояние до цели, поэтому
сообщение доходит за **O(log N)** переходов — их число растёт лишь
медленно по мере роста сети. И каждая успешная доставка кладёт в кеш
запись об обратном пути (reverse-path), так что следующее сообщение в ту
же сторону вообще пропускает обход DHT.

### 5.3 Маршрутизация от источника (путь задаёт отправитель)

Иногда отправитель уже знает точную цепочку ретрансляторов: скажем,
оператор выдал ему доверенный список, или некий инструмент проверяет
связность. Тогда он отправляет кадр `DeliveryMsg::RelayPath`, упаковав
всю цепочку внутрь. Каждый переход просто передаёт сообщение следующему
имени в списке — никаких обращений к DHT, ничего не читается из кеша.
Один кадр вмещает до 64 переходов.

```
A → RelayPath{path=[B,C,D,E,F], next_hop=0, inner=msg}
B принимает, видит path[0]=self, forward к C с next_hop=1
C → D → E → F (terminal): F декодирует inner и доставляет локально
```

Это удобно, чтобы дотянуться через неудобные конфигурации сети, с
которыми DHT справляется плохо; для цепочек ретрансляторов, которые
должны оставаться неизменными; и для отладки — чтобы проверить, дотянутся
ли два узла друг до друга вообще.

### 5.4 Запасной путь через почтовый ящик

Если сообщение не получается передать вживую — получатель спит или
исчерпан лимит переходов, — оно вместо этого ложится в почтовый ящик,
скопированное на небольшой набор узлов, чтобы пережить простой:

```
sender → STORE(MailboxRef.put(content_id, payload), 3 replicas)
recipient (на wake): FETCH(MailboxRef.list(my_node_id))
              → FETCH(content_id, ack)
```

Какие узлы держат конкретный ящик, определяет `BLAKE3(node_id)` —
сообщение раскладывается на 3 реплики. Каждая пишет его в журнал
упреждающей записи (WAL), чтобы сбой ничего не потерял, и стирает только
после того, как получатель подтвердит, что оно дошло.

## 6. Сквозное шифрование

Veil шифрует в **двух** отдельных слоях, и у каждого своя работа:

| Слой | Алгоритм | Область | Назначение |
|------|----------|---------|------------|
| Session | X25519 ephemeral + HKDF + ChaCha20-Poly1305 | per-hop | Шифрование на проводе между соседними узлами |
| E2E | ML-KEM-768 + ChaCha20-Poly1305 | sender ↔ recipient | Содержимое непрозрачно для ретрансляторов |

Сессионные ключи обновляются при каждом переподключении. Сквозной слой
запечатывает сообщение **опубликованным** ключом ML-KEM-768 получателя —
найденным в DHT или прицепленным к сессии, — поэтому ретрансляторы
посередине не прочитают содержимое, сколько бы их ни сговорилось. Маркеры
`0xE2` и `0xE3` помечают запечатанный сквозным шифрованием конверт внутри
полезной нагрузки `Forward`.

## 7. Прикладной слой

Приложение общается с узлом — который работает фоновым демоном — через
IPC, локальный канал между двумя программами на одной машине (Unix-сокет,
именованный канал Windows или TCP через петлевой интерфейс). Отправить
можно двумя способами:

- **AppSend** — датаграмма по принципу «отправил и забыл», нацеленная на
  удалённую тройку `(node_id, app_id, endpoint_id)`. Вы отправляете её и
  идёте дальше; сеть старается доставить как может.
- **Stream** — надёжный упорядоченный поток поверх veil, с контролем
  потока, чтобы быстрый отправитель не захлестнул медленного читателя.
  `AppStreamTable` демона хранит состояние каждого открытого потока.

Приложение подтверждает, кто оно, своим `app_id` — 32-байтным
дескриптором, который ему выдают при регистрации. IPC-сервер решает, что
каждому приложению дозволено: клиент в режиме Leaf не может, например,
попросить ретранслировать транзитный трафик.

## 8. Протокол на проводе вкратце

Каждый кадр, который шлёт Veil, начинается с одного и того же
фиксированного заголовка, разложенного по байтам вот так:

```
[0..4]   magic        = "OVL1" (0x4F564C31)
[4..5]   version      = 0x01
[5..6]   family       = u8  (Session, Control, Discovery, Delivery, ...)
[6..8]   msg_type     = u16 BE (variant within family)
[8..12]  reserved     = 0x00000000
[12..16] body_len     = u32 BE
[16..20] trace_id     = u32 BE (sampled tracing)
[20..24] flags+prio   = u8 prio | u8 traffic_class | u16 reserved
[24..]   body         = msg_type-specific payload
```

Заголовок — фиксированные **24 байта**. В v1 нет необязательных
довесков: если когда-нибудь понадобится новый вариант, аварийный
переключатель (kill-switch) просто сменит магическое число на новое, а не
будет навешивать расширения на старое. Тело остаётся нетронутым, пока
диспетчер не решит, что это. Полный справочник:
[WIRE_PROTOCOL.md](WIRE_PROTOCOL.md).

## 9. Проход через NAT

Большинство узлов сидят за домашним роутером, который прячет их за одним
общим публичным адресом (это и есть NAT, трансляция сетевых адресов), —
и достучаться до них напрямую трудно. Два таких узла Veil соединяет в две
фазы: сначала **обнаружение**, затем **установление связи**:

```
Phase 1 — Discovery:
  Leaf → Core: NatProbeRequest                    "какой адрес ты видишь?"
  Core → Leaf: NatProbeResponse(observed_addr)    "вижу тебя по A.B.C.D:port"
  Leaf сохраняет `observed_addr` и публикует через Discovery.

Phase 2 — Establishment:
  Peer X хочет дотянуться до Peer Y (оба за NAT):
    X публикует свой NatProbe → Y публикует свой
    Каждая сторона одновременно шлёт UDP punch-пакеты
    Первый успешный ответ выигрывает; session establishes
  Fallback: relay tunnel через общий Core node
            (конфигурируется, по умолчанию off для Leaf).
```

В локальной сети есть путь ещё короче: узлы рассылают UDP-**маяки
mesh-сети** (групповая рассылка на 239.x.x.x), чтобы два телефона в одной
Wi-Fi находили друг друга напрямую, вообще не вовлекая Core-узел.

## 10. Защита от злоупотреблений

Защита выстроена слоями, и каждый ведётся отдельно по каждому соседу,
чтобы один нарушитель не испортил всё остальным:

| Уровень | Действие |
|---------|----------|
| Session limit | Max 32 session per IP source |
| PoW challenge | First-contact PoW (16-bit dev, 24-bit prod) |
| Rate limiter | Per-peer token bucket (конфигурируется) |
| Violation tracker | 5 violations → 1 час ban; ban resets через 1 день |
| Reputation | Long-term per-peer score (uptime + relay success + vouches); transit gate 200 points |
| Memory budget | Global RAM cap с priority-based eviction |
| Congestion monitor | Real-time load; >78% → drop transit frames |

Любой обработчик диспетчера, заметивший, что сосед нарушает правила —
неверная подпись, кадр, который не удаётся разобрать, кадр, ушедший не
туда, — записывает нарушение на счёт этого соседа. Подробности в
[SECURITY.md](SECURITY.md).

## 12. Версионирование

Всё описанное выше — это **OVL1 v1** (magic `0x4F564C31`, version byte
`0x01`). Протокол растёт за счёт того, что узлы согласуют между собой,
какие возможности они поддерживают, и растёт безопасно: старый узел,
встретив незнакомое семейство кадров, просто пропускает его
(`Unknown → forward-compatible`), а не спотыкается о него.
