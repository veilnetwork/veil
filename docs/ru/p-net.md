# P-Net: приватные сети veil

По умолчанию сеть veil публичная: подключиться может любой узел. P-Net
(Private Network mode, режим приватной сети) это запрещает. Внутрь
пускают только узлы с **сертификатом участника** — подписанным
удостоверением, которое доказывает, что узел свой. Подписать сертификат
должен **владелец сети** — тот, кто этой сетью управляет.

Проверка происходит на рукопожатии OVL1 — в момент, когда два узла
впервые устанавливают защищённую сессию. Узел без действительного
сертификата отсекается ещё до того, как сессия откроется.

## Модель доверия

```
Network Owner (Ed25519-ключи в offline-хранилище)
    │  подписывает
    ▼
Membership Cert (на узел; содержит node_id + admin-флаг + valid_until)
    │  предъявляется в HELLO и верифицируется на handshake
    ▼
P-Net Member (допущен в приватную сеть)
```

Сертификат даёт одну из двух ролей:

* **Администратор** (`admin: true` в сертификате) — может выдавать
  баны на всю сеть. Такие баны расходятся по DHT (общему справочнику
  сети) и доходят до каждого участника. Администраторы обычно работают
  на узлах первичного подключения — долгоживущих узлах, через которые
  остальные входят в сеть.
* **Участник** (`admin: false`) — может только подключаться, и не более
  того. Забанить соседа он тоже может, но такой бан остаётся на одном
  этом узле и никуда не расходится.

## Почему у банов два охвата

В публичном режиме каждый бан остаётся на том узле, который его выдал.
Так задумано: это не даёт вредоносному соседу отравить общую таблицу
банов всего кластера. Иначе вышел бы приём для отказа в обслуживании
(DoS) — один злоумышленник смог бы заблокировать всех разом.

В приватном режиме это правило можно ослабить. Сетевым банам
администратора доверяют, потому что его сертификат выдан владельцем, а
владельцу доверяют по определению. Поэтому узел можно забанить один раз,
с любого узла-администратора, и бан вступит в силу повсюду.

## Настройка оператором

### 1. Сгенерировать ключи владельца (однократно)

```bash
veil-cli network gen-owner \
  --pub-out  /etc/veil/owner.pub \
  --priv-out /etc/veil/owner.priv
```

Держите `owner.priv` **офлайн** — на USB-токене или в зашифрованной
резервной копии, но не на подключённой к сети машине. Кто угодно с этим
ключом сможет подписывать новые сертификаты администраторов и захватить
сеть.

### 2. Сгенерировать идентификатор сети (network ID, однократно)

```bash
veil-cli network gen-network-id
# network_id = 948b97b51b...ea87
```

Сохраните эту 64-символьную hex-строку. Она нужна каждому участнику в
конфиге, и именно она привязывает сертификат к вашей сети.

### 3. Подписать сертификат на каждого участника

```bash
# Admin cert (bootstrap, ops-узлы):
veil-cli network sign-member \
  --owner-pub /etc/veil/owner.pub \
  --owner-priv /etc/veil/owner.priv \
  --network-id "$NETWORK_ID" \
  --member-node-id "$NODE_ID_OF_BOOTSTRAP" \
  --admin \
  --valid-days 365 \
  --out /etc/veil/network.cert

# Member cert (обычный leaf — без `--admin`):
veil-cli network sign-member \
  --owner-pub /etc/veil/owner.pub \
  --owner-priv /etc/veil/owner.priv \
  --network-id "$NETWORK_ID" \
  --member-node-id "$NODE_ID_OF_LEAF" \
  --valid-days 365 \
  --out /etc/veil/network.cert
```

#### Параметры срока действия

По умолчанию сертификат истекает через `--valid-days` дней (365, если флаг
опущен). До наступления этого дня участник обязан выполнить ротацию —
получить от владельца свежий сертификат, — иначе он будет заблокирован.

Ротация — это морока для узлов парка, до которых трудно дотянуться:
встраиваемые устройства, резервные копии в air-gapped-окружении и тому
подобное. Для них передайте `--no-expiry`, чтобы выпустить
сертификат, который не истекает никогда.

```bash
# Бессрочный сертификат — отозвать можно только через DHT-бан или ротацию ключа владельца:
veil-cli network sign-member \
  --owner-pub /etc/veil/owner.pub \
  --owner-priv /etc/veil/owner.priv \
  --network-id "$NETWORK_ID" \
  --member-node-id "$NODE_ID_OF_FLEET_DEVICE" \
  --no-expiry \
  --out /etc/veil/network.cert
```

На уровне формата бессрочный сертификат кодируется как
`valid_until_unix = 0` (значение-сентинел: ноль означает «без срока
действия», а не реальную дату). Подкоманды `verify-cert` и `inspect-cert`
печатают `NEVER` вместо отметки времени.

Есть подвох. Чтобы отозвать одно такое устройство, кроме как ротацией
ключа владельца, его придётся забанить через DHT. Если это устройство
офлайн или в air-gapped-окружении, бан не достигнет его, пока оно не
переподключится к сети, — и до тех пор оно остаётся доверенным.

Для шагов выше понадобится `node_id` каждого участника. `node_id` — это
`BLAKE3(public_key)`, то есть хеш BLAKE3 от открытого ключа личности узла
(поле `public_key` в блоке `[identity]` в `node.toml`). Вычислить его
можно командой `blake3sum` или вспомогательным скриптом
`build-testnet-configs.py` на контроллере.

### 4. Настроить узел

Добавьте в `node.toml`:

```toml
[network]
mode = "private"
network_id = "948b97b51b...ea87"
owner_pubkey = "<base64 ed25519 owner pubkey>"  # содержимое owner.pub
owner_algo = "ed25519"
membership_cert = "/etc/veil/network.cert"

# Опциональный defense-in-depth — только cert'ы, чей member_node_id
# здесь указан, считаются admin'скими даже если cert.admin = true.
# Пустой список (default) безоговорочно доверяет admin-флагу в cert'е.
admin_node_ids = [
  "<hex node_id доверенного admin 1>",
  "<hex node_id доверенного admin 2>",
]
```

### 5. Перезапустить

Проверочный шлюз рукопожатия — тот самый механизм, который всё это и
обеспечивает, — собирается один раз при запуске узла. Поэтому новые
настройки вступают в силу только после перезапуска. С этого момента узел
требует сертификат от каждого входящего соседа и сам предъявляет свой
сертификат в исходящем HELLO — первом сообщении рукопожатия.

## Бан с узла-администратора

```bash
veil-cli network ban <NODE_ID_HEX> --reason "спам"
```

Вот что эта команда запускает. Она идёт по локальному admin-сокету IPC
(закрытому каналу между CLI и работающим демоном) как
`AdminCommand::PNetBan`. Демон подписывает `BanEntry` ключом личности
узла, применяет бан локально и рассылает его по DHT K ближайшим соседям.
Каждый остальной участник подхватывает бан на следующем тике
`p_net_ban_sync` — примерно раз в 60 с.

Убедиться, что бан дошёл до принимающих узлов:
```bash
veil-cli peers banned
```

## Допуск на уровне приложений (ogate / oproxy)

Шлюз P-Net в демоне решает, может ли сосед вообще открыть сессию OVL1.
Перед приложениями поверх демона стоит тот же вопрос про их собственный
трафик — это ogate (TUN-мост, который переносит IP-пакеты) и oproxy
(прокси SOCKS5/HTTP). Вместо того чтобы держать собственный
неизменяемый список `allowed_node_ids`, они могут повторно использовать
проверку, которую демон уже выполнил.

Работает это так. Каждое приложение отправляет
`LocalAppMsg::PnetStatusQuery` по своему сокету IPC и читает в ответ
`MembershipCert`, который демон закэшировал для этого соседа. На стороне
демона сертификат сохраняется во время рукопожатия в карту
`verified_peer_certs` (по одной записи на соседа) и отдаётся через
`PnetStatusProvider`. То есть приложение само ничего не проверяет — оно
просто спрашивает у демона то, что тот уже знает.

### oproxy-server

В `server.toml`:

```toml
pnet_required = true
allowed_node_ids = []   # пусто + pnet_required=true → "доверяй любому cert-verified peer"
allow_all = false        # не нужно когда pnet_required установлен
```

Что происходит, когда приходит входящий поток:

1. Исходный `node_id` сверяется со списком `allowed_node_ids` — это уже
   существующий шлюз.
2. Если `pnet_required = true`, oproxy дополнительно спрашивает демон
   запросом `peer_pnet_status(&src)` по IPC. Поток отвергается с
   `Denied`, если сосед не допущен (`admitted=false`) или у него нет
   сертификата (`has_cert=false`).

Если этот запрос к демону не удаётся, oproxy закрывается наглухо —
отвергает поток, а не пропускает его. Вы сами включили строгий шлюз;
запасной путь «в случае ошибки пропускать» тихо свёл бы его на нет.

### ogate

В `ogate.toml`:

```toml
mode = "authorized"
pnet_required = true

[[peers]]
node_id = "deadbeef..."
addr_v4 = "10.99.0.2"
```

Что происходит при запуске и затем при перечитывании конфига по сигналу
SIGHUP (это сигнал, который велит ogate заново прочитать конфиг):

1. ogate подключается к демону.
2. Проходит по списку `[[peers]]` и для каждой записи вызывает
   `peer_pnet_status`.
3. Отбрасывает любого соседа, у которого нет одновременно сертификата и
   допуска (`has_cert && admitted`), записывая предупреждение на каждого
   отброшенного.
4. Строит таблицу маршрутизации из тех соседей, что прошли отбор.

Сочетайте это с `mode = "authorized"` для эшелонированной защиты — двух
независимых проверок вместо одной. Сосед должен и иметь проверенный
сертификат, и присутствовать в списке `[[peers]]`.

### Порядок действий оператора

```bash
# 1. Daemon в P-Net mode (operator-side, см. секции выше).

# 2. Выпустить cert для peer'а который будет использовать oproxy:
veil-cli network sign-member \
  --owner-pub /etc/veil/owner.pub \
  --owner-priv /etc/veil/owner.priv \
  --network-id "$NETWORK_ID" \
  --member-node-id "$PEER_NODE_ID" \
  --no-expiry \
  --out /etc/veil/peer-cert.bin

# 3. Сторона peer'а: установить cert, перезапустить daemon. Daemon
#    презентует cert в HELLO, daemon b1 верифицирует + кэширует.

# 4. На b1: настроить oproxy-server.toml с pnet_required = true.
sudo oproxy-server --gen-config > /etc/oproxy/server.toml
sudo vim /etc/oproxy/server.toml   # установить pnet_required = true
sudo systemctl restart oproxy-server

# 5. Verify: peer открывает stream → admitted. Случайный non-verified
#    peer → Denied + log entry.
```

## Что отвергается

| Что произошло | Сообщение при рукопожатии |
|---|---|
| Узел в публичном режиме подключается к приватному кластеру | `peer did not present a membership cert (network is private)` |
| Сертификат подписан другим владельцем (чужая сеть) | `cert verification failed: cert network_id does not match local: expected=... got=...` |
| Сертификат просрочен | `cert verification failed: cert expired at unix=...` |
| `cert.member_node_id` не совпадает с подтверждённым `node_id` соседа | `cert is not for this peer: cert.member_node_id=... peer_node_id=...` |
| Содержимое сертификата повреждено | `cert blob decode failed: ...` |

## Раскатка через Ansible

В репозитории есть два плейбука: `ansible/deploy-pnet.yml` переводит
кластер в приватный режим, а `ansible/revert-pnet.yml` возвращает его в
публичный. Оба идут с `serial: 1` — по одному хосту за раз, — поэтому
кластер всё время сохраняет кворум и не уходит в полную темноту.

Прежде чем их запускать, на контроллере должны быть:
- `/tmp/pnet/owner.pub` + `/tmp/pnet/owner.priv` + `/tmp/pnet/network_id.hex`
- По одному cert'у на хост в `/tmp/pnet/<cfg-name>.cert` (cfg-name =
  b1/b2/b3 + n1..n5 — см. карту `inv_to_cfg` в `deploy-pnet.yml`).

```bash
# Раскатать:
ansible-playbook -i inventory.yml deploy-pnet.yml

# Откатить (убирает [network] блок + cert, рестартит в public mode):
ansible-playbook -i inventory.yml revert-pnet.yml
```

## Архитектурные заметки

* **Стоимость проверки сертификата**: одна проверка подписи Ed25519,
  примерно ~30 μs на современном чипе x86 — дёшево. Шлюз выполняет её на
  каждом входящем подключении. Сертификат заново декодируется на каждом
  рукопожатии и не кэшируется, поэтому изменение списка разрешённых
  администраторов вступает в силу сразу при перечитывании конфига, а не
  после следующего перезапуска.
* **Сколько времени бан расходится по сети**: ≤60 с — задаётся
  интервалом задачи применения. Можно ускорить, выдав один и тот же бан
  с каждого узла-администратора, но рассылка по DHT (K=8 ближайшим
  соседям) обычно и так накрывает кластер за один тик.
* **Как выводится ключ DHT**:
  `BLAKE3(network_id || ":bans:" || banned_node_id)`. Принимающий узел
  проверяет и подпись бана, **и** что бан записан под совпадающий ключ.
  Эта вторая проверка не даёт записать бан под неправильным ключом.

## Связанный код

* `crates/veil-identity/src/network_cert.rs` — кодирование и проверка
  сертификата
* `crates/veil-identity/src/network_ban.rs` — кодирование `BanEntry`,
  проверка цепочки доверия и ключ DHT
* `crates/veil-identity/src/network_access.rs` — `NetworkAccessGate`
  (обёртка вокруг проверки, выполняемой при рукопожатии)
* `crates/veil-node-runtime/src/runtime/p_net_ban_sync.rs` —
  `publish_p_net_ban` и периодическая задача применения
* `crates/veil-cli/src/cmd/network_cmd.rs` — обработчики подкоманд
  `veil-cli network …`

## Ограничения / открытая работа

* **Нет списка отзыва**: отозвать один сертификат по требованию нельзя.
  Украденный сертификат администратора остаётся действительным до
  момента `valid_until_unix`. Чтобы вывести сертификаты администраторов
  из обращения досрочно, перевыпустите их с новой парой ключей
  администратора и раздайте заново.
* **Потеря ключа владельца катастрофична**: кто угодно с закрытым ключом
  владельца может выпускать сертификаты администраторов, которые проходят
  шлюз где угодно. Берегите его как корневой ключ CA (удостоверяющего
  центра) — храните на машине без доступа к сети, в HSM (защищённом от
  взлома устройстве для ключей) или на бумажной копии.
* **Нет связи между сетями**: администратор сети A не может дотянуться до
  сети B. Каждая сеть — отдельная, ни с чем не пересекающаяся область
  доверия.
