# Несколько устройств на одной личности

Эта страница про то, как держать одну **личность** (identity) —
ваш криптографический «паспорт» в сети — сразу на нескольких
устройствах. Может быть, вы хотите распределить нагрузку между
флотом серверов. А может, просто хотите, чтобы одни и те же
сообщения приходили на телефон, ноутбук и десктоп. И то и другое
работает одинаково, и эта страница разбирает как.

Про протокол под капотом — см.
[`identity-model.md`](identity-model.md).

---

## 1. Два режима: балансировка нагрузки и мессенджер

Сообщения могут расходиться по вашим устройствам двумя способами.
Какой из них сработает, решает отправитель, а не вы. Адресуясь к
вашей личности, он прикладывает
[`InstanceTag`](../../crates/veil-proto/src/recipient.rs) — маленький
флаг, который говорит, какие из ваших устройств должны получить
сообщение:

| Тег | Смысл | Когда применяется |
|-----|---------|----------|
| `InstanceTag::Any` | «Любой один активный экземпляр» | Балансировка нагрузки между флотом серверов |
| `InstanceTag::All` | «Все активные экземпляры» | Обмен сообщениями на нескольких устройствах |
| `InstanceTag::Specific(id)` | «Именно этот экземпляр» | Адресная доставка / продолжение сессии |

Одна личность может работать в **обоих** режимах сразу.
Потребительский мессенджер берёт `All` для переписки человека с
человеком — тогда сообщение получает каждое устройство.
Бизнес-интеграция на той же личности берёт `Any` для флота
mail-server-экземпляров — тогда отвечает лишь один из них.

## 2. Что такое экземпляр

**Экземпляр** (instance) — это один процесс veil, работающий на
одном устройстве. У каждого экземпляра есть четыре своих вещи:

- Случайное 16-байтное имя — **`instance_id`**. Оно пишется в
  `~/.config/veil/instance_id` при первом запуске и дальше остаётся
  неизменным на всю жизнь устройства. Перезагрузки, смены личности
  и восстановления из master-seed его не трогают.
- Свой ключ подписи Ed25519 — **`identity_sk`**. Каждый такой ключ
  мастер-ключ заверяет по отдельности. Поэтому, если `identity_sk`
  одного устройства украдут, вы отзываете только *этот* подключ
  (subkey), а остальные продолжают работать.
- Своя пара ключей шифрования ML-KEM-768, заверенная тем же
  `identity_sk`.
- Место в общем **`IdentityRegistry`** — списке, который каждый
  сосед достаёт, прежде чем направить сообщение вам.

```
             identity_id (стабильный)
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
 identity_keys[0]  [1]             [2]
 bound:            bound:          bound:
  laptop           phone           server-farm
  instance_id_A    instance_id_B   instance_id_C
     ▲                 ▲               ▲
  sig_sk_A          sig_sk_B        sig_sk_C
  mlkem_A           mlkem_B         mlkem_C
```

Личность — это просто дерево выше. Добавление устройства наращивает
новый лист. Удаление устройства подрезает один лист и не трогает
остальные.

## 3. Добавление устройства — церемония сопряжения

**Сопряжение** (pairing) — это привязка нового устройства к
личности, которой вы уже владеете. Два устройства проводят короткий
обмен туда-сюда — ту самую «церемонию», — и в конце у нового
устройства есть свои ключи, а сеть знает, что ему можно доверять.
Схема ниже прослеживает весь обмен, а текст после неё объясняет,
зачем нужен каждый шаг.

```
          Primary-устройство                       Новое устройство
          (с master_seed)                          (свежая установка)

          $ veil-cli identity pair-invite                    │
            ↳ pair_secret сгенерирован                       │
            ↳ PairingInvite опубликован                      │
            ↳ QR отображён: veil:pair?id=X&secret=Y          │
            ↳ Показан OOB 6-значный код: "123-456"           │
                                                             │
                                              сканировать   ◄┤
                                                       QR    │
                                                             │  подключение напрямую
                                                             │  к источнику через `secret`
                                 ┌──────────────────────►    │
                                 │                           │
                                 │                           │  target генерирует
                                 │                           │  свежий identity_sk
                                 │                           │  (master_seed НИКОГДА
                                 │                           │   не передаётся)
                                 │                           │
          source разлочивает    ◄┤                           │
          master                                             │
          master_sk сертифицирует                            │
          target'ский identity_sk:                           │
            IdentityKey {                                    │
              pubkey: target_pk,                             │
              bound_instance_id: target_id,                  │
              master_sig: ...                                │
            }                                                │
          Дописано в                                         │
          IdentityDocument.                                  │
          Republished.          ─────────────────────────►   │
                                                             │
          OOB-проверка: source показывает "Target: XXX-XXX?" │
                       target показывает тот же              │
                       детерминированный код, выведенный     │
                       из session-ключа                      │
          Пользователь визуально сравнивает,                 │
          подтверждает на source. ──────────────────────►    │
                                                             │
                                 InstanceEntry target'а   ◄──┤
                                 дописан в                   │
                                 InstanceRegistry.           │
                                                             │
                                 DeviceLinkedEvent           │
                                 push'ится существующим      │
                                 instance'ам.                │
```

**Почему это безопасно**:

- `master_seed` — корневой секрет, который управляет всей
  личностью, — никогда не покидает основное устройство. Новое
  устройство создаёт свой `identity_sk`, и заверить его может
  только мастер. Так что даже уже скомпрометированное устройство не
  сможет заверить следующие.
- Оба устройства показывают по короткому коду подтверждения,
  посчитанному по отдельному каналу (OOB, out-of-band) — то есть по
  пути, который злоумышленнику не виден; в данном случае это ваши
  собственные глаза, сравнивающие два экрана. Это срывает попытку
  поддельного устройства втихую отсканировать QR, предназначенный
  настоящему: показать настоящий код может только настоящее
  устройство. У злоумышленника, перехватившего QR, получится другой
  ключ сессии, а значит и другой код, и сравнение не сойдётся.
- `pair_secret` живёт всего 5 минут (TTL) и срабатывает ровно один
  раз.

**Команды, которые вы и правда набираете**:

На основном устройстве распечатайте приглашение — URI плюс
QR-код, который новое устройство сможет отсканировать. Параметр
`--endpoint` обязателен: он говорит новому устройству, по какому
адресу перезвонить.
```bash
veil-cli identity pair-invite --ttl-secs 300 --endpoint tcp://HOST:PORT
# QR отрисован + OOB-код показан.
```

Чтобы провести всю интерактивную церемонию со стороны основного
устройства — открыть приёмник, принять обратный звонок, сверить
коды и заверить мастер-ключом новый подключ — используйте
`pair-listen`:
```bash
veil-cli identity pair-listen --endpoint tcp://HOST:PORT
# Bind'ит listener, печатает URI + QR, выполняет source-сторону.
```

На новом устройстве — телефоне, который сканирует, — передайте
отсканированный URI как **позиционный** аргумент (флага `--qr`
нет):
```bash
veil-cli identity pair-accept <veil:pair?…-url>
# Показывает OOB-код — пользователь визуально сравнивает с primary.
# Если коды совпадают, пользователь тапает "confirm" на primary.
```

Сверка кодов на глаз — обычный путь на обеих сторонах. Флаг
`--yes-i-compared-codes` пропускает этот запрос, и существует он
только для скриптовых тестов — руками к нему не тянитесь.

За 60–90 секунд новое устройство полностью оживает.

## 4. Удаление устройства

> ⚠️ **Команды `identity revoke` пока нет.** Сегодня отзыв (revoke)
> устройства — это ручная операция уровня протокола: с устройства,
> где есть мастер-сид, вы правите `IdentityDocument` и заново его
> публикуете. Отдельной подкоманды, которая сделала бы это за вас,
> нет. Подкоманды `veil-cli identity`, которые *есть*: `create`,
> `show`, `rotate`, `restore`, `claim-name`, `qr`, `pair-invite`,
> `inspect-uri`, `pair-listen`, `pair-accept`, `export-qr-backup`,
> `import-qr-backup`, `standalone`, `delegate-device`, `migrate`,
> `dht-key`, `name-dht-key`.

### 4.1. Как это работает

Делает это тот, у кого мастер-ключ. Берём `identity_sk` нужного
устройства — подключ из `IdentityDocument.identity_keys`, к
которому оно привязано, — и добавляем его в набор `revoked_keys`
документа. Повышаем `document_version`, переподписываем и заново
публикуем обновлённый `IdentityDocument`. У личности не больше
`MAX_IDENTITY_KEYS = 8` живых подключей, так что документ остаётся
компактным. Как только сосед достанет новую версию, он перестаёт
принимать любые будущие кадры, подписанные отозванным подключом.
А мёртвый `InstanceEntry` просто вытесняется из реестра при
следующей публикации.

### 4.2. Как быстро это расходится

Заново опубликованный документ доходит до соседей двумя путями.
Медленный путь — это DHT (общий справочник), который переиздаёт
запись по своему расписанию, и это могут быть часы (вплоть до
DHT TTL). Быстрый путь — это сплетни (gossip) и прямая отправка по
мере того, как сессии переподключаются; для соседей, с которыми вы
уже общаетесь, это секунды. Отдельных флагов `scheduled` и
`compromise` сегодня нет; разница только в том, как вы действуете.
Если устройство скомпрометировано, публикуйте заново сразу же и
полагайтесь на сплетни, а не ждите следующей плановой публикации.

## 5. Семантика доставки сообщений

### 5.1. Режим мессенджера (`InstanceTag::All`)

Здесь отправитель шифрует сообщение **по разу для каждого активного
устройства** — этот приём называют веерной рассылкой (fan-out). Он
достаёт каждый `MlKemKeyCert` из реестра получателя и собирает по
одному `FanoutEnvelope` на сертификат. Каждый конверт несёт
instance_id плюс шифротекст ML-KEM и шифротекст AEAD, и открывается
он только под seed'ом декапсуляции ML-KEM одного-единственного
устройства.

```
Sender                              Recipient identity @alice
   │
   │  fanout_encrypt(plaintext, certs=[A,B,C])
   │     → [env_A, env_B, env_C]
   │
   │─────── DELIVERY_FORWARD ─────────►  Ноутбук Алисы    (instance A)
   │                                     Телефон Алисы    (instance B)
   │                                     Десктоп Алисы    (instance C)
   │
   │  Каждое устройство:
   │    - выбирает envelope, чей instance_id == self
   │    - decapsulate'ит ML-KEM под собственным seed'ом
   │    - расшифровывает plaintext
   │    - доставляет в application-слой
```

- **Почтовый ящик** (mailbox) хранит копии для устройств, которые
  сейчас не в сети. Он раскладывает их по получателю (`receiver_id`),
  а не по устройству, поэтому нет ни покурсорных ACK по каждому
  экземпляру, ни параметра `instance_stale_after`, который надо было
  бы крутить. Каждый блоб (blob) лежит под первичным ключом
  `(receiver, content_id)`. Уборка идёт по индексу вытеснения с
  ключом `(deposited_at_be || receiver || content_id)` плюс по
  байтовым квотам — на каждого отправителя и общей. Когда квота
  заполняется, первыми уходят самые старые отложенные записи
  (сначала блобы анонимного пула, потом идентифицированного), а любой
  блоб удаляется, как только его подтвердят (ACK).
- **Одноразовые предключи X3DH** (one-time prekeys) вступают в дело,
  когда получатель вообще не в сети. Отправитель берёт
  неиспользованный предключ из пула, который опубликовал получатель,
  и упаковывает сообщение к этому предключу, а не к долгоживущему
  сертификату ML-KEM. Предключ сгорает при первой расшифровке —
  именно это даёт асинхронным сообщениям прямую секретность
  (forward-secrecy): старое сообщение нельзя вскрыть ключом,
  украденным позже.

### 5.2. Режим балансировки нагрузки (`InstanceTag::Any`)

Здесь veil выбирает лишь *один* из активных экземпляров. Выбирает
по опубликованному `InstanceEntry.last_seen_unix_ms` и локальной
оценке репутации. Создаётся ровно один шифротекст, и открывает его
ровно один экземпляр.

Типичный пример — почтовый шлюз с личностью `@mailserver` и тремя
региональными экземплярами. Клиенты шлют на `mailserver:Any`, а
veil направляет каждое сообщение в тот регион, который ближе всего
к отправителю по задержке.

### 5.3. Адресный режим (`InstanceTag::Specific`)

Этот нужен, когда разговор уже закрепился за конкретным
экземпляром, и остальные пакеты должны идти к нему же
(непрерывность сессии). Дедупликация сессий идёт по ключу
`(identity_id, instance_id)`, так что два экземпляра одной личности
могут держать каждый свою отдельную сессию с одним соседом.

## 6. Что публикует `InstanceRegistry`

Нынешний `InstanceEntry` несёт всего четыре поля: `instance_id`,
`bound_identity_key_idx`, `label` и `last_seen_unix_ms`. Полей
`mailbox_anchor`, `transports` и `encrypted_contact` нет. Прежнее
разделение Tier-A / Tier-B — и блок настроек
`[identity.multi_device.tier_b]`, который шёл вместе с ним, —
**ушло**. Слой шифрования Tier-B убрали, а `mailbox_anchor` и
`encrypted_contact` ушли вместе с ним.

Подсказки по транспорту — намёки на то, как достучаться до
устройства, — теперь ходят отдельно, по сплетням
`SignedTransportAnnouncement`, а не лежат в реестре. Итог — это
приватность. Тот, кто пассивно вычитывает `InstanceRegistry` из
DHT, не находит ничего, что можно сопоставить: ни якорей, ни
адресов транспорта. Всё, что он видит, — сколько устройств вы
держите, их непрозрачные `instance_id` и какие label выбрал
оператор. Формат на проводе — см.
[`identity-model.md`](identity-model.md).

## 7. Репутация — на личность, а не на устройство

Репутация привязана к `identity_id`. Каждый экземпляр пополняет
один общий счёт и им же пользуется. Так задумано. Флот
mailserver-экземпляров под одной личностью набирает вес сообща. А
злоумышленник, скомпрометировавший `identity_sk` одного
устройства, в одиночку испортить репутацию личности не может:
отзыв убирает этот ключ из сети раньше, чем ущерб успеет
накопиться.

## 8. Синхронизация состояния приложения (пока в проекте)

> 🚧 **Статус:** только проект. Примитив `AppState` ещё не стал
> настоящим типом. Ниже описано то, к чему мы стремимся. Пока же
> приложения, которым нужна синхронизация между устройствами, сами
> заводят DHT-слоты по любой удобной им схеме app-key.

Контакты, список блокировок, настройки, профиль — всё, что
приложение хочет держать синхронным между устройствами
пользователя, — поедет на штуке под названием `AppState`. Это один
DHT-слот на кортеж `(identity_id, app_id, key)`, зашифрованный под
общим секретом личности `app_state_secret` и подписанный любым
активным `identity_sk`.

Каждое связанное устройство может читать и писать. Обновления
монотонны по версии: пришедшее не по порядку просто отбрасывается.
Каждый блоб не больше 4 KB, так что, если нужно больше места,
разнесите состояние приложения по нескольким ключам.

## 9. FAQ

**В:** Могут ли два экземпляра моей личности работать одновременно
на одном IP?

**О:** Да. У них разные `instance_id`, поэтому дедупликация сессий
держит их соединения порознь. Соседи просто видят два активных
экземпляра за одной личностью.

**В:** Можно ли поделить один `identity_sk` между двумя
устройствами и обойтись без сопряжения?

**О:** Нет. У каждого экземпляра свой `identity_sk` — так
задумано. Сопряжение и существует ровно для того, чтобы ни один
ключ никогда не делился между устройствами. Именно это удерживает
последствия взлома в пределах одного устройства.

**В:** Сколько устройств я могу связать?

**О:** Шестнадцать — таков предел `IdentityRegistry`. Для флотов
побольше распределяйте нагрузку по нескольким личностям.

**В:** Что будет, если я отзову ноутбук, а телефон неделю не в
сети?

**О:** Телефон по-прежнему будет принимать сообщения, адресованные
именно ему. Когда он переподключится, то достанет свежий
`IdentityDocument`, увидит, что его собрата отозвали, и покажет
оператору уведомление в духе DeviceLinkedEvent. Ваше имя, репутация
и собственный identity_sk телефона остаются нетронутыми.

**В:** А если телефон скомпрометируют, пока ноутбук не в сети?

**О:** Отзовите подключ телефона с ноутбука, как только тот
вернётся в сеть. Пока отзыв не дойдёт до конкретного соседа — через
переиздание в DHT или через прямую отправку, когда сессии
переподключатся, — этот сосед может ещё принимать кадры,
подписанные ключом телефона. В худшем случае путь через DHT
занимает часы. По сплетням ближние соседи подхватывают это за
секунды.
