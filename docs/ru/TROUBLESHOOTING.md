# Решение проблем

> Что-то пошло не так? Каждый пункт ниже устроен одинаково: симптом →
> вероятная причина → решение. Парный документ к
> [OPERATIONS.md](OPERATIONS.md) и [MONITORING.md](MONITORING.md).

## Быстрая сортировка проблем

Прежде чем углубляться, снимите слепок того, что узел делает прямо сейчас. Многие
решения ниже ссылаются на этот вывод, так что начать стоит именно с него:

```bash
veil-cli -c node.toml node show           # version + uptime + counts
veil-cli -c node.toml peers list          # configured peers
veil-cli -c node.toml peers banned        # активные баны
veil-cli -c node.toml sessions list       # установленные сессии
veil-cli -c node.toml node dht routing    # DHT k-bucket'ы
veil-cli -c node.toml node metrics        # снапшот counter'ов
journalctl -u veil-node --since "10 min ago" > /tmp/veil.log
```

## Узел не запускается / сразу падает

### `compile_error: release build without seeds`
В релизной сборке нет записей о начальных (seed) узлах, и не включён ни флаг
сборки `production-seeds`, ни `allow-empty-seeds` — узлу попросту неоткуда войти
в сеть, и он отказывается собираться. См.
[OPERATIONS.md → Настройка seed-узла](OPERATIONS.md#настройка-seed-узла). Для
тестовой (testnet) сборки быстрое решение — `cargo build --release --features allow-empty-seeds`.

### `error: identity validation failed: PoW score N < required 24`
Указанный в конфигурации одноразовый код (nonce) в секции `[identity]` больше не
проходит порог сложности — обычно потому, что сложность подняли в более свежей
сборке. Просто намайните новый:

```bash
veil-cli config init --force /path/to/config.toml
# или для sovereign identity:
veil-cli identity create --veil-dir /path/to/veil
```

### `error: admin socket already in use`
Предыдущий запуск `veil-cli node run` оставил после себя устаревший файл сокета.
Пока ни один процесс veil на самом деле не работает, его можно спокойно удалить и
запуститься заново:

```bash
rm /run/veil/admin.sock
systemctl start veil-node
```

### Узел запускается, потом тихо выходит
Причину подскажет журнал — начните с
`journalctl -u veil-node --since "5 min ago"`. Чаще всего виноват пропавший
`identity_document.bin` после переноса или восстановления. Если дело в этом,
восстановите его из BIP-39-фразы, как описано в
[OPERATIONS.md → Потеря личности](OPERATIONS.md#потеря-identity--восстановление-из-bip-39).

## Узлы не соединяются

### `WARN peer.connect.failure ... Connection refused`
Дальний конец либо не слушает, либо на пути стоит межсетевой экран (firewall).
Проверьте путь со стороны узла, который инициирует соединение:

```bash
nc -vz <remote_host> <remote_port>
# Connection refused → service down или firewall reset
# Connection timeout → firewall drop или нет маршрута
```

Если же вы на принимающей стороне, `veil-cli node show` должен показывать
`listens_active` > 0. Если не показывает — посмотрите строку журнала
`listen.start` на ошибки привязки к порту.

### `INFO session.banned ... node_id=XXXX — banned peer rejected`
Одна из сторон забанила другую — либо вы их, либо они вас. Посмотрите:

```bash
veil-cli peers banned | grep <node_id_short>
# Если найдено: кто-то явно забанил этот узел, проверьте колонку `reason`.

# Чтобы снять ошибочный ban:
veil-cli peers unban <NODE_ID_HEX>
# NB: баны персистятся в <config-dir>/bans.json — переживают рестарт.
```

Если бан симметричный (стиль ban-script, когда каждая сторона банит другую),
снять `unban` нужно с обеих сторон — только тогда связь восстановится полностью.

### `WARN peer.nonce_mismatch ... old=XXX new=YYY`
Узел на другом конце заново намайнил свою личность — например, после повышения
сложности. Делать ничего не нужно: при перезагрузке одноразовый код (nonce) в
вашей секции `peers[]` сам обновится из рукопожатия. Стоит написать удалённому
оператору, только если изменение оказалось неожиданным.

### Узел соединяется и тут же отваливается (handshake.success → session.close)
Обычно это значит, что не совпали версии протокола OVL1 или достигнут предел по
числу сессий. Загляните в журнал:

```bash
journalctl -u veil-node | grep -E "handshake|session\.close" | tail -20
# Смотрите "session limit reached" или "version mismatch"
```

Если дело в пределе, поднимите `[session].max_concurrent` (по умолчанию 65536,
хотя в некоторых конфигурациях он ниже).

## Сообщения чата / приложений не доставляются

### `RuntimeError: no active OVL1 session to NNNN…`
Это `IPC_SEND_ERR_NO_ROUTE` от локального узла: маршрутизация вообще не нашла
пути до получателя. Пройдите по шагам по порядку — каждый сужает место, где
обрывается путь:

1. **Есть ли прямая сессия до получателя?**
   `veil-cli sessions list -v | grep <dst_node_id>` — если есть, проблема
   дальше по цепочке (узел забанен или сломан). Если её нет, идите дальше.

2. **Знает ли о получателе какой-нибудь k-bucket в DHT?**
   `veil-cli node dht routing | grep <dst_node_id>` — если записи нет, значит
   рассылка по DHT после рукопожатия не дошла. В большинстве случаев это лечит
   перезапуск.

3. **Не забанен ли он?**
   `veil-cli peers banned | grep <dst_node_id>` — если он в списке, вот и
   причина. Снимите бан командой `peers unban`.

4. **Не фрагментирована ли сеть?** Проверьте, что у двух узлов есть общий
   сосед. Даже в кольцевых топологиях (5 узлов, бан через одного) цепочка
   «рекурсивный запрос + ответ по обратному пути» всё равно должна срабатывать.
   Убедитесь, что `veil-cli node show | grep version` соответствует текущей
   сборке.

### `RuntimeError: E2E key for NNNN… not yet cached`
Это `IPC_SEND_ERR_NO_E2E_KEY`. Маршрут известен, но ML-KEM-ключ получателя ещё
не закэширован. Так бывает, когда кэш маршрутов наполнился через обычную
рассылку RouteAnnounce (в ней ML-KEM-ключа нет) или когда попутно прикреплённый
RouteResponse так до нас и не дошёл (ошибка в сборках до 467.2f).

Хорошая новость в том, что это обычно лечится само. Сначала убедитесь, что
сборка — после 467.2f (`node show | grep version`). При повторной отправке вызов
IPC запускает свежий рекурсивный запрос, а тот несёт попутный RouteResponse с
ML-KEM-ключом. Если повторные попытки всё равно падают, перезапустите узел
получателя — его `mlkem.key` мог повредиться; на следующем старте он создаётся
заново.

### Сообщения доставляются долго (5+ с)
Обычно на причину указывают несколько счётчиков:

- `route_selection_avg_rtt` — высокое время передачи (RTT) означает медленный
  узел выше по пути.
- `decrypt_failures_total` — повторы или рассинхрон ключей.
- Накапливающиеся повторные передачи `pending_ack` — проверьте через
  `veil-cli node metrics | grep ack`.
- Время ожидания (TTL) у chat_client по умолчанию 30 с; снизьте его, если
  хотите, чтобы отправка быстрее сообщала об ошибке.

### Сообщения тихо теряются (без ошибки, без доставки)
Когда сообщения просто пропадают без ошибки, кадр, скорее всего, больше
`[session].max_frame_body` — узел отбрасывает слишком большие кадры, никого не
уведомляя. Убедитесь, что у отправителя и получателя одинаковый предел (по
умолчанию 1 MiB). Второй вариант: сообщение разбито на части (Chunk), а
отправляющий узел не договорился о разбиении — поэтому проверьте, что обе стороны
на OVL1 minor ≥ 2.

## Баны / злоупотребления

### `INFO session.banned` для узла, который должен быть разрешён
Скорее всего, это автоматический бан от `kill_session`, который ставит короткий
временный бан на 30 с. Можно просто подождать, пока он истечёт, либо сразу снять
его через `peers unban`.

Если бан держится дольше, загляните в `<config-dir>/bans.json`:

```bash
cat /var/lib/veil/bans.json | jq .
# Редактируйте / удаляйте записи; они перезагружаются на следующем старте узла
```

### Резкий всплеск метрики бан-шторма
За всплеском обычно стоит один источник. Чтобы его найти:

- Запустите `journalctl -u veil-node | grep session.banned` и посмотрите, какие
  `node_id` банятся — не один ли это кластер с общим IP-адресом источника?
- Проверьте `peers banned` на поток свежих записей.
- Если это настоящее злоупотребление, подумайте о том, чтобы ужесточить
  ограничения частоты в секции `[abuse]` или пределы на один IP.

### Нужно забанить целую подсеть /24
`peers ban` так пока не умеет — он работает только по отдельному `node_id`.
Запасной путь — правило межсетевого экрана на уровне хоста
(`iptables -A INPUT -s 10.0.0.0/24 -j DROP`).

## Странности DHT / маршрутизации

### `node dht routing` показывает меньше контактов, чем ожидалось
В сборках до Epic-467.2c на каждый k-bucket стояло ограничение частоты (не
больше одной вставки в секунду), которое отбрасывало одновременные добавления
после рукопожатий. Проверьте версию сборки: начиная с 467.2c-d это исправлено.

Если же вы только что перезапустились и PEX ещё не успел отработать (по
умолчанию он обходит сеть каждые 120 с), подождите пару минут и проверьте снова.

### `WARN recursive.response.relay_dropped`
Ответ на рекурсивный запрос не удалось переслать обратно тому, кто его начал, —
нет пути через DHT или кэш к `reply_to`. Это говорит о сильной фрагментации
сети. Проверьте список забаненных пар и убедитесь, что у инициатора есть хотя бы
один общий сосед с отвечающим узлом.

### Зацикливание в журнале `recursive.query` (тот же query_id ttl=40 каждые 525 мс)
Инициатор повторяет запрос снова и снова, потому что рекурсивный ответ так и не
пришёл. Сверьтесь с журналом отвечающего узла по `recursive.response.dropped` —
если запись там есть, отвечающий не смог достучаться обратно. Это исправлено до
467.2e, так что убедитесь, что сборка актуальна.

## Почтовый ящик

### `WARN mailbox.quota_reject`
Отправитель упёрся в свой предел частоты. Сначала выясните, кто это:

```bash
journalctl -u veil-node | grep mailbox.quota_reject | tail -20
```

Если это законный всплеск, поднимите `[mailbox].max_per_sender`; если
злоупотребление — забаньте отправителя.

### Почтовый ящик растёт неограниченно
Очистка почтового ящика должна каждую минуту вытеснять просроченные записи (у
которых TTL вышел за `mailbox_ttl_secs`, по умолчанию 7 д). Если
`veil_storage_evictions_total` не растёт, задача очистки могла застопориться —
проверьте, что `veil-cli node show | grep uptime` сходится с тем, насколько
свежими выглядят счётчики метрик. В крайнем случае перезапустите узел.

### Спящий получатель не просыпается
Проверьте, что объявление о пробуждении было и отправлено
(`veil_sleep_advertisements_emitted_total`), и принято
(`veil_sleep_advertisements_accepted_total`). Если отправлено, но не принято —
сломан шлюз или канал до получателя.

## IPC / chat-клиент

### `chat_client.py: connection refused on app.sock`
Обычно причин три: узел не запущен, в конфигурации стоит `[ipc].enabled = false`
или путь к сокету у клиента не совпадает с настоящим. Начните с проверки самого
сокета:

```bash
ls -la /path/to/veil/app.sock
# srwxr-xr-x — сокет существует с правильными правами
```

Если файл на месте, но в соединении всё равно отказано, узел мог умереть —
проверьте `node show`.

### Клиент соединяется, потом соединение сразу закрывается/сбрасывается (без записи в журнале демона)
Рукопожатие IPC отбрасывается ещё до завершения, и демон при этом ничего не
пишет в журнал — оттого случай и сбивает с толку. Скорее всего, дело в
несовпадении uid между пользователями (**cross-user peer-uid mismatch**). Демон
держит на сокете app-IPC проверку uid на уровне ядра (`SO_PEERCRED` /
`getpeereid`): он молча сбрасывает (`drop()`) любое соединение, у которого uid
собеседника отличается от его собственного, и **исключения для root нет**. Тот
же заслон охраняет admin-сокет.

Об это спотыкаются, когда IPC-клиент (`chat_client`, `ogate`, `oproxy`) или
admin-CLI запущен под другим пользователем, чем демон, — классика — под **root**
против демона, работающего под пользователем `veil`.

Решение — запускать IPC-клиент и admin-CLI под **тем же пользователем**, что и
демон:

```bash
sudo -u veil veil-cli -c node.toml node show
sudo -u veil ogate up
```

То есть **не** запускайте `ogate`, `oproxy` или admin-CLI под root против демона,
работающего не под root. Если нужна настройка TUN без root, выдайте
`CAP_NET_ADMIN` пользователю демона, а не запускайте всё под root. (На IPC через
TCP и через именованный канал Windows проверка uid собеседника ничего не делает:
`uid_matches_local` всегда true, так что это касается только сокетов Unix.)

### `chat_client: timeout waiting for reply`
Сама отправка могла пройти успешно, но ответ потерялся на обратном пути.
Проверьте счётчики `veil_pending_ack_*` у отправителя и `route_miss_total` у
получателя. Чтобы полностью исключить сеть и проверить только путь IPC, запустите
`chat_server.py` на том же узле, что и клиент, ради чистого эхо-теста.

## Производительность

### Высокая загрузка CPU (процесс `veil-cli`)
- `veil_decrypt_failures_total` — высокое значение указывает на лавину повторов
  или рассинхрон ключей; путь обработки сбоя AEAD дорог.
- `veil_active_sessions` — если их слишком много, пора наращивать ресурсы.
- Запущенный `lazy_miner` — это нормально во время начального доказательства
  работы (первые ~30 с); потом он должен утихнуть.

### Высокий расход памяти
- `veil_storage_evictions_total` должен расти. Если не растёт, вытеснение
  застопорилось — проверьте журнал упреждающей записи (WAL) почтового ящика и
  предел DHT.
- `node dht list | wc -l` — если значение подбирается к `max_store_entries`,
  либо поднимите предел, либо смиритесь с вытеснением.

### Высокий исходящий трафик
Проверьте темп роста `veil_transport_bytes_tx_total`. Для роли ретранслятора
или шлюза высокий исходящий трафик нормален. Для листового узла стоит
присмотреться — через `veil-cli sessions list` поищите неожиданные исходящие
узлы.

## Приватность / утечки

### В журналах есть IP-адреса узлов
На уровне INFO журналы по умолчанию включают строки вроде
`transport=tcp://1.2.3.4:5678` и `source=inbound(peer-N)` для рукопожатий. Если у
вас развёртывание с упором на приватность, поставьте
`[global].log_level = "warn"`, чтобы отсечь болтовню уровня INFO — журналы WARN и
ERROR ссылаются только на `node_id`, а он и без того непрозрачен.

Полноценная проверка на персональные данные (PII) ещё впереди, так что пока
обходной путь — фильтрация по log_level.

### `node show` показал URL метрик с токеном
Строки запроса вида `?token=...` вырезаются из отображения `metrics_endpoint`.
Если вы всё же видите токен в выводе, сборка устарела — обновитесь до актуальной
(проверьте через `node show | grep version`).

## Когда ничего не помогает

Когда ничего из перечисленного не подошло, вернитесь к основам:

1. Снимите слепок состояния: `node show`, `peers list`, `peers banned`,
   `sessions list -v`, `node dht routing`, `node metrics` и записи журнала за
   последние 30 минут.
2. Перезапустите узел. Это сохраняет ваши баны, значения DHT и личность, а
   временное состояние очищает.
3. Если всё ещё не работает, заведите задачу (issue) с собранным состоянием и
   подходящим куском журнала вокруг того момента, когда начались симптомы.
