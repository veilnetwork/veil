# Установка и первый узел

Это руководство проведёт вас от одной команды `curl` до работающего узла veil.
Оно подойдёт и новичку, который впервые видит проект, и оператору, который
поднимает целый парк серверов.

Установщик скачивает из GitHub Releases готовые сборки и проверяет каждую через
**sha256** — короткий «отпечаток» файла, который подтверждает, что файл дошёл
целым и не был подменён. Набор инструментов сборки Rust вам не нужен — он нужен
только при [сборке из исходников](#сборка-из-исходников).

---

## Коротко — однострочники

**Linux / macOS:**

```sh
curl --proto '=https' --tlsv1.2 -sSf \
  https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.sh | sh
```

**Windows (PowerShell):**

```powershell
irm https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.ps1 | iex
```

Затем запустите узел:

```sh
veil-cli config init      # свежая identity + конфиг
veil-cli node run         # запуск в фоне
veil-cli node show        # node id, аптайм, пиры
```

Готово — вы в сети. Дальше расскажем, как добавить инструменты gateway и proxy,
поднять публичный сервер и тонко настроить установку.

---

## Что устанавливается

По умолчанию ставится только **`veil-cli`** — сам узел, который к тому же умеет
обновлять себя. Он попадает в каталог, которым владеете вы, поэтому `sudo` не
нужен:

| Платформа | Путь |
|-----------|------|
| Linux / macOS | `~/.veil/bin` (добавляется в `PATH` через `~/.veil/env`) |
| Windows | `%USERPROFILE%\.veil\bin` (добавляется в пользовательский `PATH`) |

Всего программ четыре, и поставить можно любой их набор:

| Программа | Роль | Сторона |
|-----------|------|---------|
| `veil-cli` | Узел: вход в сеть, маршрутизация, DHT, личность, самообновление | клиент **или** сервер |
| `ogate` | Мост IP поверх veil (виртуальная локальная сеть) | сервер / шлюз |
| `oproxy-client` | Локальный SOCKS5 / HTTP / TProxy → veil | клиент |
| `oproxy-server` | Выходной / proxy-сервер veil | сервер |

Чтобы поставить больше, чем один узел:

```sh
# всё (узел + ogate + oproxy client & server)
curl -sSf https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.sh | sh -s -- --all

# конкретный набор
... | sh -s -- --components ogate,oproxy-server
```

На Windows:

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.ps1))) -All
```

## Запуск узла

### Клиент / лист (по умолчанию)

*Лист* — это узел, который сам тянется *наружу*, в сеть, а не принимает входящие
подключения. Публичный адрес ему не нужен, и он спокойно работает за NAT — так
называют домашний роутер, который раздаёт один общий адрес всем устройствам в
вашей сети.

```sh
veil-cli config init --profile mobile   # лист с учётом батареи (или без --profile — обычный dev)
veil-cli node run                        # фоновый демон
veil-cli node show                       # статус
veil-cli node stop                       # корректная остановка
```

Несколько команд, чтобы заглянуть под капот: `node health`, `node bandwidth`,
`node metrics` и `node bootstrap-status` — последняя показывает запасные пути в
сеть на случай, если адрес стартового узла (сида) заблокируют.

### Сервер / ретранслятор (публичная точка входа)

Сервер открывает публичную «дверь» — точку входа, через которую другие узлы
впервые попадают в сеть. Возьмите профиль `censorship-target`: он слушает
`wss://0.0.0.0:443`, выставляет подставное имя сайта (decoy-SNI), чтобы
соединение выглядело как обращение к другому сайту, и включает режим сети mesh.
Заодно поднимите сложность доказательства работы (Proof of Work):

```sh
veil-cli config init --profile censorship-target --difficulty 24
# отредактируйте конфиг (адрес listen, SNI, режим [network], пути persist)
veil-cli config show
veil-cli node run
```

Для **защищённого сервера, который работает круглосуточно**, возьмите скрипт
первоначальной настройки со сборкой из исходников. Он заводит отдельного
пользователя `veil`, каталог данных `/var/lib/veil` и службу `systemd`, а в конце
печатает публичную строку подключения (join-блоб), по которой к вам подключаются
другие:

```sh
sudo PUBLIC_IP=203.0.113.10 LISTEN_PORT=443 ROLE=core \
  ./scripts/install-bootstrap.sh
```

См. [Руководство администратора](../en/admin-guide.md) и
[Operations](../en/OPERATIONS.md) — там про транспорты (способы передачи),
метрики и управление парком серверов.

> **Устойчивость к блокировкам.** Готовые сборки по умолчанию уже включают
> `tls-boring` — он постоянно меняет «отпечаток» соединения, так что дважды оно
> не выглядит одинаково. Чтобы сделать трафик как можно незаметнее, прочитайте
> [p-net.md](../en/p-net.md) и комментарии `censorship-target` в вашем конфиге.

## ogate — IP поверх veil

`ogate` переносит обычный IP-трафик через veil, и группа машин начинает работать
как одна частная локальная сеть, даже если они далеко друг от друга. Для этого он
создаёт TUN-устройство (виртуальную сетевую карту) — поэтому ему нужны права
`CAP_NET_ADMIN` или root, а на Windows — Администратор.

```sh
ogate gen-config -o ogate.toml          # шаблон с комментариями
# заполните: имя сети, node_id пиров, виртуальные IP
sudo ogate up --config ogate.toml
ogate show                              # разобранный конфиг, без открытия ресурсов
```

Полный справочник: [ogate.md](../en/ogate.md).

## oproxy — proxy-клиент и сервер

Отправляйте трафик приложений со своей машины через veil на выходной сервер —
тот делает запросы наружу от вашего имени.

**Клиент** (ваша машина — локальный SOCKS5/HTTP-proxy):

```sh
oproxy-client --gen-config > oproxy-client.toml   # задайте server_node_id + [[inbound]] listeners
oproxy-client --config oproxy-client.toml
# направьте браузер/приложение на настроенный SOCKS5/HTTP порт
```

**Сервер** (выходной узел):

```sh
oproxy-server --gen-config > oproxy-server.toml
oproxy-server --config oproxy-server.toml
```

Каждый адрес назначения можно направить по-своему — через veil, напрямую или
вовсе заблокировать — и настроить автоматический переход на запасной путь. Всё
это описано в [oproxy.md](../en/oproxy.md).

## Параметры установщика

Вот флаги, которые понимает `install.sh`. Когда передаёте скрипт в `sh` по
конвейеру (через `|`), ставьте их после `sh -s --`:

| Флаг | Значение |
|------|----------|
| `--all` | Поставить все четыре сборки |
| `--components a,b` | Только указанный набор |
| `--version X.Y.Z` | Закрепить версию (по умолчанию — самая свежая) |
| `--prefix /usr/local` | Ставить в `<prefix>/bin` (на всю систему) |
| `--bin-dir <dir>` | Ставить прямо в `<dir>` |
| `--libc musl\|gnu` | libc для Linux x86_64 (по умолчанию `musl` — статическая, работает везде) |
| `--no-modify-path` | Не трогать профиль оболочки |
| `--quickstart` | Сразу создать конфиг и запустить узел после установки |
| `-y`, `--yes` | Без вопросов, в неинтерактивном режиме |
| `--no-verify` | Пропустить проверку sha256 (не рекомендуется) |

У `install.ps1` те же параметры в виде ключей PowerShell (`-All`, `-Version`,
`-Components`, `-BinDir`, `-NoModifyPath`, `-Quickstart`, `-NoVerify`). А когда
передаёте скрипт в `iex` по конвейеру, задавайте всё через переменные окружения:
`$env:VEIL_COMPONENTS`, `$env:VEIL_VERSION`, `$env:VEIL_REPO`.

Чтобы установить из форка или зеркала, задайте на любой платформе переменную
окружения `VEIL_REPO=owner/repo`.

## Обновление

`veil-cli` умеет обновляться сам. Он берёт *манифест* — небольшой файл, который
оператор подписал, чтобы поручиться за новую версию, — и обновляется по нему:

```sh
veil-cli update check
veil-cli update apply       # проверяет подпись перед заменой бинарника
```

Либо просто запустите установщик заново — он всегда берёт самую свежую версию.
Чтобы обновить `ogate` или `oproxy`, тоже запустите установщик заново: это
обычные сервисные программы, сами себя они не обновляют.

## Проверка установленного

Перед установкой установщик сверяет отпечаток SHA-256 каждой сборки с
опубликованным `sha256-<triple>.txt`. Если захотите проверить ещё раз сами:

```sh
sha256sum ~/.veil/bin/veil-cli
# сравните с ассетом sha256-<triple>.txt на странице релиза
```

К каждому релизу также прилагается **подписанный `manifest-<triple>.bin`** — это
`UpdateManifest`, подписанный ключом релиза из холодного хранилища (он лежит
офлайн, и украсть его по сети нельзя). Если хотите убедиться, что ничего не
подменили, пересоберите сборку из помеченного тегом коммита через
`scripts/build-release.sh` и проверьте, что SHA-256 совпадает байт в байт — см.
[release.yml](../../.github/workflows/release.yml).

## Удаление

```sh
# Linux / macOS
rm -rf ~/.veil
# затем уберите строку "# veil" из ~/.profile / ~/.bashrc / ~/.zshrc
```

```powershell
# Windows
Remove-Item -Recurse -Force $env:USERPROFILE\.veil
# затем уберите каталог bin из PATH (Система > Переменные среды)
```

Данные узла — конфиг, личность и сохранённое состояние — лежат в отдельном
каталоге, который задаёт операционная система. Если хотите стереть и их, команда
`veil-cli config locate` покажет точный путь.

## Сборка из исходников

Это нужно, только если для вашей платформы нет готовой сборки (например, для
**Intel macOS**) или если вы хотите дорабатывать сам код. Сборка по умолчанию
тянет **BoringSSL (`tls-boring`)** и **RocksDB (`rocksdb-cold`)** — обе написаны
на C/C++, поэтому, чтобы их собрать, понадобится набор инструментов сборки
C/C++:

```sh
# Зависимости Debian/Ubuntu для дефолтной (BoringSSL + RocksDB) сборки:
sudo apt-get install -y cmake golang-go nasm ninja-build build-essential

git clone https://github.com/veilnetwork/veil
cd veil
cargo build --release --bin veil-cli --bin ogate --bin oproxy-client --bin oproxy-server \
  --features veil-bootstrap/production-seeds
# бинарники появятся в target/release/
```

Для подписанных воспроизводимых релизов — таких, что любой может пересобрать и
получить ровно те же байты, — используйте `scripts/build-release.sh --target <triple>`.
Чтобы собрать статическую сборку для Linux на машине с macOS, см.
`scripts/cross-build-linux-musl.sh`.

## Поддерживаемые платформы

Готовые сборки мы публикуем для перечисленных ниже целей. Каждую обозначает её
*triple* — связка из процессора, операционной системы и libc, под которую она
собрана:

| Triple | Примечания |
|--------|------------|
| `x86_64-unknown-linux-musl` | **по умолчанию для Linux x86_64** — статическая, любой дистрибутив |
| `x86_64-unknown-linux-gnu` | сборка под glibc (`--libc gnu`) |
| `aarch64-unknown-linux-gnu` | ARM64 Linux |
| `aarch64-apple-darwin` | macOS Apple Silicon |
| `x86_64-pc-windows-msvc` | Windows 10/11 (x64) |

Для Intel macOS (`x86_64-apple-darwin`) и ARM64 Windows готовых сборок нет — для
них [соберите из исходников](#сборка-из-исходников) или запускайте x64-сборку под
эмуляцией.

## Безопасно ли `curl … | sh`?

Кого-то настораживает, когда скрипт сразу передают в оболочку, — и это понятно.
Вот что этот скрипт делает на самом деле: качается по HTTPS (TLS 1.2 и новее, то
есть соединение зашифровано), перед установкой сверяет SHA-256 каждой сборки,
при обычной установке для одного пользователя не просит root и трогает только
`~/.veil` плюс ту единственную строку `PATH`, которую добавляет в профиль вашей
оболочки. Если хотите сперва прочитать — скачайте его и посмотрите перед
запуском:

```sh
curl -fsSLO https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.sh
less install.sh        # ревью
sh install.sh
```
