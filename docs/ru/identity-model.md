# Модель identity в veil

**Статус**: дизайн зафиксирован 2026-04-20.

Это эталонная спецификация слоя личности в veil. Под «личностью» здесь мы
понимаем постоянное имя узла и ключи, которые доказывают, что имя принадлежит
именно ему. Ваша личность принадлежит только вам — её не выдаёт никакая
компания, и никакой реестр не может её отобрать. Из чего она состоит:

- стабильный **`node_id`** — ваш постоянный адрес, вычисленный из вашего ключа;
- **главный ключ** (master key), который вы храните в безопасности, и
  подписанные им короткоживущие **ключи устройств** — по одному на каждый
  телефон, ноутбук или сервер;
- **делегирования** — подписанные главным ключом «разрешения», которые говорят
  «этот ключ устройства действительно мой» и перевыпускаются автоматически,
  пока не истёк их срок;
- **режим одного устройства** (standalone) для тех, у кого устройство одно, и
- **режим нескольких устройств** (multi-device), где одну личность делят между
  собой много устройств.

Каждый из этих пунктов подробно разобран ниже; список — это просто карта.

## Краткая шпаргалка

```
node_id   = BLAKE3(master_pubkey)              // стабильный, выведен из master pk
device_id = BLAKE3(device_pubkey)              // детерминированный адрес устройства;
                                                  verifier отвергает несоответствие
Delegation = IdentityKey {                     // per-device делегирование
    pubkey,                                     // Ed25519 device pubkey
    device_id,                                  // = BLAKE3(pubkey)
    valid_from_unix,
    valid_until_unix,                           // по умолчанию 7 дней; перевыпуск
                                                //   при half-validity через
                                                //   maintenance tick
    master_sig,                                 // master_sk подписывает cert
}

Standalone-режим:
  master_pk == device_pk                        // устройство И ЕСТЬ master
  ⟹ node_id == device_id == BLAKE3(device_pk)  // одиночное self-signed делегирование
```

**Отзыва ключей здесь нет** — нет способа разослать команду «отмени этот ключ
прямо сейчас». Он и не нужен. Каждое делегирование само истекает через
короткое окно (`valid_until_unix`, по умолчанию 7 дней), а исправное
устройство обновляет своё делегирование на середине этого окна. Поэтому, если
устройство украли, его делегирование просто перестают обновлять, и оно само
умирает в течение 7 дней — никаких экстренных действий не требуется.

## Цели

1. **Стабильная личность.** Ваш `node_id` переживает смену ключей, потерю
   устройства и восстановление после взлома. Зарегистрировав его, вы владеете
   им, пока сами от него не уйдёте.
2. **Свободная регистрация.** Зарегистрироваться может любой, у кого есть пара
   ключей. Нет привратника, нет центрального реестра, нет DNS. Короткий срок
   жизни делегирований делает ту же работу по сдерживанию потока, что иначе
   делала бы задачка-доказательство работы на уровне документа. (Доказательство
   работы, или PoW, — это небольшая задачка для процессора, которая делает спам
   дорогим; она по-прежнему охраняет заявки на имена, об этом ниже.)
3. **Режим одного устройства по умолчанию.** Если устройство у вас одно —
   только телефон, только ноутбук, — вам незачем проводить церемонию с главным
   ключом. При первом запуске среда выполнения тихо собирает `IdentityDocument`
   на одно устройство, где главный ключ и ключ устройства — это один и тот же
   ключ.
4. **Несколько устройств.** Одна личность, много устройств — телефон, ноутбук
   и десктоп. Каждое устройство подписывает своим ключом, а главный ключ
   ручается за этот ключ на 7 дней подряд. Автоматический перевыпуск на
   середине срока держит честные, долго работающие устройства в актуальном
   состоянии.
5. **Балансировка нагрузки.** Один и тот же `node_id` может вести к нескольким
   устройствам, выбирая одно по рейтингу, времени отклика или по тому,
   насколько недавно его видели.
6. **Ноль внешнего доверия.** Нет DNS, с которым надо сверяться, нет
   поручителей, нет восстановления через знакомых. Единственное, на что вы
   опираетесь, — криптография.
7. **Прямая секретность для асинхронных сообщений.** Если ваши ключи утекут
   позже, прошлые сообщения останутся запечатанными (для этого используются
   предключи в духе X3DH).
8. **Компрометация, ограниченная по времени.** Здесь нет живого канала отзыва,
   который надо было бы охранять. Вместо этого у каждого делегирования есть
   7-дневный `valid_until`, и главный ключ перевыпускает его на середине срока.
   Сертификат украденного устройства устаревает в течение 7 дней, что бы ни
   делал оператор.

## Концептуальная модель

### Режим нескольких устройств

```
master_seed (32 B random, BIP39 24-word backup, cold storage)
     │
     │ HKDF-SHA256(·, "veil.master.v1")
     ▼
 master_sk  ─ подписывает делегирования (~еженедельно через auto-reissue,
     │      ad-hoc на событиях delegate-device + rotate-device)
 master_pk  (Ed25519 по умолчанию; Falcon-512 — opt-in для post-quantum)
     │
     │ BLAKE3(master_pubkey)            ← голый хэш, без domain tag
     ▼
 node_id  [u8; 32]   ← СТАБИЛЕН НАВСЕГДА
     │
     ├── IdentityDocument (DHT-запись)
     │     ├── master_pubkey (plaintext; verifier пересчитывает node_id отсюда)
     │     ├── identity_keys[]  — per-device делегирования (≤ 8), каждое
     │     │                       master-сертифицировано со своим коротким
     │     │                       valid_until_unix (по умолчанию 7 дней)
     │     ├── document_sig активным subkey'ем (sig_key_idx)
     │     └── (нет revocation list, нет master_freshness_sig, нет PoW —
     │           короткая валидность делегирований + частая republish — это
     │           и есть механизм freshness)
     │
     ├── InstanceRegistry (отдельная DHT-запись)
     │     └── подписана любым активным identity_sk
     │
     ├── NameClaim-записи (отдельно на каждое имя)
     │     └── подписаны любым активным identity_sk
     │
     └── PrekeyBundle (X3DH, per-device)
           └── ML-KEM ephemeral + fallback ключи, master-сертифицированы
```

### Режим одного устройства

```
device_sk_seed (32 B Ed25519 seed, сгенерирован runtime'ом или из конфига [identity])
     │
 device_sk = master_sk = identity_sk    ← все три — один и тот же ключ
     │
 device_pk = master_pk = identity_pk    ← все три — один и тот же pubkey
     │
     │ BLAKE3(device_pubkey)
     ▼
 node_id == device_id   [u8; 32]        ← коллапсируют; одиночный subkey
                                          покрывает обе роли
     │
     ├── IdentityDocument (DHT-запись)
     │     ├── master_pubkey == identity_keys[0].pubkey
     │     ├── identity_keys[0]: self-signed делегирование
     │     │   (master_sig произведён device_sk'ом, выступающим в роли master)
     │     └── document_sig одиночным subkey'ем
```

Байты на проводе в обоих режимах совершенно одинаковы. Сторонний наблюдатель
не отличит документ одного устройства от только что созданного документа на
несколько устройств, у которого пока одно делегирование: и там, и там
`master_pubkey` равен `identity_keys[0].pubkey`.

**Инварианты, которые стоит запомнить**:

- Ваш `node_id` меняется **только** если вы потеряли главное зерно (master
  seed) — 32-байтный секрет, из которого выводится всё остальное. Это
  катастрофа — потеря того же рода, что и утрата зерна Bitcoin-кошелька.
- В режиме нескольких устройств главное зерно живёт **только на вашем основном
  устройстве**. Каждое остальное устройство получает свой независимый ключ
  устройства — через сопряжение или командой `identity delegate-device`, — и
  главный ключ его сертифицирует.
- В режиме одного устройства отдельного главного зерна нет вовсе — ключ
  устройства *и есть* главный ключ. Делегирование обновляет само себя примерно
  каждые 3.5 дня на служебном тике, и делать вам для этого ничего не нужно.
- Делегирование каждого устройства держится само по себе. Если одно из них
  скомпрометировано, оно просто истекает в течение 7 дней; главный ключ
  попросту перестаёт его обновлять.
- Несколько устройств под одним `node_id` — это основной сценарий, а не
  редкий случай: именно на нём работают балансировка нагрузки и мессенджер на
  нескольких устройствах.
- veil держит по одной сессии на пару `(node_id, instance_id)` для каждого
  собеседника. Этот `active_instance_id` — 16-байтная заглушка, оставленная
  ради обратной совместимости и взятая из первой половины `device_id`
  (`device_id[..16]`); новый код должен предпочитать полный 32-байтный
  `device_id`. (Замечание: эта детерминированная прокладка-совместимости
  отличается от локально сохранённого *случайного* `instance_id` в
  `~/.config/veil/instance_id`, который адресует `Recipient::Specific` —
  см. [multi-device.md](multi-device.md).)

## Криптографические примитивы

| Назначение | Примитив |
|---|---|
| Хэш (привязка личности, PoW, обязательства) | BLAKE3-256 |
| Долгосрочная подпись | Ed25519 (по умолчанию), Falcon-512 или постквантовые гибриды Ed25519+Falcon-512 / Ed25519+Falcon-1024 (рекомендуются для долгоживущих личностей) |
| Получение ключа из главного зерна | HKDF-SHA256 |
| Симметричное шифрование | ChaCha20-Poly1305 |
| Кодирование резервной копии главного зерна | BIP39 (английский, 24 слова) |
| Парольный KDF (зашифрованный файл главного ключа) | Argon2id |

Каждая подпись привязана к именованной строке-контексту, поэтому подпись,
сделанную для одной цели, нельзя переиграть так, будто она означает другую.
Обратите внимание, чего здесь *нет*: нет ни `REVOKE_CONTEXT`, ни
`FRESHNESS_CONTEXT`, потому что нет живого отзыва и нет отдельной подписи
актуальности. Единственное, что говорит «это ещё действует», — это
`valid_until_unix` самого документа и собственный `valid_until_unix` каждого
ключа:

```rust
const CERTIFY_CONTEXT: &[u8] = b"veil.certify.v1";
const DOC_SIG_CONTEXT: &[u8] = b"veil.identity_doc.v1";
const PAIRING_INVITE_SIG_CONTEXT: &[u8] = b"veil.pairing_invite.v1";
const PREKEY_BUNDLE_SIG_CONTEXT:  &[u8] = b"veil.prekey_bundle.v1";
```

### Получение главного ключа

```
master_sk = HKDF-SHA256(
    salt: None,
    ikm:  master_seed,  // 32 байта
    info: b"veil.master.v1",
    len:  32 (Ed25519) или 48 (размер seed'а Falcon-512)
)
```

### Привязка node_id

```
node_id = BLAKE3(master_pubkey)        // 32 байта; голый хэш, без domain tag
```

Это обычный BLAKE3-хэш без дополнительной метки — BLAKE3 — это быстрая,
современная хеш-функция, которую veil использует повсюду. Это в точности
совпадает с тем, как `cfg::NodeId::from_public_key` вычисляет адрес в среде
выполнения. В режиме одного устройства один и тот же открытый ключ подаётся в
обе формулы, поэтому `node_id == device_id == BLAKE3(device_pubkey)` —
байт в байт.

Могут ли два разных алгоритма подписи дать один и тот же `node_id` — скажем,
ключ Ed25519 и ключ Falcon-512, попавшие в один и тот же результат? На
практике — нет. BLAKE3 выдаёт 256-битное значение, и вдобавок байт алгоритма
лежит внутри окружающего сертификата `IdentityKey`, который verifier проверяет
отдельно.

### Привязка device_id

Каждое делегирование устройства несёт поле `device_id` прямо в открытом виде,
и verifier выбрасывает любой сертификат, где это поле не совпадает с хэшем
ключа:

```
device_id = BLAKE3(device_pubkey)      // 32 байта; такая же форма, как у node_id
```

Раз адрес — это просто хэш ключа, любой может пересчитать его прямо из данных
на проводе и убедиться в нём; верить отправителю на слово ни в чём не нужно.

## Формат IdentityDocument на проводе

DHT-ключ: `BLAKE3("veil.identity_dht.v1" || node_id)`.

Настоящее определение лежит в `crates/veil-proto/src/identity_document.rs`;
схема ниже приведена для чтения, а не как источник истины.

```
Layout (canonical-байты — все целые big-endian, если не указано иное):
[0..2]       magic = "ID"                     u16 BE ('I'=0x49, 'D'=0x44)
[2]          version = 1                       u8
[3..35]      node_id                           [u8; 32]
[35]         master_algo                       u8 (0=Ed25519, 2=Falcon-512, 3=Ed25519+Falcon-512, 4=Ed25519+Falcon-1024)
[36..38]     master_pubkey_len                 u16 BE
[38..38+L]   master_pubkey                     [u8; L]  (L=32 или 897)
[...]        issued_at_unix                    u64 BE
[...]        valid_until_unix                  u64 BE   (≤ issued_at + 30d)
[...]        sig_key_idx                       u16 BE
[...]        identity_keys_count               u8
[...]        для каждого IdentityKey:          varies (см. ниже)
[last]       document_sig_len                  u16 BE
[last]       document_sig                      [u8; S]
```

**Жёсткие лимиты**, проверяемые в момент разбора документа:
- не более 8 ключей устройств (`identity_keys_count ≤ MAX_IDENTITY_KEYS = 8`);
- окно актуальности не может превышать 30 дней
  (`valid_until_unix - issued_at_unix ≤ MAX_FRESHNESS_WINDOW_SECS = 30 дней`);
- весь документ должен уместиться в 16 КиБ
  (`MAX_IDENTITY_DOCUMENT_BYTES = 16384` байт). Этого потолка хватает, чтобы
  вместить полностью провёрнутый гибридный документ Falcon, и он совпадает с
  самым большим значением, которое хранит DHT.

Обратите внимание, чего в документе нет. Нет ни защиты от повторов
`document_version`, ни `revocation_seq` / `revoked_keys[]` / `RevocationEntry`,
ни `freshness_hour` / `master_freshness_sig` / `pow_nonce` на уровне документа,
ни `extensions_root`. Эту работу вместо них делает короткий срок жизни
делегирований, и единственное, что отмечает документ как актуальный, — его
собственный `valid_until_unix`.

### `IdentityKey` (делегирование устройства)

```
[0]           algo                       u8
[1..3]        pubkey_len                 u16 BE
[3..3+L]      pubkey                     [u8; L]
[...]         device_id                  [u8; 32]   (= BLAKE3(pubkey))
[...]         valid_from_unix            u64 BE
[...]         valid_until_unix           u64 BE     (per-key срок,
                                                     по умолчанию issued_at + 7 дней)
[...]         master_sig_len             u16 BE
[...]         master_sig                 [u8; S]
```

`master_sig` покрывает:
```
CERTIFY_CONTEXT
|| node_id
|| algo
|| len(pubkey) as u16 BE
|| pubkey
|| device_id
|| valid_from_unix
|| valid_until_unix
```

### Подпись документа

`document_sig` — это подпись по каноническим байтам всех полей выше, кроме
самих `document_sig_len` и `document_sig`:

```
DOC_SIG_CONTEXT || canonical_bytes_up_to_doc_sig
```

Подписывает тот ключ устройства, который сейчас активен и на который указывает
`sig_key_idx`. В режиме одного устройства этот активный ключ *и есть* главный,
так что `document_sig` и единственный `IdentityKey.master_sig` происходят от
одного и того же ключа.

## Алгоритм verifier'а

На вход он принимает две вещи: документ для проверки (`doc: IdentityDocument`)
и текущее время (`now_unix_secs: u64`). Вот что он делает по порядку.
Настоящий код лежит в `crates/veil-identity/src/verify.rs`.

1. Magic `"ID"` и версия.
2. Пересчитать `node_id = BLAKE3(master_pubkey)`, отвергнуть при
   несовпадении.
3. Проверить `now ≤ doc.valid_until_unix` (окно актуальности документа).
4. Для каждого `IdentityKey`:
   - **4a.** `device_id == BLAKE3(pubkey)` (детерминированная
     привязка). Отвергнуть `DeviceIdMismatch`.
   - **4b.** `now ≤ key.valid_until_unix` (срок этого делегирования).
     Отвергнуть `KeyExpired`.
   - **4c.** Проверить `master_sig` через `master_pubkey` по
     `CERTIFY_CONTEXT || node_id || algo || len(pubkey) || pubkey ||
     device_id || valid_from || valid_until`.
5. Проверить, что `sig_key_idx` в допустимых границах.
6. Проверить `document_sig` через `identity_keys[sig_key_idx]` по
   `DOC_SIG_CONTEXT || canonical_signing_bytes()`.

Возвращает `ValidatedIdentity {
  node_id,
  master_algo, master_pubkey,
  active_identity_pubkey, active_identity_algo, active_key_idx,
  active_device_id,                  // детерминированный адрес устройства
  active_instance_id,                // compat-прокладка: device_id[..16]
}`.

Обратите внимание на три вещи, которых verifier не делает никогда: он не
заглядывает в сохранённый список отзыва, не проверяет никакого доказательства
работы на уровне документа и не ищет отдельную подпись актуальности. Ничего из
этого не существует.

## Операции жизненного цикла

### Создание — несколько устройств (`veil-cli identity create`)

1. Сгенерировать `master_seed = OsRng::gen(32)`.
2. Вывести `master_sk`, `master_pk`; вычислить `node_id =
   BLAKE3(master_pk)`.
3. Сгенерировать `identity_sk_0` первого устройства (эфемерный
   случайный ключ, не выведенный из главного).
4. Вычислить `device_id_0 = BLAKE3(identity_pk_0)`.
5. `master_sig_0 = master_sk.sign(CERTIFY_CONTEXT || node_id || algo
   || len(identity_pk_0) || identity_pk_0 || device_id_0
   || valid_from || valid_until)`. По умолчанию `valid_until =
   now + 7 дней`.
6. Собрать IdentityDocument с одной записью IdentityKey.
7. `document_sig = identity_sk_0.sign(DOC_SIG_CONTEXT || canonical)`.
8. Показать BIP39-фразу; пользователь записывает её.
9. По желанию сохранить зашифрованный файл главного ключа (Argon2id +
   ChaCha20-Poly1305).
10. Опубликовать IdentityDocument в DHT (среда выполнения делает это при
    первом старте).

### Создание — одно устройство (`veil-cli identity standalone`)

1. Сгенерировать `device_sk_seed = OsRng::gen(32)`.
2. `device_pk = derive(device_sk)`; `node_id = device_id =
   BLAKE3(device_pk)`.
3. Самоподписанное делегирование: device_sk выступает в роли master_sk.
   `master_sig = device_sk.sign(CERTIFY_CONTEXT || node_id || ... )`.
4. `document_sig = device_sk.sign(DOC_SIG_CONTEXT || canonical)`.
5. Записать `identity_document.bin` + `device_identity_sk.bin`.

Эти шаги вы редко выполняете вручную. При первом запуске, если
`identity_document.bin` ещё нет, а в конфиге `[identity]` уже есть пара ключей
Ed25519, среда выполнения сама проходит шаги с 1 по 5. Это путь
автоматической первичной настройки (auto-bootstrap).

### Автоматический перевыпуск на середине срока

Делегирование, которое обновляют, только когда оно вот-вот умрёт, оставляло бы
слишком мало запаса. Поэтому служебный цикл проверяет его на каждом тике
очистки (по умолчанию примерно раз в 30 секунд) и обновляет на середине срока:

1. Прочитать активный `IdentityKey.valid_until_unix`.
2. Если `now + DELEGATION_VALIDITY_SECS / 2 < valid_until` — ничего не делать
   (осталось больше половины окна).
3. Иначе, в **режиме одного устройства**, среда выполнения переподписывает
   на месте: `master_sk == device_sk == self.identity_sk` уже в памяти. Новое
   `valid_until = now + 7 дней` (снова полное окно).
4. В **режиме нескольких устройств** тик ничего не делает, так как главный
   ключ живёт на другом устройстве. Здесь вы обновляете вручную. До истечения
   текущего делегирования запустите `veil-cli identity delegate-device
   --pubkey-file ... --validity 7d` на главном устройстве. Затем перенесите
   обновлённый документ на нужное устройство как вам удобно — флешкой, по
   `scp` или QR-кодом — и положите его в `<veil_dir>/identity_document.bin`.
   Среда выполнения следит за временем изменения этого файла (опрос раз в
   60 секунд в `runtime/sovereign_republish.rs`), замечает правку и заново
   публикует новый документ в DHT.

### Сопряжение нового устройства (церемония с QR-кодом)

Чтобы добавить второе устройство, оба устройства проводят короткий обмен через
QR-код. Рабочую реализацию см. в
`crates/veil-cli/src/cmd/sovereign_identity.rs::{pair_invite, pair_listen,
pair_accept}`.

### Что делать при компрометации

Допустим, устройство украли. Живого канала отзыва, чтобы крикнуть «отмени этот
ключ», здесь **нет**. Вместо этого решение наступает само собой:

1. Вы перестаёте обновлять делегирование скомпрометированного устройства. В
   режиме нескольких устройств это значит, что вы просто больше не запускаете
   `delegate-device` для его ключа. В режиме одного устройства, если утёк сам
   главный секретный ключ, вы проворачиваете ключ устройства командой
   `identity standalone --force`.
2. В течение 7 дней сертификат устаревает по мере того, как его
   `valid_until_unix` уходит в прошлое. С этого момента verifier'ы его
   отвергают (шаг 4b, `KeyExpired`), и собеседники перестают принимать от
   этого устройства рукопожатия.
3. Сессии, уже идущие со скомпрометированным устройством, продолжаются, пока
   не наступит их собственный срок смены ключа.

Размен очевиден: вы ждёте до 7 дней вместо отмены за минуты, а взамен протокол
остаётся компактным — нет рассылки отзывов, нет кэша отзыва, нет поля
`revoked_keys[]`, нет `master_freshness_sig`.

## Поиск имён

Имя позволяет удобному для человека прозвищу заменить собой `node_id`. В именах
используются только ASCII-символы `[a-z0-9#_-]`, и регистр не учитывается —
перед хэшированием всё приводится к нижнему регистру:

```
name_dht_key = BLAKE3("veil.name_claim_dht.v1" || u16_be(len(name)) || name.as_bytes())
```

Значение NameClaim содержит:
- строку имени
- node_id
- встроенное `cert_proof` (master_pubkey + master_sig по подписывающему
  identity_pubkey) для проверки в офлайне
- signing_identity_key_idx
- подпись любым активным identity_sk
- одноразовое число (nonce) доказательства работы, пропорциональное редкости

Найти имя — короткая цепочка: распознаватель достаёт NameClaim, читает из него
`node_id`, достаёт IdentityDocument этого узла, проверяет цепочку сертификатов
и в итоге получает `ValidatedIdentity`.

Стоимость доказательства работы растёт тем выше, чем имя реже — а значит и
ценнее, — и именно это не даёт массово расхватывать короткие, желанные имена:
- 1–3 буквенных ASCII-символа: 28–30 (часы работы процессора)
- 4–6 символов: 22–26 (минуты)
- 7–12 символов: 18–22 (секунды-минуты)
- С дискриминатором (`alice#1234`): 14 (~1 с)
- Длинное случайное (`AnonXYZ_7a3bf2`): 12 (~1 с)

## InstanceRegistry

Это небольшая отдельная DHT-запись, которая перечисляет, какие устройства
сейчас в сети. Она обновляется всякий раз, когда устройство поднимается или
отваливается, и остаётся компактной — обычно меньше 2 KB. Настоящее
определение лежит в `crates/veil-proto/src/instance_registry.rs`.

```
DHT-ключ = BLAKE3("veil.instances_dht.v1" || node_id)
```

Пара вещей отсутствует намеренно: здесь нет блока `tier_b`, и ни
`mailbox_anchor`, ни зашифрованные подсказки контактов сюда не входят.
Подсказки о том, как достучаться до устройства, передаются отдельно — в gossip-
сообщениях `SignedTransportAnnouncement`. Вот текущая форма:

```
InstanceRegistry {
    node_id:                  [u8; 32],
    instances:                Vec<InstanceEntry>,   // ≤ 16
    reg_version:              u64,                    // монотонный
    created_at_unix:          u64,
    signing_identity_key_idx: u16,                    // индекс в IdentityDocument
    sig:                      Vec<u8>,                // любой активный identity_sk
}

InstanceEntry {
    instance_id:              [u8; 16],   // обрезанный device_id (compat-прокладка)
    bound_identity_key_idx:   u16,        // → IdentityDocument.identity_keys[i]
    label:                    String,     // ≤ 32 B, опционально
    last_seen_ms:             u64,        // грубая гранулярность
}
```

## Маршрутизация: InstanceTag

Адресуя сообщение, вы указываете *кому* (`node_id`) и *на какое устройство*
(метку `InstanceTag`). Именно метка позволяет одной личности разойтись на много
устройств. Вот как `Recipient` выглядит на проводе:

```
Recipient {
    node_id:      [u8; 32],
    instance_tag: InstanceTag,
}

enum InstanceTag {
    Any,                    // load-balanced — veil выбирает одно устройство
    All,                    // fan-out broadcast — все устройства
    Specific([u8; 16]),     // direct — точный instance_id (= device_id[..16])
}
```

Замечание по реализации: внутри среда выполнения по-прежнему опирается на
16-байтный `instance_id` (первую половину полного 32-байтного `device_id`),
когда устраняет дубликаты сессий и направляет доставку.

## Модель угроз

Этот раздел разбирает, какие атаки veil останавливает, какие оставляет на вас
и какие намеренно не пытается обрабатывать. О повседневной стороне безопасности
см. [`docs/recovery.md`](recovery.md) и
[`docs/opsec-user-guide.md`](opsec-user-guide.md).

### От чего veil защищает

В каждой строке ниже названа атака и защита, которая её отбивает:

- Угон `node_id` через атаку нахождения прообраза на BLAKE3 — невозможно
  (2^256).
- Подделка `device_id` — verifier отвергает любой сертификат, где
  `device_id != BLAKE3(pubkey)` (детерминированная привязка).
- Утечка повседневного `identity_sk` (вредонос, украденный ноутбук) —
  скомпрометированный сертификат устаревает в течение 7 дней, поскольку
  главный ключ перестаёт его перевыпускать.
- Захват чужих имён — доказательство работы, пропорциональное редкости
  (по-прежнему в слое NameClaim).
- Фишинг при сопряжении по QR — код подтверждения по отдельному каналу,
  который сверяют обе стороны.
- Наводнение обновлениями документа со скомпрометированного ключа — квота DHT
  на каждую личность.
- Атака затмения (eclipse) на поиск личности — кворум из нескольких реплик.
- Прямая секретность для прошлых асинхронных сообщений — одноразовые предключи
  X3DH.

**Что veil намеренно не пытается делать**:
- Реагировать на компрометацию быстрее 7 дней. Используйте `identity
  standalone --force` (при компрометации главного ключа на одном устройстве)
  или `identity rotate` и дождитесь естественного 7-дневного окна (несколько
  устройств).
- Защищать от атак на устаревший отзыв. Канала отзыва нет, поэтому устаревать
  нечему.

### Что лежит на вас (ваша OpSec)

Это не ошибки, которые veil может исправить, — это то, что зависит от того, как
вы бережёте свои секреты:

- Потеря физической резервной копии `master_seed` — личность потеряна
  навсегда (как с зерном Bitcoin-кошелька).
- Компрометация хранения главного ключа (физическая кража бумаги,
  несанкционированный доступ к разблокированному зашифрованному файлу) —
  полный захват, в рамках этой личности невосстановимо. Что помогает: бумага
  в банковской ячейке, аппаратные ключи, стойкие парольные фразы.
- Социальная инженерия (пользователь сам раскрывает BIP39-фразу) — описано в
  opsec-user-guide.md как предупреждения; протокол этому помешать не может.
- Вредонос на устройстве во время показа BIP39, перехват нажатий клавиш при
  вводе паролей — это ответственность за безопасность самого устройства.
- Смена алгоритма, когда придёт квантовая угроза — `master_algo` и алгоритм
  каждого подключа позволяют перейти на Falcon-512 или гибрид
  Ed25519+Falcon-512/1024, но запускает это пользователь.

## Гибкость алгоритмов

И `master_algo: u8`, и `IdentityKey.algo: u8` у каждого ключа хранят, каким
алгоритмом подписи этот ключ пользуется, поэтому одна личность может смешивать
алгоритмы и со временем мигрировать. Вы можете:

- Начать с личности на Ed25519.
- Позже добавить подключи Falcon-512 (главный ключ их сертифицирует).
- В итоге перевести главный ключ на постквантовый алгоритм через повторный
  выпуск (новый master_sk, новый node_id; имена и репутация требуют
  доказательства миграции или повторной заявки — это будущая работа).
