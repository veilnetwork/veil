# Как работает veil-сеть

> Это дружелюбный, обзорный рассказ: общая картина, из чего сеть состоит и
> как сообщение на самом деле доходит от одного человека к другому. Если нужно
> полное описание на уровне исходников (форматы байтов на проводе, каждая
> константа, правила блокировок, как подсистемы взаимодействуют между собой) —
> оно в [ARCHITECTURE_FULL.md](ARCHITECTURE_FULL.md).
>
> Совсем впервые видите Veil? Начните с [start-here.md](start-here.md) — там
> объяснены базовые слова. Эта страница считает, что вы их уже знаете.

## Общая картина

Veil — это сеть без компании посередине. Она децентрализованная (ею не владеет
ни один центральный сервер), каждое сообщение зашифровано «от двери до двери»
(прочитать его могут только отправитель и конечный получатель), и сеть сама
находит для вас людей — даже когда они сидят за домашним роутером или ненадолго
ушли в офлайн.

Сообщение проходит через короткую цепочку узлов — работающих копий Veil —
примерно так:

```
App ←→ Leaf ←→ Core ←→ Core ←→ Core ←→ Leaf ←→ App
```

Ваше приложение разговаривает с ближайшим узлом, тот передаёт сообщение дальше
другим узлам, а последний вручает его приложению вашего друга.

Бо́льшую часть работы делают несколько идей, и дальше страница разбирает каждую
по очереди:

- **Сквозное шифрование** запечатывает каждое сообщение (ML-KEM-768 для обмена
  ключами, ChaCha20-Poly1305 для самого содержимого), так что промежуточные узлы
  видят лишь зашифрованные байты.
- **DHT-маршрутизация** позволяет сети найти кого угодно примерно за `O(log N)`
  переходов — буквально за горстку, даже когда узлов миллионы, — пользуясь общим
  справочником под названием Kademlia DHT (о нём ниже).
- **Обход NAT** автоматически пробивается сквозь домашние роутеры, чтобы двое
  людей за файрволами всё равно могли связаться.
- **Локальное обнаружение в mesh** позволяет соседним устройствам находить друг
  друга напрямую, даже без интернета.
- **Доставка офлайн** оставляет сообщение в почтовом ящике (mailbox), пока
  получателя нет, и вручает его, как только тот вернётся.

## Два вида узлов

Ролей всего две, и разница между ними простая: сколько работы узел делает за
всех остальных.

| Роль | DHT | Relay | Mailbox | Gateway | Типичная среда |
|------|-----|-------|---------|---------|----------------|
| **Leaf** | - | - | - | - | Мобильный телефон, IoT-сенсор |
| **Core** | yes (K=20) | yes | yes | yes | Сервер, VPS, домашний сервер |

**Leaf** (лист) — лёгкий узел: ваш телефон, ноутбук, крошечный сенсор. Он
входит в сеть, отправляет и принимает собственный трафик, но не несёт сеть на
своих плечах.

**Core** (ядро) — полноценный узел, и каждое ядро тянет свою долю наравне с
другими. Все они помогают с четырьмя общими делами:

- **DHT** — хранят свою часть общего справочника.
- **Ретранслятор (relay)** — передают дальше сообщения, конечный получатель
  которых не они сами.
- **Почтовый ящик (mailbox)** — держат сообщения для тех, кто сейчас офлайн.
- **Шлюз (gateway)** — ведут *записи о привязке* (attachment records) —
  пометки о том, через какое ядро сейчас доступен тот или иной лист.

Чтобы поднять ядро, нужно немного доказательства работы — минимум 24 бита,
быстрая одноразовая задачка, из-за которой поддельные личности дороги (см.
[Личность и доказательство работы](#личность-и-доказательство-работы)). Если вы
предпочитаете, чтобы ядро не работало как шлюз, можно отключить именно это дело
через `[gateway] enabled = false`.

Это единственные две роли. Если вам попадутся старые имена вроде `Relay`,
`Gateway` или `CoreRouter` — это история: частью протокола они не являются.

## Личность и доказательство работы

Прежде чем войти в сеть, узел заводит себе личность. **Личность** — это просто
пара ключей (открытый ключ, который видят все, и закрытый, который держите
только вы) плюс небольшое доказательство, что вы немного потрудились над её
созданием. Из открытого ключа Veil вычисляет постоянный адрес узла (его
`node_id`). Вот рецепт:

```
keygen(Ed25519 | Falcon512) → (pubkey, privkey)
mine_nonce(pubkey, privkey, difficulty=24) → nonce
node_id = BLAKE3(pubkey)
identity_proof = (pubkey, nonce, sign(pubkey, nonce))
```

Подписывать можно либо алгоритмом **Ed25519** (маленький и быстрый), либо
**Falcon-512** (крупнее, зато устойчив к будущим квантовым компьютерам). Оба —
полноправные: вы выбираете для каждого узла через `[identity] algo`, где также
доступны гибриды `ed25519+falcon512` и `ed25519+falcon1024`, подписывающие сразу
обоими. Что бы вы ни выбрали, хеширование открытого ключа через BLAKE3 даёт один
и тот же 32-байтный `node_id`.

**Доказательство работы** — это та самая маленькая задачка из рецепта
(`mine_nonce`): вы подбираете число, при котором хеш получается определённого
вида. Она тратит немного процессорного времени, и в этом весь смысл: штамповать
одноразовые личности дорого для спамеров и пустяк для честного пользователя,
создающего одну. Базовая сложность — 24 бита (в debug-сборках понижена до 16,
чтобы тестирование оставалось быстрым). По мере роста сети она подстраивается
сама по формуле `24 + ceil(log2(N / 100K))`, где `N` — размер сети, о котором
узлы договариваются через DHT-записи на эпоху.

## Рукопожатие (OVL1)

Когда два узла соединяются впервые, они проходят **рукопожатие** — короткий
обмен репликами, в котором здороваются, доказывают, кто они такие,
договариваются о том, что каждый поддерживает, и вырабатывают общий секретный
ключ. После рукопожатия всё, что они шлют, зашифровано. Этот обмен (он называется
OVL1) выглядит так:

```
Client → Server: Hello(magic="OVL1", version=1, node_id)
Server → Client: Hello
Client → Server: Identity(algo, pubkey, nonce, node_id, mlkem_ek?)
Server → Client: Identity
Client → Server: Capabilities(role_bits, flags, max_frame, ovl1_minor=1)
Server → Client: Capabilities
Client → Server: KeyAgreement(X25519_pubkey)
Server → Client: KeyAgreement(X25519_pubkey)
  [HKDF-SHA256 → tx_key, rx_key, session_id]   (lex-order swap of tx/rx)
Client → Server: SessionConfirm(session_id, HMAC)
Server → Client: SessionConfirm
  [Все последующие frame'ы: ChaCha20-Poly1305 AEAD encrypted]
```

Если читать шаги сверху вниз: стороны здороваются, предъявляют свои личности,
сверяют возможности, а затем каждая шлёт одноразовое открытое значение X25519. Из
этих двух значений обе стороны независимо выводят одну и ту же пару ключей (один
для отправки, другой для приёма) через HKDF-SHA256, убеждаются, что они совпали,
и с этого момента каждый кадр запечатан с помощью ChaCha20-Poly1305.

Одну деталь стоит отметить отдельно: ключ инкапсуляции ML-KEM-768 —
постквантовый ключ, который используется для сквозного шифрования, — едет внутри
`IdentityPayload` (1184 байта; длина `mlkem_pk_len=0` просто означает, что этот
узел такого ключа не публикует). А вот ключи, которые защищают *само это
соединение*, берутся из обмена X25519 плюс HKDF-SHA256. То есть ML-KEM сегодня
*не* применяется для защиты самого канала — только для сквозного запечатывания
содержимого сообщений.

Ни один ключ не вечен. Соединение тихо договаривается о свежих ключах —
выполняет **смену ключей** (rekey), — как только достигнут любой из трёх
пределов: отправлено 128 GiB кадров, прошло 32 дня или счётчик nonce пошёл по
кругу. Пределы по объёму и по времени можно настроить через
`[session] rekey_bytes_threshold` и `rekey_time_threshold_secs`.

## Как обрабатывается кадр

Всё, что идёт по проводу, приходит в виде **кадра** (frame) — одного
самодостаточного пакета с небольшим заголовком. Когда кадр приходит, узел читает
заголовок, расшифровывает тело и направляет кадр нужному обработчику по его
*семейству* (family) — категории, к которой он относится. Всё из незнакомого
узлу семейства просто игнорируется — именно это позволяет протоколу расти, не
ломая более старые узлы. Семейства:

```
bytes → FrameHeader decode → AEAD decrypt → family switch:
  Session  → Hello/Identity/Capabilities/KeyAgreement/SessionConfirm, Rekey, Ticket, Padding
  Control  → Ping/Pong, NatProbe*, Keepalive, Backpressure, Epidemic
  Discovery→ FindNode, FindValue, Store, Delete, Attachment, Mailbox/AppEndpoint lookup
  Delivery → Forward, Mailbox PUT/Fetch/Ack, Transit, RecursiveRelay, Chunks
  Routing  → RouteAnnounce/Withdraw (+Aliased), RouteRequest/Response, PoW, RouteDiscover
  App      → AppOpen, AppData, AppRtData, AppReceipt, AppWindowUpdate
  Mesh     → MeshBeacon, MeshForward, MeshAck
  PeerExchange → Walk, Challenge, Response, Result
  Tunnel   → IpPacket (TUN/TAP)
  RelayChain → Hop (onion)
  Diag     → DiagPing/Pong, TraceProbe/Hop
  Unknown  → ignored (forward-compatible)
```

## Как находится маршрут: gossip плюс DHT

Чтобы отправить сообщение, узлу сначала надо понять, какому соседу его передать.
**Маршрутизация** — это и есть способ это выяснить, и Veil сочетает два подхода.

Первый — **gossip** (молва): узнав маршрут, узел сообщает о нём ближайшим соседям
через `ROUTE_ANNOUNCE`. У таких объявлений есть *TTL* (time to live — маленький
счётчик переходов, который не даёт им расходиться слишком далеко), здесь равный
2, так что весть проходит всего на шаг-другой и сеть не захлёбывается.

Второй вступает в дело, когда gossip не подсказал ответ заранее — это *промах
кеша* (cache miss). Тогда узел обращается к общему справочнику, к DHT. Он
оборачивает сообщение в `RecursiveRelay` и шлёт его тому DHT-узлу, чей
идентификатор математически ближе всех к получателю (ближе по расстоянию XOR — о
нём в разделе про DHT). Каждый узел по пути задаёт один и тот же вопрос: есть ли
у меня живое соединение прямо до получателя? Если да — доставляет; если нет —
передаёт на шаг ближе. Если за 20 переходов живой путь так и не нашёлся,
сообщение откатывается в почтовый ящик и ждёт там.

```
A announces → B (TTL=1) → C (TTL=0, stop)
A → route cache miss → RecursiveRelay(dst=D)
  → closest node X → есть ли у X сессия до D? → доставлено!
  → нет → forward в ещё более близкий Y → ... → mailbox fallback
```

Приятная мелочь: когда доставка через `RecursiveRelay` удаётся, каждый узел
запоминает путь обратно, кешируя `originator → peer_id`. Так у ответа маршрут уже
готов и искать заново не приходится.

## Доставка сообщения: три пути

Когда вы что-то отправляете, Veil пробует самый дешёвый рабочий маршрут и при
необходимости откатывается к следующему. Путей три — от самого быстрого к самому
терпеливому.

**Путь 1 — Напрямую** (маршрут уже известен, *попадание в кеш*). Узел находит
получателя в своём кеше маршрутов, узнаёт следующий переход, и сообщение
отправляется прямиком туда:
```
Sender → FORWARD(dst) → route_cache.lookup(dst) → next_hop → ... → Recipient
```

**Путь 2 — Через DHT** (маршрута в кеше нет, *промах кеша*). Узел вместо этого
спрашивает общий справочник, перескакивая всё ближе и ближе, пока не дойдёт до
узла с живым соединением до получателя — не более чем за 20 переходов:
```
Sender → FORWARD(dst) → cache miss → RecursiveRelay(dst, hop=20)
  → DHT hop chain → узел с живой сессией до dst → доставлено
```

**Путь 3 — Почтовый ящик** (получатель офлайн). Когда напрямую не дотянуться ни
до кого, сообщение оставляют в почтовом ящике, чтобы забрать позже. Здесь чуть
больше подвижных частей, поэтому шаги расписаны ниже:
```
Sender → MAILBOX_PUT → Primary (attachment gateway получателя из DHT)
  Primary:
    хранит локально
    select_quorum_replicas:
      shard_target = BLAKE3("shard" || recipient_id || shard_id)
      pool         = DHT.find_closest_nodes(shard_target, (replica_count-1)*4)
      отфильтровать self, origin, low-battery, unreliable relays
      взять        replica_count - 1 replicas
    MAILBOX_REPLICATE → реплики (envelope encrypted for privacy)
    ждём write_quorum DeliveryStatus::QUEUED → ACK sender'у

Recipient выходит online:
  MAILBOX_FETCH → primary gateway
    локальный store → DHT fallback → fan-out MAILBOX_FETCH_REPLICA на реплики
    SEC check: recipient_node_id == authenticated peer_id
```

Самое хитрое здесь в том, что хранители резервных копий (**реплики**) выбираются
по формуле, а не по договорённости — выбор **детерминированный**. Любое ядро,
которое видит DHT, вычисляет тот же `shard_target` и приходит к тому же набору
ближайших реплик. Поэтому отправителю и получателю вообще не нужно обмениваться
адресами хостов или о чём-то договариваться заранее: математика указывает обоим
на одни и те же почтовые ящики.

## Общий справочник (DHT)

**DHT** (distributed hash table — распределённая хеш-таблица) — это общий
справочник сети. Целиком его не держит ни один узел: у каждого своя часть, а
вместе они умеют ответить на вопрос «где сейчас этот узел?». Veil использует схему
Kademlia, и работает она благодаря нескольким идеям.

**Расстояние — это математика.** Kademlia измеряет, насколько «близки» два
идентификатора, через **расстояние XOR**: выровняйте два идентификатора бит к
биту и сосчитайте, где они различаются. К географии это не имеет отношения — это
просто число, и оно даёт каждому узлу одинаковое представление о том, кто близок
к данному адресу.

**У каждого узла — упорядоченный список контактов.** Этот список разбит на 256
*k-бакетов* (k-bucket), в каждом до `K` контактов (`K = 20`, значение из
исходной статьи о Kademlia). Бакеты упорядочены по расстоянию, так что узел знает
много соседей поблизости и немного — далёких; именно это нужно, чтобы шаг за
шагом эффективно дойти до любого адреса.

**Поиск идёт итеративно.** Вместо того чтобы спросить один узел и ждать, узел
начинает с `K` ближайших известных ему контактов, опрашивает по `α = 3` за раунд,
добавляет любые более близкие контакты из ответов и повторяет — каждый раунд
оказываясь ближе к цели, пока не сойдётся, максимум за `MAX_ROUNDS = 20` раундов.

**Хранилище разбито на шарды.** Адресное пространство поделено на 256 *шардов*
(шард — это просто первый байт ключа, `shard_id = key[0]`), и каждый узел берёт
на себя 16 ближайших к нему шардов. Фильтрация `STORE` по шардам — по желанию
(opt-in).

**У хранилища два уровня.** Свежие, часто используемые записи лежат в быстром
*горячем* (hot) уровне в памяти; остальные — в *холодном* (cold) уровне. Записи
повышаются в горячий уровень при обращении (в порядке «дольше всех не
использованные» вытесняются первыми), опускаются обратно в холодный, когда горячий
переполняется, и удаляются совсем только при переполнении холодного. По умолчанию
холодный уровень тоже в памяти, но его можно положить на диск в хранилище RocksDB
через `[dht] cold_store_path` (cargo-feature `rocksdb-cold`, включён по умолчанию
для `veil-cli`). Это меняет потолок из оперативной памяти на потолок по диску —
спокойно за миллион записей — и сохраняет данные между перезапусками. Если feature
не собрана или RocksDB не удалось открыть, происходит тихий откат на холодный
уровень в памяти — с записью в лог при старте.

**Защита от окружения.** *Атака затмения* (eclipse) пытается забить ваш список
контактов узлами под управлением злоумышленника, отрезав вас от честной сети.
Чтобы это было трудно, один бакет принимает не более `K/4 = 5` контактов из
любой одной подсети `/24` IPv4 (или `/48` IPv6) — так что ни один маленький
уголок интернета не может захватить вашу картину сети.

**Записи аутентифицируются**, чтобы никто не мог сохранить или удалить что-то от
вашего имени:
- `StorePayload` может нести Ed25519-подпись над `key || value`.
- `DeletePayload` обязан нести `algo + pubkey + signature` (любой алгоритм подписи
  личности — Ed25519, Falcon-512 или гибрид Ed25519+Falcon) и принимается только
  при `BLAKE3(pubkey) == key` — то есть удалить ключ может лишь его настоящий
  владелец.

## Как лист становится доступным (привязка)

Лист сидит за домашним роутером, поэтому подключиться к нему напрямую другие не
могут. Чтобы всё равно оставаться доступным, лист **привязывается** к ядру и
просит его быть своей публичной точкой контакта. Ядро публикует в DHT маленькую
подписанную *запись о привязке*, которая гласит: «чтобы достучаться до этого
листа, идите через меня». Тот, кому нужен лист, находит эту запись и узнаёт, куда
прокладывать маршрут:

```
Leaf стартует → attach к Core → AnnounceAttachment(node_id, role, gateways, mailboxes, expires_at)
  → signed → сохранено в DHT по attachment_key(node_id)
Peer хочет достучаться до Leaf → GetAttachment(node_id) → узнаёт Core gateways/mailboxes → route
```

## Сквозное шифрование

Именно оно держит ваши сообщения в тайне от всех, кроме того, с кем вы
разговариваете. Отправитель запечатывает сообщение так, что вскрыть его может
только конечный получатель; каждый узел посередине лишь несёт закрытую коробку.
Veil использует ML-KEM-768 (постквантовую схему), чтобы договориться об
одноразовом общем секрете, а затем ChaCha20-Poly1305, чтобы зашифровать им
содержимое:

```
sender: (ct, ss) = ML-KEM-768.Encaps(recipient_ek)
        plaintext_envelope → ChaCha20-Poly1305(ss, nonce) → ciphertext
        send(E2E_MARKER || ct || ciphertext)

recipient: ss = ML-KEM-768.Decaps(dk, ct)
           plaintext = ChaCha20-Poly1305.open(ss, nonce, ciphertext)
```

Поскольку секрет устанавливается напрямую между двумя конечными точками,
ретрансляторы посередине имеют дело только с шифротекстом — зашифрованным видом.
Открытый текст они не видят никогда.

## Как пройти сквозь домашние роутеры (обход NAT)

Большинство людей сидят за **NAT** — так называется преобразование адресов,
которое делает домашний роутер, позволяя множеству устройств делить один
публичный IP. Это удобно, но означает, что никто снаружи не может просто взять и
дозвониться до вас. Как же связаться двоим, которые *оба* за NAT? Им немного
помогает ретранслятор, с которым каждый из них и так может разговаривать.
Ретранслятор их знакомит — сообщает каждой стороне, где другая, по-видимому,
находится, — и они открывают прямой путь; а если не получилось, ретранслятор
проводит трафик через себя сам:

```
A за NAT'ом → подключается к Relay R
A хочет достучаться до B (тоже за NAT'ом):
  A → R: NatProbe(observed-адрес B)
  R → B: NatProbeRelay(observed-адрес A)
  B открывает порт для A → A соединяется напрямую
  Fallback: relay-tunnel через R
```

## Как найти соседей поблизости (mesh)

Veil умеет соединять устройства в одной локальной сети напрямую — удобно для
комнаты, полной IoT-устройств, или там, где большого интернета нет вовсе. Каждое
устройство выкрикивает небольшой *маячок* (beacon) — приветствие, разосланное
всем в локальном сегменте, — каждые 30 секунд. Ближайший шлюз его слышит, узнаёт
собрата по Veil и устанавливает сессию. Этот шлюз затем работает мостом между
маленькой локальной mesh-сетью и большой сетью Veil:

```
IoT-device ← UDP beacon (multicast/broadcast, интервал 30 с) → Gateway
  Gateway видит beacon → auto-discover → устанавливает veil-сессию
  Gateway мостит local mesh ↔ global veil
```

Каждый маячок несёт `node_id` отправителя, `realm_id` (UUID, называющий, к какой
логической сети он относится), его транспортные URI и подписанные алгоритм с
открытым ключом — чтобы слушающие могли ему доверять. `realm_id` позволяет разным
сетям Veil делить один физический провод, не смешиваясь: узел просто игнорирует
любой маячок, чей `realm_id` не совпадает с его собственным.

## Как узнать о новых соседях (PEX)

Узлу всегда хочется знать ещё несколько соседей — и чтобы оставаться хорошо
связанным, и чтобы находить новые способы до них дотянуться. **Обмен соседями**
(peer exchange, PEX, семейство кадров 11) делает это случайным блужданием: запрос
бродит от узла к узлу, и тот узел, на который он попал, отвечает свежей порцией
соседей и их транспортных адресов. Чтобы это нельзя было использовать для спама,
блуждание по пути включает небольшую задачку-доказательство работы:

```
Originator → seed:        PexWalk (walk_id, pubkey, nonce, signature, TTL)
Terminator → originator:  PexChallenge (PoW challenge)
Originator → terminator:  PexResponse (solution, origin_sig)
Terminator → originator:  PexResult (peer list with transport URIs)
```

Блуждание работает с любым алгоритмом подписи: `crypto::verify_message` проверяет
`origin_sig` как Ed25519, когда открытый ключ длиной 32 байта, или как Falcon-512,
когда он длиннее.

## Как не пускать злоупотребления

Открытой сети нужны средства защиты от наводнений, спама и любителей ездить за
чужой счёт. Veil выстраивает несколько слоёв, и каждое новое входящее соединение
проходит через них по порядку — каждый слой дёшево проверить и можно быстро
отклонить нарушителя, пока он не успел дорого обойтись:

```
Inbound-подключение:
  1. Per-IP лимит сессий (max 32)
  2. PoW challenge (если сконфигурирован)
  3. Handshake timeout (10 с — `HANDSHAKE_TIMEOUT_SECS`)
  4. Per-peer rate limiter (token bucket)
  5. Violation tracker (5 violations → ban)
  6. Ban list (авто-expire после TTL)
  7. Congestion backpressure (>78% → drop transit)
  8. Reputation gate (200 баллов для transit'а)
```

## Как видеть, что происходит (наблюдаемость)

Когда хочется понять, как себя чувствует ваш узел, Veil даёт несколько окошек
внутрь:

- **Prometheus-метрики** — запросите `GET /metrics` и получите счётчики и
  датчики по всем подсистемам, готовые для графиков и оповещений.
- **Структурированное логирование** — каждая строка имеет вид
  `[timestamp] LEVEL event message`, а при желании читать машинами доступен
  формат JSON-L.
- **Захват для отладки** — команда CLI `debug capture` записывает кадры в файл
  прямо на лету, чтобы рассмотреть их поближе.
- **DiagPing** — измеряет задержку туда-обратно через всю сеть Veil, от конца до
  конца.
- **Буфер трассировки** — держит в памяти последние N событий обработки кадров,
  короткий «бортовой самописец» для отладки.
