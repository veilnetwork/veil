# Модель безопасности

Эта страница — модель безопасности и модель угроз узла Veil. Она написана так,
чтобы аудитор мог прочитать её сверху вниз: каждая строка называет конкретную
атаку, защиту, которая работает уже сегодня, и место в коде, где эта защита
живёт. Узкие термины разбираются при первом появлении.

Про охват. «Реализовано» значит, что защита есть в рабочем коде, а не что она
доказанно оптимальна. Раздел [Открытые риски](#открытые-риски) в самом низу
намеренно честен в том, что всё ещё слабо. Прочитайте его.

## Модель угроз

Каждая строка ниже соединяет одну атаку с мерой защиты, которая её блокирует (или
притупляет). Несколько терминов из таблицы:

- **Атака Sybil** — один оператор плодит уйму поддельных личностей, чтобы получить
  непомерное влияние. Veil заставляет каждую личность стоить процессорного времени
  (доказательство работы, PoW), и наводнение становится дорогим.
- **Атака «затмение» (eclipse)** — противник окружает ваш узел узлами, которыми
  управляет сам, отрезая вас от честной сети. Правила разнообразия в таблице
  маршрутизации не дают одному сетевому соседству занять все ваши слоты.
- **Повтор (replay)** — противник заново отправляет перехваченное ранее сообщение
  в надежде, что на него отреагируют дважды. Защита от дублей (dedup) помнит уже
  виденное и отбрасывает повтор.
- **DHT** — распределённая хеш-таблица: общий справочник адресов, который все узлы
  ведут вместе. Отравление (poisoning), злоупотребление удалением и перечисление
  (enumeration) — это способы, которыми противник пытается испортить её или
  выгрести из неё данные.

| Атака | Меры защиты | Статус |
|--------|-----------|--------|
| **Sybil** | PoW-сложность (по умолчанию 16, adaptive/epoch-based; `MAX_POW_DIFFICULTY=24` — жёсткий потолок) | Реализовано |
| **Eclipse (DHT)** | /24 subnet diversity в k-bucket'ах (K/4=5 max на subnet) | Реализовано |
| **Mailbox flood** | Reject при заполнении (без eviction); per-sender квота; global 100K cap | Реализовано |
| **Replay (routing)** | Двухуровневый dedup: per-(origin,via,seq) + per-(origin,seq); `MAX_ROUTE_ANNOUNCE_AGE_SECS=300` | Реализовано |
| **DHT poisoning** | Валидация `expires_at`; подписанные STORE-анонсы | Реализовано |
| **DHT delete abuse** | `DeletePayload` требует `(algo, pubkey, signature)`; `BLAKE3(pubkey)==key` (только self-owned) | Реализовано |
| **DHT seed exhaustion** | HashSet-based O(1) dedup в iterative lookups | Реализовано |
| **DHT enumeration** | FIND_NODE V2 + FIND_VALUE отдают только node_id (транспорты до-разрешаются per-hop через `ResolveTransport`); Public-only + half-cap фильтр на closest-node ответах | Реализовано (C-06) |
| **Gateway spoofing (session)** | `peer_roles` кэш сверяется с handshake capabilities | Реализовано |
| **Mesh-beacon spoofing (on-link)** | Неподписанные beacon'ы отбрасываются по умолчанию (`require_signed_beacons=true`); role-флаги не анонсируются, пока не задан `advertise_role_in_beacon` | Реализовано (C-03) |
| **Rate flood** | Per-peer token bucket → violation tracker (5 strikes / 5 мин) → ban list | Реализовано |
| **Connection flood** | `MAX_SESSIONS_PER_IP=32`; опциональный PoW challenge на handshake | Реализовано |
| **Congestion** | Backpressure при >78% load; adaptive fan-out режется вдвое при >50% | Реализовано |
| **Transit abuse** | Reputation gate: `MIN_REPUTATION_FOR_TRANSIT=200` | Реализовано |
| **Reputation inflation (поддельный delivery ACK)** | DELIVERED ACK несёт BLAKE3-MAC от `content_id` под per-message E2E-ключом; репутация начисляется только при валидном MAC | Реализовано (C-09) |
| **Cross-algo substitution** | Все подписи верифицируются через `crypto::verify_message(algo, ...)`; algo-байт идёт на проводе | Реализовано |
| **Traffic analysis** | Опциональные `SessionMsg::Padding` кадры выровнены по MTU | Реализовано |

## Криптографические примитивы

Это кирпичики, на которых стоит всё остальное. Несколько терминов. **AEAD**
(шифрование с аутентификацией и привязанными данными) одновременно прячет
сообщение и ловит любую его подмену. **HKDF** — функция вывода ключей: из одного
общего секрета она делает несколько ключей под конкретные задачи. **Nonce** —
число, используемое ровно один раз на шифрование, чтобы один и тот же открытый
текст никогда не давал одни и те же байты дважды; повторное использование ломает
шифр, поэтому мы делаем rekey (переходим на свежие ключи) до того, как счётчик
успеет переполниться. **PoW** — та самая задачка доказательства работы из
таблицы выше. Пометка «PQ» отмечает постквантовые варианты, выбранные так, чтобы
устоять даже против будущего квантового компьютера.

| Назначение | Алгоритм | Замечания |
|---------|-----------|-------|
| Identity | Ed25519, Falcon-512 или гибриды Ed25519+Falcon-512/1024 (PQ) | Конфигурируется per-node; `node_id = BLAKE3(pubkey)` |
| Session key exchange | X25519 ephemeral DH | HKDF-SHA256 (salt = `local_id XOR remote_id`, info = `"ovl1-session-v1"`) даёт `tx_key`/`rx_key`/`session_id`; lex-order swap tx/rx ключей даёт обеим сторонам зеркальные назначения |
| Session encryption | ChaCha20-Poly1305 | Per-frame AEAD; 12-байтный counter nonce; rekey при 128 GiB / 32 днях / counter wrap (конфигурируется через `[session] rekey_bytes_threshold` + `rekey_time_threshold_secs`) |
| E2E encryption | ML-KEM-768 encapsulation + ChaCha20-Poly1305 | Маркеры `0xE2` (E2E) / `0xE3` (meta-E2E, скрывает отправителя) |
| Хеширование | BLAKE3 | Node ID, DHT-ключи, PoW, content hashing, HMAC (`keyed`) |
| PoW | `BLAKE3(pubkey ‖ nonce ‖ sign(pubkey, nonce))` с заданным числом ведущих нулевых бит (по умолчанию 16) | Последовательный; адаптивно растёт к потолку `MAX_POW_DIFFICULTY=24` по мере роста сети |
| Mailbox replica encryption | HKDF(primary_mlkem_dk) + ChaCha20-Poly1305 | Реплики хранят непрозрачные blob'ы |

## Защита ключевого материала

Секретные ключи — это главная ценность, поэтому с ними обращаются особо. Сквозь
весь код проходят два правила: ключи никогда не утекают в отладочные логи, а
долгоживущие секреты закреплены в памяти, чтобы их не сбросило на диск и не
выхватило из аварийного дампа.

- `PowParams`, `Base64PrivateKey` и `Base64PublicKey` вычищают свои секреты из
  отладочного вывода.
- У `SessionKeys` для этого есть собственная реализация `Debug`.
- `IdentityConfig` и `MetricsConfig` тоже вычищают свой отладочный вывод (C-12).
- Сессионные ключи выводятся через HKDF-SHA256. Какой ключ для отправки, а какой
  для приёма, две стороны согласуют, упорядочив оба своих `node_id` по алфавиту,
  поэтому назначение выходит зеркальным с каждой стороны.
- Если счётчик nonce когда-либо переполняется, это обнаруживается, и сессия делает
  rekey прежде, чем какой-либо nonce успеет повториться.

## Открытые риски

Честные пробелы. Каждый — известная слабость с планом, а не неожиданность.

| Риск | Описание | План защиты |
|------|-------------|-----------------|
| Shard filtering bypass | `shard_filtering` — opt-in (default false) | Включить по умолчанию когда сеть > 1M узлов |
| Reputation cold start | Новые узлы стартуют со score 0 → не могут transit'ить сразу | Меры защиты TBD (peer vouches через `ReputationAttestation` дают некоторое ускорение) |
| Ключевой материал в памяти | Master- и identity-seed'ы mlock'нуты (`SensitiveBytesN`) + `madvise(MADV_DONTDUMP)`; часть session-AEAD-ключей всё ещё в heap | Реализовано (seed'ы); session-ключи в работе |
| Разрыв в версии протокола | `OVL1_MINOR_VERSION = 1`, но фичи gate'ятся при >=5 | Поднять версию с полным test coverage |
