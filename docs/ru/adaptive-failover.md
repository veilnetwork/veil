# Адаптивный переход на запасной путь

Путь к соседу (пиру) или получателю может отказать в любой момент. Физический
канал обрывается. Промежуточный ретранслятор падает. Кто-то фильтрует трафик
на линии. В такой ситуации Veil должен перестроить маршрут сам — этот
автоматический переход на запасной путь и называют failover. Вместе его дают
три механизма.

1. **Кэш многозвенных маршрутов** (всегда включён). `RouteCache` держит до
   `MAX_ROUTES_PER_DST = 4` кандидатов `next_hop` на каждого получателя.
   `next_hop` — это ближайший сосед, которому кадр передают по дороге к цели.
   Кэш ранжирует кандидатов по составному рейтингу (score), который сочетает
   задержку туда-обратно (RTT), число звеньев и репутацию соседа. `lookup_all`
   возвращает отсортированный список; `dispatch_delivery` выбирает лучший.

2. **Быстрое понижение пути при закрытии сессии** (всегда включён).
   Когда сессия с соседом закрывается — штатно или аварийно — диспетчер сразу
   вызывает `RouteCache::demote_via(closed_peer, factor=4.0)`. Этот вызов
   домножает рейтинг каждого кэшированного маршрута через этого соседа на 4×.
   Штраф выталкивает такие маршруты из полосы ECMP / multi-path, и на ближайшем
   же вызове `lookup_all` выигрывают альтернативные `next_hop`. (ECMP, или
   равностоимостная многопутёвка, — это набор маршрутов, чьи рейтинги настолько
   близки, что их считают равноценными.) Маршруты НЕ удаляются — сосед может
   вернуться. Они остаются запасным путём на крайний случай, когда всё
   остальное хуже.

   Без этого хука рейтинги обновлял бы только следующий цикл ROUTE_PROBE. Проба
   — это периодический измерительный пакет, а цикл идёт по адаптивному интервалу
   5–120 с, поэтому переход на запасной путь занимал десятки секунд. С хуком —
   меньше одного RTT.

3. **Доставка по нескольким путям** (по желанию, по умолчанию выключена). Ею
   управляют две настройки в `[routing]`:

       multi_path_enabled = true
       redundant_send = true

   Когда обе включены, чувствительные к задержке кадры (те, у которых
   `prio ≤ multi_path_min_priority`) отправляются дважды — по каждому из двух
   лучших путей. Получатель отбрасывает дубликат, сверяя `content_id`. Это
   стоит вдвое больше полосы на затронутых приоритетах. Взамен вы получаете
   устойчивость на уровне p99 к отказу одиночного пути: когда один путь
   падает, второй уже доставил кадр, поэтому заметного сбоя нет.

## Действия оператора

Посмотреть текущее состояние маршрутизации для получателя:

```sh
veil-cli node routes <dst_node_id>
```

Заголовок показывает активные настройки multi-path. Тело — основной `next_hop`
плюс альтернативы, каждая помечена `(alt)`. После закрытия сессии понижённый
основной маршрут несёт свой штраф 4×, поэтому за миллисекунды альтернативы
получают рейтинг ниже него — и становятся тем путём, который Veil использует
на деле.

## Когда включать multi-path

По умолчанию режим выключен, потому что удваивает полосу на затронутых
приоритетах. Включайте его, когда выполнено всё перечисленное:

- Развёртывание с запасом по полосе — локальная сеть или дата-центр.
- Задержка на уровне p99 важнее общей стоимости. Это про интерактивные
  приложения и приложения реального времени.
- В сети соседей есть естественная избыточность: не меньше 3 хорошо связанных
  опорных соседей на регион, чтобы два лучших пути были по-настоящему
  независимы.

На тестовой сети из 3 узлов смысла нет. Там оба «пути» — это один и тот же
сосед дважды, и выигрыша никакого.

## Проверка

```sh
# Активный конфиг виден в header'е routes:
veil-cli node routes
# → [routing config] multi-path: ON (paths=2, prio≤1)  |  redundant-send: ON  |  ecmp_band=0.20

# Углубиться в конкретный destination после разрыва его session'а:
veil-cli node routes 4f2a... | head -10
# Смотри: primary score должен скакнуть в 4× в момент закрытия session'а,
# (alt)-записи становятся эффективным best path'ом.
```
