# Справочник конфигурации OVL1

Все поля файла `config.toml`, собранные в одном месте.

Конфиг читается при запуске `veil-cli node run`. Где он лежит, зависит от ОС (XDG на Linux, `AppData` на Windows); чтобы узнать точный путь, выполните `veil-cli config locate`.

---

## Формат файла

Файл в формате **TOML**. Большинство секций необязательны — пропустите секцию, и применятся её значения по умолчанию. Две вещи обязательны: секция `[Identity]` и хотя бы один элемент `[[peers]]` или `[[listen]]`. Без них у узла нет ни личности, ни куда подключаться.

```toml
# Пример минимального конфига (leaf-узел)
persist_enabled = true

[Identity]
algo       = "ed25519"
role       = "leaf"
public_key = "BASE64..."
private_key = "BASE64..."
nonce      = "AAAAAA=="

[[peers]]
peer_id    = "0x00000001"
algo       = "ed25519"
public_key = "BASE64..."
nonce      = "AAAAAA=="
transport  = "tls://gateway.example.com:9443"
```

---

## Верхний уровень

### `persist_enabled`

| Тип | По умолчанию |
|-----|-------------|
| `bool` | `true` |

Главный выключатель всего, что пишется на диск. Когда `false`, ни один `*_persist_path` не читается при старте и не пишется во время работы — удобно для одноразовых узлов, CI и отладки.

## `[global]`

Настройки среды выполнения tokio и логирования.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `runtime_flavor` | enum | `"multi_thread"` | Тип среды выполнения tokio. Значения: `"multi_thread"`, `"current_thread"` |
| `worker_threads` | `u16` или отсутствует | не задано | Число рабочих потоков. Если не задано, берётся `num_cpus`. Только для `multi_thread` |
| `max_blocking_threads` | `u16` или отсутствует | не задано | Размер пула блокирующих потоков (для `spawn_blocking`). Если не задано, действует значение tokio по умолчанию (512) |
| `thread_keep_alive_ms` | `u64` или отсутствует | не задано | Сколько живёт простаивающий блокирующий поток, в мс |
| `thread_name` | `string` или отсутствует | не задано | Префикс имени рабочих потоков, чтобы их было видно в `ps` и `top` |
| `thread_stack_size` | `usize` или отсутствует | не задано | Размер стека рабочих потоков в байтах |
| `admin_socket` | `string` или отсутствует | не задано | URI бэкенда администрирования: `"unix:///abs/path/to/admin.sock"` (Linux/macOS) или `"tcp://127.0.0.1:0?runtime_dir=/abs/path"` (Windows или когда доменные сокеты недоступны). По TCP `admin.port` и `admin.token` пишутся в `runtime_dir`, откуда их читают клиенты; валидатор не допускает адреса вне loopback (`::1` и `localhost` разрешены). |
| `logs` | enum | `"stderr"` | Куда писать логи. Значения: `"stderr"`, `"file"` |
| `log_file` | `string` или отсутствует | не задано | Путь к файлу логов. Учитывается только когда `logs = "file"` |
| `log_level` | enum | `"info"` | Минимальный уровень логов. Значения: `"debug"`, `"info"`, `"warn"`, `"error"` |
| `log_format` | enum | `"text"` | Формат строк лога. Значения: `"text"` (для чтения человеком), `"json"` (NDJSON) |
| `admin_max_connections` | `usize` | `32` | Сколько подключений к сокету администрирования допускается одновременно |
| `require_signed_config` | `bool` | `false` | При `true` узел отказывается загружать неподписанный конфиг (Этап 11d) — см. подпись конфига в [OPERATIONS](OPERATIONS.md) |
| `tls_ech_grease` | `bool` | `true` | Слать TLS **ECH GREASE**, чтобы middlebox не отличал соединения с поддержкой ECH от остальных. `false` только для CDN, застрявших на TLS 1.2 |
| `bootstrap_dns_domain` | `string` или отсутствует | не задано | DNS-домен первичного подключения — сиды приходят в TXT-записях, ещё один запасной слой для входа в сеть |
| `bootstrap_https_urls` | `[string]` | `[]` | HTTPS- (и `.onion`-) URL с **подписанным** seed-бандлом — последний резерв, когда clearnet-сиды заблокированы |
| `bootstrap_tor_socks_proxy` | `string` или отсутствует | не задано | SOCKS5-прокси (например `"socks5://127.0.0.1:9050"`) для загрузки `.onion`-`bootstrap_https_urls` через Tor |
| `trusted_bundle_issuer_pubkey` | `string` или отсутствует | не задано | Запиненный открытый ключ издателя, против которого должен проверяться каждый подписанный seed-бандл |
| `legacy_allow_unsigned_bootstrap` | `bool` | `false` | Принимать неподписанные bootstrap-бандлы (устаревшее). По умолчанию `false`; `.onion`-источники всегда требуют подписи, независимо от флага |
| `discovered_peers_cache_path` | `string` или отсутствует | не задано | Кэш пиров из прошлых запусков — запасной путь для входа в сеть, если известные seed-IP лягут |

**Пример:**

```toml
[global]
runtime_flavor    = "multi_thread"
worker_threads    = 4
admin_socket      = "/var/run/veil/admin.sock"
logs              = "file"
log_file          = "/var/log/veil/node.log"
log_level         = "warn"
log_format        = "json"
```

## `[Identity]`

> Допускается также написание `[identity]` (строчными). Оба варианта эквивалентны.

Криптографическая личность узла — единственная секция, без которой узел не запустится.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `algo` | enum | `"ed25519"` | Алгоритм подписи. Значения: `"ed25519"`, `"falcon512"`, `"ed25519+falcon512"`, `"ed25519+falcon1024"` (последние два — постквантовые гибриды) |
| `role` | enum | `"leaf"` | Роль узла в сети. Значения: `"leaf"`, `"core"` |
| `public_key` | `string` | — | Открытый ключ, закодированный в base64. **Обязательно** |
| `private_key` | `string` | — | Закрытый ключ, закодированный в base64. **Обязательно** |
| `nonce` | `string` | `"AAAAAA=="` (4 нулевых байта) | PoW-нонс для `node_id = BLAKE3(pubkey \|\| nonce)`. Генерируется при `config init` |
| `node_id` | `string` или отсутствует | вычисляется | Явный hex node_id (64 символа). Если не задан, вычисляется из `public_key` + `nonce` |
| `key_passphrase` | `string` или отсутствует | не задано | Пароль для расшифровки зашифрованного закрытого ключа, прописанный прямо в конфиге. Не рекомендуется — лучше варианты с файлом или запросом ниже |
| `key_passphrase_file` | `string` или отсутствует | не задано | Путь к файлу с паролем ключа (держите mode `0600`) |
| `key_passphrase_prompt` | `bool` | `false` | При `true` спрашивать пароль ключа в диалоге при старте |
| `lazy_mining` | `bool` | `false` | Вычислять PoW-нонс в фоне, не заставляя `config init` ждать |
| `max_lazy_difficulty` | `u8` | `64` | Наибольшая сложность, которую попробует фоновый майнер |

> Имена — не ключи конфига. Заявите имя командой `veil-cli identity claim-name <name>`, и запущенный узел переопубликует его в DHT.

**Роли узла:**

| Роль | Описание |
|------|----------|
| `leaf` | Мобильный или лёгкий узел. В DHT не участвует, работает через core-узлы и mailbox |
| `core` | Полноправный участник сети: DHT (K=20), relay/forwarding, gateway (attachment records) и mailbox. Рекомендуемая сложность PoW ≥ 24 (значение `--difficulty` по умолчанию — `16`; `MAX_POW_DIFFICULTY = 24` — жёсткий потолок) |

Legacy-значения `"relay"`, `"gateway"`, `"core_router"` удалены — парсер теперь
принимает только `"leaf"` или `"core"`.

**Пример:**

```toml
[Identity]
algo        = "ed25519"
role        = "core"
public_key  = "MCowBQYDK2VwAyEA..."
private_key = "MC4CAQAwBQYDK2Vw..."
nonce       = "AAAAAA=="
```

## `[[peers]]`

Постоянные пиры, с которыми узел держит открытым исходящее соединение. Одна запись — одно соединение.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `peer_id` | `string` (hex u32) | — | Локальный идентификатор пира, например `"0x00000001"`. **Обязательно** |
| `public_key` | `string` | — | Открытый ключ пира (base64). **Обязательно** |
| `nonce` | `string` | — | PoW-нонс пира (base64). **Обязательно** |
| `transport` | `string` | — | URI транспорта, к которому подключаться (см. [Формат URI транспортов](#формат-uri-транспортов)). **Обязательно** |
| `algo` | enum | `"ed25519"` | Алгоритм подписи пира. Значения: `"ed25519"`, `"falcon512"` |
| `tls_cert` | `string` или отсутствует | не задано | PEM-сертификат для mTLS (на стороне клиента) |
| `tls_key` | `string` или отсутствует | не задано | Закрытый ключ для mTLS (на стороне клиента) |
| `tls_ca_cert` | `string` или отсутствует | не задано | CA-сертификат для проверки сертификата пира |

**Пример:**

```toml
[[peers]]
peer_id    = "0x00000001"
algo       = "ed25519"
public_key = "MCowBQYDK2VwAyEA..."
nonce      = "AAAAAA=="
transport  = "tls://gateway.example.com:9443"

[[peers]]
peer_id    = "0x00000002"
algo       = "ed25519"
public_key = "MCowBQYDK2VwAyEB..."
nonce      = "AAAAAA=="
transport  = "quic://core.example.com:9444"
```

## Формат URI транспортов

Поле `transport` использует единый формат URI во всех секциях, где оно есть (`[[peers]]`, `[[listen]]`, `[[bootstrap_peers]]`). Реализация: `crates/veil-transport/src/uri.rs`.

### Поддерживаемые схемы

| Схема | Направление | Описание |
|-------|-------------|----------|
| `tcp://HOST:PORT` | исходящее / входящее | Обычный TCP, без шифрования |
| `tls://HOST:PORT` | исходящее / входящее | TCP + TLS 1.3 |
| `quic://HOST:PORT` | исходящее / входящее | UDP + QUIC (TLS встроен); поддерживает двунаправленные потоки, подпотоки и датаграммы |
| `ws://HOST:PORT/PATH` | исходящее / входящее | WebSocket через TCP; доступен как байтовый поток или поток сообщений |
| `wss://HOST:PORT/PATH` | исходящее / входящее | WebSocket через TLS; доступен как байтовый поток или поток сообщений |
| `socks://PROXY:PORT/TARGET:PORT` | только исходящее | TCP через SOCKS5-прокси |
| `sockstls://PROXY:PORT/TARGET:PORT` | только исходящее | TLS через SOCKS5-прокси |
| `unix:///path/to/socket` | только входящее | Unix-сокет (только IPC) |

**Адреса привязки для `[[listen]]`:** используйте `0.0.0.0` или `[::]`, чтобы принимать соединения на всех интерфейсах; `127.0.0.1` или `unix://` — чтобы слушатель оставался локальным.  
**Для `[[peers]]` / `[[bootstrap_peers]]`:** DNS-имя или IP удалённого узла.  
**IPv6:** оборачивайте адрес в квадратные скобки — `tcp://[::1]:9000`, `tls://[2001:db8::1]:443`.

### Query-параметры для TLS-схем

Схемы `tls://`, `quic://`, `wss://` и `sockstls://` принимают query-параметры:

| Параметр | Повтор | Описание |
|----------|--------|----------|
| `sni=NAME` | один | Переопределить SNI (Server Name Indication) для TLS-рукопожатия. По умолчанию совпадает с `host`; у `sockstls://` — с `target_host` |
| `alpn=PROTO` | много | Добавить ALPN-протокол. Можно указывать несколько раз |

**Примеры:**

```
# TLS с явным SNI (полезно при IP-подключении или reverse-proxy)
tls://10.0.0.1:9443?sni=node.example.com

# TLS с несколькими ALPN
tls://example.com:443?alpn=h2&alpn=http/1.1

# QUIC с ALPN
quic://example.com:9443?sni=example.com&alpn=h3

# WebSocket Secure с переопределённым SNI
wss://10.0.0.1:443/veil?sni=gateway.internal
```

### SOCKS5-прокси

Формат: схема `://proxy_host:proxy_port/target_host:target_port`

```
# TCP через SOCKS5-прокси
socks://127.0.0.1:1080/remote.example.com:9000

# TLS через SOCKS5-прокси (SNI автоматически = target_host)
sockstls://127.0.0.1:1080/remote.example.com:9443

# TLS через SOCKS5 с явным SNI и ALPN
sockstls://127.0.0.1:1080/10.0.0.5:9443?sni=remote.example.com&alpn=h2
```

### TLS-сертификаты

Параметры `tls_cert`, `tls_key` и `tls_ca_cert` в секциях `[[listen]]` / `[[peers]]` применяются только к схемам `tls://`, `quic://` и `wss://`. Для `tcp://`, `ws://` и `unix://` они игнорируются.

| Параметр | В `[[listen]]` | В `[[peers]]` |
|----------|---------------|--------------|
| `tls_cert` | Сертификат сервера (PEM, leaf или fullchain) | Клиентский сертификат (mTLS) |
| `tls_key` | Закрытый ключ сервера | Закрытый ключ клиента |
| `tls_ca_cert` | CA для проверки клиентов (mTLS) | CA для проверки сертификата сервера |

> Не кладите CA-сертификат в поле `tls_cert` слушателя: rustls отклоняет CA-сертификат, использованный как end-entity-сертификат сервера.

### Отладка транспортов

Подкоманда `veil-cli debug transport` даёт проверить соединение вручную, не поднимая полноценный узел.

**Примеры подключения (клиент):**

```bash
veil-cli debug transport connect tcp://127.0.0.1:9001
veil-cli debug transport connect tls://example.com:443?sni=example.com&alpn=h2
veil-cli debug transport connect quic://example.com:443?alpn=h3
veil-cli debug transport connect unix:///tmp/veil.sock
veil-cli debug transport connect socks://127.0.0.1:1080/1.1.1.1:9001
veil-cli debug transport connect sockstls://127.0.0.1:1080/example.com:443?sni=example.com
veil-cli debug transport connect ws://127.0.0.1:8080/veil
veil-cli debug transport connect wss://example.com:443/veil?alpn=http/1.1
```

**Примеры слушателя (сервер):**

```bash
veil-cli debug transport listen tcp://0.0.0.0:9001
veil-cli debug transport listen unix:///tmp/veil.sock
veil-cli debug transport listen tls://0.0.0.0:9443?sni=localhost&alpn=h2
veil-cli debug transport listen quic://0.0.0.0:9444?alpn=h3
veil-cli debug transport listen ws://0.0.0.0:8080/veil
veil-cli debug transport listen wss://0.0.0.0:8443/veil?alpn=http/1.1
```

Для `listen` с TLS-схемой (`tls://`, `wss://`, `quic://`) временный самоподписанный сертификат создаётся за вас. Чтобы проверить ближе к продакшену, передайте настоящие сертификаты:

```bash
# Слушатель с реальным сертификатом
veil-cli debug transport listen tls://0.0.0.0:9443 \
  --tls-cert ssl/server-fullchain.pem \
  --tls-key ssl/server.key

# Клиент с кастомным CA
veil-cli debug transport connect tls://127.0.0.1:9443 \
  --tls-ca-cert ssl/ca.pem

# Аналогично для WSS и QUIC
veil-cli debug transport listen wss://0.0.0.0:8443/veil \
  --tls-cert ssl/server-fullchain.pem --tls-key ssl/server.key
veil-cli debug transport connect wss://127.0.0.1:8443/veil \
  --tls-ca-cert ssl/ca.pem
```

Флаги `--tls-cert`, `--tls-key` и `--tls-ca-cert` работают одинаково для `tls://`, `wss://` и `quic://`.

## `[[listen]]`

Входящие слушатели. Каждый — это один порт, на котором узел принимает соединения.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `id` | `string` (hex u32) | — | Локальный идентификатор слушателя, например `"0x00000001"`. **Обязательно** |
| `transport` | `string` | — | URI транспорта, к которому привязаться (см. [Формат URI транспортов](#формат-uri-транспортов)). **Обязательно** |
| `advertise` | `string` или отсутствует | не задано | Адрес, анонсируемый пирам вместо `transport`. Нужен за обратным прокси: привязка к `localhost:9443`, анонс `wss://nginx.example.com:443/veil` |
| `relay` | `string` или отсутствует | не задано | Hex node_id relay-узла, через который можно добраться до этого слушателя (для NAT). Включается в `RouteResponsePayload.relay_ids` |
| `tls_cert` | `string` или отсутствует | не задано | PEM-сертификат сервера (TLS/WSS) |
| `tls_key` | `string` или отсутствует | не задано | Закрытый ключ сервера |
| `tls_ca_cert` | `string` или отсутствует | не задано | CA-сертификат для проверки клиентов (mTLS) |
| `visibility` | enum | `"public"` | Насколько слушатель виден. `"public"` — анонсируется в PEX + DHT; `"trusted"` — не анонсируется, доступен по приглашению вне сети; `"hidden"` — как trusted, но ещё проверяет `allowlist_node_ids` при рукопожатии |
| `psk_file` | `string` (путь) или отсутствует | не задано | Путь к файлу с PSK (32 байта, base64) для слушателя `obfs4-tcp://`. Переопределяет глобальный `[transport].obfs4_psk_file`, так что PSK можно разделить по группам — публичный слушатель на общем для развёртывания PSK плюс семейный на отдельном |
| `allowlist_node_ids` | `[string]` | `[]` | Список 32-байтовых node_id в hex, которым разрешено проходить аутентификацию на этом слушателе. Обязателен для `visibility = "hidden"`; необязательное усиление для `"trusted"`. Пустой — без списка разрешённых |
| `group_label` | `string` или отсутствует | не задано | Понятная человеку метка группы (например `"family"`, `"snowflake"`). Логика демона её не использует — она лишь попадает в логи и метрики |
| `ephemeral` | таблица или отсутствует | не задано | Ротация случайного порта, чтобы порты не группировались. См. [`[listen.ephemeral]`](#listenephemeral) ниже |

**Пример:**

```toml
[[listen]]
id        = "0x00000001"
transport = "tls://0.0.0.0:9443"
tls_cert  = "/etc/veil/server.crt"
tls_key   = "/etc/veil/server.key"
advertise = "tls://node.example.com:9443"

[[listen]]
id        = "0x00000002"
transport = "ws://0.0.0.0:8080/veil"

# trusted-only obfs4 listener для семейного круга
[[listen]]
id          = "0x00000003"
transport   = "obfs4-tcp://0.0.0.0:5556"
visibility  = "trusted"
psk_file    = "/etc/veil/family.psk"
group_label = "family"
```

### `[listen.ephemeral]`

Меняет порт прослушивания случайным образом по расписанию, чтобы порты не группировались (в стиле snowflake).
Демон перепривязывается к свежему порту из `range` каждые `rotation`, а пиры узнают новый URI из подписанной рассылки `TransportMigrationNotify`.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `range` | `[u16, u16]` | — | Диапазон портов, оба конца включительно, например `[10000, 60000]`. Узкий диапазон помогает имитировать конкретный протокол (`[3306, 3306]` — MySQL-SSL). **Обязательно** |
| `rotation` | `string` (duration spec) | — | Как часто менять порт: `"30s"`, `"5m"`, `"3h"`, `"7d"`. **Обязательно** |
| `bind_retries` | `u32` | `64` | Сколько попыток привязки делать при коллизии (`EADDRINUSE`). 0 = одна попытка |
| `grace_period` | `string` (duration spec) | `"30m"` | Сколько старый слушатель живёт после ротации, доводя начатые рукопожатия, прежде чем его закроют |

**Ограничения:**

- Работает только для слушателей `obfs4-tcp://` (или другого транспорта, чей URI поддерживает `with_host_port`).
- Нужна личность Ed25519 — кадр `TransportMigrationNotify` подписывается через ed25519-dalek; гибрид Falcon-512 + Ed25519 пока не поддерживается.
- Если перепривязка к новому порту не удалась, пишется предупреждение, а старый слушатель продолжает работать. Пиры, чьи кэши уже указывают на новый URI, идут запасным путём через DHT.

**Пример:**

```toml
[[listen]]
id        = "0x00000004"
transport = "obfs4-tcp://0.0.0.0:5556"   # стартовый порт (после первой ротации игнорируется)
psk_file  = "/etc/veil/ephemeral.psk"

[listen.ephemeral]
range         = [50000, 60000]
rotation      = "1h"
grace_period  = "30m"
bind_retries  = 64
```

**Логи и метрики:** при каждой ротации демон пишет структурированные логи на уровне info:

- `listen.rotation.spawned` — при старте, подтверждает, что задача ротатора поднялась
- `session.migration.notify.applied` — на стороне пира, когда он получил и применил рассылку
- `listen.rotation.swap_sent` — на ротирующем узле, после успешной перепривязки
- `listen.swap` — цикл приёма переключился на новый слушатель
- `listen.rotation.rebind_failed` (warn) — новая привязка не удалась; старая продолжает работать
- `listen.rotation.bind_failed` (warn) — ротатор не смог выбрать порт из диапазона

### `[listen.on_demand]`

Слушатель по запросу привязывает порт **только когда его попросят**, а не на старте.
Обычно `ss -tlnp` не показывает ничего; порт открывается лишь после успешного
PoW-рукопожатия, обслуживает ограниченное число сессий (или до истечения TTL) и
закрывается сам.

**Требования:**

- `visibility = "stealth"` обязательно (иначе проверка конфига выдаст ошибку при старте)
- Личность узла Ed25519 (гибрид Falcon-512 на этом слое не поддерживается)
- **Один stealth-слушатель** на узел (несколько stealth = TODO в Slice 6+)

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `range` | `[u16, u16]` | — | Диапазон портов, оба конца включительно, для привязки по запросу. **Обязательно** |
| `pow_difficulty` | `u32` | — | Требуемая сложность PoW в ведущих нулевых битах BLAKE3. Production: 24 (~16M попыток ≈ 0.5 сек CPU). Минимум 8. **Обязательно** |
| `ttl` | `string` (duration) | — | TTL слота: `"5m"`, `"300s"`. По истечении задача приёма выходит и слушатель закрывается, даже если ни одной сессии не пришло. **Обязательно** |
| `max_concurrent` | `usize` | `16` | Максимум одновременных слотов по запросу. Не даёт оплаченному через PoW всплеску исчерпать таблицу дескрипторов |
| `rate_limit` | `string` (`"N/period"`) | `"3/h"` | Лимит на одного запрашивающего: `"3/h"` (3 выдачи в час на открытый ключ), `"1/m"`, `"10/30s"` |
| `max_accepts` | `usize` | `1` | Сколько сессий слот принимает до выхода из работы. 1 = одноразовая встреча |
| `bind_retries` | `u32` | `64` | Попыток привязки при EADDRINUSE |

**Пример:**

```toml
[[listen]]
id        = "0x00000005"
transport = "obfs4-tcp://example.com:0"   # port игнорируется для stealth
visibility = "stealth"
advertise = "obfs4-tcp://example.com"      # advertise_host для composing response URI

[listen.on_demand]
range          = [50000, 60000]
pow_difficulty = 24
ttl            = "5m"
max_concurrent = 16
rate_limit     = "3/h"
max_accepts    = 1
```

**Логи (info-уровень):**

- `rendezvous.controller.wired` — при старте, подтверждает, что контроллер привязан к диспетчеру
- `rendezvous.request.rejected reason=<кат>` — запрос отклонён (категории: decode, verify, not_our_target, rate_limited, concurrency_exhausted, bind_failed)
- `rendezvous.response.sent peer_id=<8 hex> new_port=<N>` — подписанный ответ отправлен инициатору
- `rendezvous.on_demand.listener.spawned listen_id=<id> local_addr=<addr> ttl_remaining=<sec> accepts_remaining=<N>` — слушатель по запросу поднят
- `rendezvous.on_demand.scanner_dropped` — соединение с забаненного IP отброшено до рукопожатия
- `rendezvous.on_demand.budget_exhausted` — все `max_accepts` слотов израсходованы, задача приёма выходит
- `rendezvous.on_demand.listener.ttl_or_shutdown` — истёк TTL или останавливается среда выполнения
- `rendezvous.on_demand.listener.exited` — последняя запись перед закрытием слушателя

**Prometheus metrics (Slice 7):**

- `veil_rendezvous_requests_received_total` (counter) — всего запросов получено
- `veil_rendezvous_requests_granted_total` (counter) — выдано подписанных ответов
- `veil_rendezvous_requests_rejected_decode_total` (counter)
- `veil_rendezvous_requests_rejected_verify_total` (counter)
- `veil_rendezvous_requests_rejected_not_our_target_total` (counter)
- `veil_rendezvous_requests_rejected_rate_limit_total` (counter)
- `veil_rendezvous_requests_rejected_concurrency_total` (counter)
- `veil_rendezvous_requests_rejected_bind_failed_total` (counter)
- `veil_rendezvous_slots_in_use` (gauge) — текущее число активных слушателей по запросу

Доля выдач — `granted / received`. Высокий `rejected_verify_total` при низком `granted` означает одно из двух: клиенты вычисляют слишком слабый PoW (поднимите `pow_difficulty`) либо кто-то подделывает запросы (и слои защиты от злоупотреблений делают своё дело). Высокий `rejected_concurrency_total` означает, что `max_concurrent` слишком тесный для нормальной нагрузки.

**Что пока не сделано (Slice 6+):**

- **Маршрутизация через посредника.** На этом слое обрабатывается только сторона цели для кадра запроса, а это предполагает, что у запрашивающего уже есть OVL1-сессия с целью — бессмыслица для stealth-слушателя без открытого порта. Полная интеграция через посредник-relay в PEX/DHT появится в Slice 6.
- **Сквозные интеграционные тесты.** Slice 8.

До Slice 6 stealth-слушатель работает в режиме «прицеплен к пути диспетчеризации, но недостижим через посредника» — это удобно для модульного тестирования контроллера и наблюдения через метрики, пока вы вручную подаёте кадры.

## `[[bootstrap_peers]]`

Bootstrap-пиры наполняют DHT-таблицу маршрутизации при первом старте узла. Используются только при старте: узел выполняет FIND_NODE(self), затем сессия закрывается — если только этот пир не указан ещё и в `[[peers]]`.

В отличие от `[[peers]]`, соединения с bootstrap-пирами не держатся открытыми.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `transport` | `string` | — | URI транспорта (см. [Формат URI транспортов](#формат-uri-транспортов)). **Обязательно** |
| `public_key` | `string` | — | Открытый ключ bootstrap-узла (base64). **Обязательно** |
| `nonce` | `string` | `"AAAAAA=="` | PoW-нонс bootstrap-узла |
| `algo` | enum | `"ed25519"` | Алгоритм подписи. Значения: `"ed25519"`, `"falcon512"`, `"ed25519+falcon512"`, `"ed25519+falcon1024"` (последние два — постквантовые гибриды) |
| `tls_cert` | `string` или отсутствует | не задано | PEM-сертификат (для TLS-транспорта) |
| `tls_ca_cert` | `string` или отсутствует | не задано | CA-сертификат для проверки |

**Пример:**

```toml
[[bootstrap_peers]]
transport  = "tcp://bootstrap1.example.com:9000"
public_key = "MCowBQYDK2VwAyEA..."
nonce      = "AAAAAA=="

[[bootstrap_peers]]
transport  = "tcp://bootstrap2.example.com:9000"
public_key = "MCowBQYDK2VwAyEB..."
```

## `[transport]`

Настройки транспортного уровня и средства обхода цензуры (уход от DPI). Все
ключи необязательны.

> **Бэкенд TLS.** Бинарник `veil-cli` по умолчанию использует бэкенд
> BoringSSL (cargo-feature `tls-boring`, в составе `default = ["rocksdb-cold",
> "tls-boring"]`). Он формирует Chrome-подобный отпечаток JA3/JA4 в TLS
> ClientHello и умеет его ротировать. Бэкенд `rustls` — запасной путь
> при сборке `--no-default-features`; он не умеет менять ClientHello, поэтому
> подсекция `[transport.tls_fingerprint]` на нём игнорируется.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `default_sni` | `string` или отсутствует | не задано | Хостнейм SNI по умолчанию в TLS ClientHello, когда исходящий URI не задаёт `?sni=...`, а цель не loopback. Задайте что-то вроде `"www.google.com"` — и DPI на пути увидит популярный домен вместо настоящего хостнейма узла. Если не задано, в качестве SNI используется хост цели |
| `obfs4_psk_file` | `string` (путь) или отсутствует | не задано | Путь к файлу с pre-shared key obfs4 (32 байта, base64 в одну строку). Если задан, включает транспорт `obfs4-tcp://`: сервер проверяет MAC на входящих рукопожатиях, клиент добавляет MAC в исходящие. Единый сетевой PSK. Если не задано, транспорт `obfs4-tcp` выключен |
| `webtunnel_secret_path` | `string` или отсутствует | не задано | Секретный путь webtunnel (например `/_t/random-32-chars`). Включает режим туннеля на серверной стороне транспорта `webtunnel-wss://` |
| `webtunnel_auth_token_file` | `string` (путь) или отсутствует | не задано | Файл с токеном авторизации webtunnel (32 случайных байта в base64). Передаётся в заголовке `X-Veil-Auth` рядом с секретным путём |
| `webtunnel_decoy_dir` | `string` (путь) или отсутствует | не задано | Каталог с приманкой webtunnel: статические файлы, которые отдаются пробам, не совпавшим с секретным путём или авторизацией. Хорошо подойдёт снимок какого-нибудь нейтрального сайта. Если не задано, отдаётся минимальная встроенная HTML-страница |
| `webtunnel_response_floor_ms` | `int` (мс) или отсутствует | не задано / `0` | Необязательный порог времени ответа против проб. Если задан, сервер придерживает любой ответ — и апгрейд туннеля `101`, и приманку — пока с момента чтения запроса не пройдёт хотя бы столько миллисекунд, чтобы проба не могла отличить реальную точку туннеля от обычной приманки по задержке. Добавляет до стольких же миллисекунд к задержке рукопожатия, поэтому включается по желанию. Ставьте выше худшего времени выборки приманки (для приманки-прокси — выше задержки бэкенда), иначе приманка всё равно может выйти за порог. `0`/не задано — ответ сразу по готовности |
| `outbound_socks_fallback_proxy` | `string` или отсутствует | не задано | URL SOCKS-прокси, на который можно опереться, когда прямой дозвон раз за разом не удаётся (блокировка на уровне AS, перехват маршрута провайдером). Формат `socks5://127.0.0.1:9050` (локальный Tor) или `socks5://proxy.example:1080`. Если не задано, пробуются только прямые соединения |
| `bandwidth_mimicry_enabled` | `bool` | `false` | Имитация профиля пропускной способности (P2 #7). Пока это **заготовка**: поле распознаётся, но слой формирования трафика ещё не подключён. Установка в `true` без `experimental_allow_noop_mimicry` — ошибка валидации (fail-closed) |
| `bandwidth_mimicry_profile` | `string` или отсутствует | не задано | Имя профиля для `bandwidth_mimicry_enabled`: `"chrome-browsing"`, `"cdn-download"`, `"interactive-chat"`. Пока тоже заготовка |
| `experimental_allow_noop_mimicry` | `bool` | `false` | Подтверждает, что `bandwidth_mimicry_enabled` сейчас пустая заготовка, и соглашается запустить демон без настоящей имитации. Обязателен вместе с `bandwidth_mimicry_enabled = true` |
| `obfs4_accept_variants` | `[string]` | `[]` | **Аварийный выключатель, серверная сторона**: какие варианты протокола obfs4 принимать, в порядке приоритета. Пусто (разрешается в `["v1"]`) сохраняет поведение до Phase 2. Значения: `"v1"`, `"v2"` |
| `obfs4_client_variant` | `string` или отсутствует | не задано | **Аварийный выключатель, клиентская сторона**: вариант протокола obfs4 для исходящих `obfs4-tcp://`. Если не задано, разрешается в `v1`. Значения: `"v1"`, `"v2"`. Переключайте на `"v2"` только после того, как у всех целевых серверов `obfs4_accept_variants` включает `v2` |

### `[transport.rotation]`

Ротирует транспортные соединения по расписанию. Периодически закрывает и заново
создаёт нижележащее TCP/TLS-соединение каждой сессии, чтобы DPI не мог
классифицировать поток по времени его жизни (эвристика «эта HTTPS-сессия живёт
6 часов — это VPN» теряет сигнал). При рукопожатии каждая сессия берёт случайное
время жизни из диапазона `[min_lifetime_secs, max_lifetime_secs]`. Секция
**всегда сериализуется** — как средство обхода цензуры, оператор должен видеть
её в своём конфиге.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `min_lifetime_secs` | `i64` | `1800` | Наименьшее время жизни сессии в секундах (30 мин). `-1` выключает весь механизм ротации. Положительные значения меньше 60 отвергаются валидацией |
| `max_lifetime_secs` | `i64` | `3600` | Наибольшее время жизни сессии в секундах (1 час). `-1` выключает ротацию. Должно быть `>= min_lifetime_secs`, когда оба положительны |

### `[transport.tls_fingerprint]`

Управляет отпечатком TLS ClientHello на исходящих `tls://` / `wss://`
соединениях. Действует **только на сборках с `tls-boring`** — бэкенд `rustls`
не умеет менять ClientHello и игнорирует эту секцию. Секция **всегда
сериализуется**; как и `[transport.rotation]`, это средство обхода цензуры,
поэтому остаётся видимым в конфиге.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `mode` | enum | `"rotate"` | Как выбирается отпечаток: `"pinned"` (всегда `profile`), `"rotate"` (перебирать профили из `rotation` на свежих соединениях, пока одно не завершит рукопожатие), `"random"` (свежий случайный ClientHello каждый раз). По умолчанию `"rotate"` — переживает блокировки: когда один JA3 заблокирован, узел переходит к следующему |
| `profile` | enum | `"chrome"` | Профиль для режима `"pinned"`. Токены профилей: `chrome`, `firefox`, `safari`, `ios`, `android`, `random` |
| `rotation` | `[string]` | `["chrome", "firefox", "safari"]` | Упорядоченный список профилей для перебора в режиме `"rotate"` |
| `sticky` | `bool` | `true` | В режиме `"rotate"` оставаться на последнем профиле, завершившем рукопожатие, вместо того чтобы начинать перебор заново |

### `[transport.tls_client]`

Хранилище доверия для **исходящего** TLS узла (HTTPS-бутстрап, webtunnel). Необязательно.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `connect_timeout_ms` | `u64` или отсутствует | не задано | Тайм-аут (время ожидания) подключения для исходящего TLS, мс |
| `use_system_roots` | `bool` | `false` | Добавить CA-бандл Mozilla webpki-roots в клиентское хранилище доверия. По умолчанию `false` — veil доверяет только запиненным оператором CA через `trusted_ca_file`. Ставьте `true` для mesh-узлов, ходящих к публично-сертифицированным хостам |
| `trusted_ca_file` | `string` или отсутствует | не задано | PEM-файл с запиненными оператором CA, которым доверять |

**Пример:**

```toml
[transport]
default_sni                    = "www.google.com"
obfs4_psk_file                 = "/etc/veil/obfs4.psk"
outbound_socks_fallback_proxy  = "socks5://127.0.0.1:9050"

[transport.rotation]
min_lifetime_secs = 1800
max_lifetime_secs = 3600

[transport.tls_fingerprint]
mode     = "rotate"
rotation = ["chrome", "firefox", "safari"]
sticky   = true
```

## `[mesh]`

Настройки локальной UDP-сети mesh — обнаружения соседей в пределах одного сетевого сегмента. Необязательная.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `bind_addr` | `string` | — | UDP-адрес для realm-слушателя, например `"0.0.0.0:9100"`. **Обязательно** при наличии секции |
| `realm_id` | `string` | — | 32 hex-символа (16 байт) — идентификатор realm. **Обязательно** |
| `beacon_addr` | `string` | `"255.255.255.255:9100"` | Broadcast/multicast-адрес для обнаружения по маякам. **Порт** — последний оставшийся сигнал формы трафика, который выдаёт маяк (размер и период уже скрыты, когда задан `realm_psk`, C-03) — на враждебном LAN выберите нестандартный порт (одинаковый у всех участников realm) |
| `autodiscover_gateway` | `bool` | `true` | Автоматически подключаться к gateway-узлам, найденным через mesh-маяки |
| `autodiscover_max_concurrent` | `usize` | `3` | Сколько исходящих сессий к автообнаруженным gateway держать одновременно |
| `beacon_dedup_window_secs` | `u64` | `3` | Сколько секунд игнорировать повторные маяки от одного источника. `0` выключает дедупликацию |
| `autodiscover_persist_path` | `string` или отсутствует | не задано | Куда сохранять таблицу `AutoDiscoveredPeers`. Восстанавливается при старте, чтобы ближайшие gateway были известны ещё до первого маяка |
| `require_signed_beacons` | `bool` | `true` | При `true` (по умолчанию, C-03) принимаются только криптографически подписанные mesh-маяки, а неподписанные отбрасываются — это закрывает дыру с инъекцией gateway и подменой соседних линков в локальной сети. Ставьте `false` только для совместимости со старыми развёртываниями, всё ещё рассылающими неподписанные маяки — включение «только подписанные» поверх живой неподписанной сети отрезает такие узлы, поэтому сперва раскатайте подписанные маяки по всему парку |
| `advertise_role_in_beacon` | `bool` | `false` | При `true` узел анонсирует свои role-флаги (`IS_GATEWAY` / `IS_RELAY` / `HAS_INTERNET`) в mesh-маяке — что нужно пирам с `autodiscover_gateway`, чтобы распознать его как gateway. По умолчанию `false` (C-03): маяк несёт `role_flags = 0`, поэтому пассивный наблюдатель в локальной сети не может выделить узел как gateway/relay (сигнал для нацеливания/цензуры). Стабильный `node_id` транслируется в любом случае |
| `realm_psk` | `string` или отсутствует | не задан | **Необязательная обфускация UDP.** Pre-shared key в base64 (≥ 16 байт после декодирования). Если задан, mesh-датаграммы типа **DATA** **и discovery-маяки** заворачиваются в AEAD (`veil-udp-obfs`: ChaCha20-Poly1305, свежий случайный nonce + случайный паддинг на каждую датаграмму) — пассивный наблюдатель DPI/LAN видит только меняющийся шифртекст; скрыты и mesh-фрейминг, **и стабильный `node_id` / role-флаги / адрес подключения в маяках** (закрывает C-03; обнаружение тогда требует PSK, как и ожидается для защищённого realm). Ключ общий для realm (выводится по HKDF из PSK и `realm_id`); **все участники realm должны иметь одинаковый PSK**, розданный вне сети. Если не задан (по умолчанию) → mesh и маяки открытым текстом, поведение байт-в-байт неизменно. Заданный, но некорректный или слишком короткий PSK **отключает mesh**, а не откатывается молча на открытый текст |

**Пример:**

```toml
[mesh]
bind_addr                  = "0.0.0.0:9100"
realm_id                   = "deadbeefcafebabedeadbeefcafebabe"
autodiscover_gateway       = true
autodiscover_max_concurrent = 5
autodiscover_persist_path  = "/var/lib/veil/autodiscover.bin"
# realm_psk                = "BASE64_PRESHARED_KEY"  # опционально: AEAD-обфускация DATA-датаграмм (≥16 байт, общий на realm)
```

## `[session]`

Настройки сессионного уровня: keepalive, тайм-аут простоя, очереди.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `keepalive_interval_secs` | `u64` | `30` | Как часто слать keepalive-фреймы, в секундах. `0` их выключает |
| `idle_timeout_secs` | `u64` | `90` | Закрыть сессию, если за это окно не пришло ни одного фрейма. Должен быть > `keepalive_interval_secs` |
| `max_concurrent` | `usize` | `512` | Сколько OVL1-сессий держать открытыми одновременно |
| `max_per_ip` | `usize` | `32` | Сколько входящих сессий допускать с одного IP-адреса |
| `max_pending_responses` | `usize` | `256` | Сколько ожидающих RPC-ответов держать на сессию. Всё сверх — отбрасывается |
| `pending_response_ttl_ms` | `u64` | `30000` | Сколько живёт слот ожидающего ответа, в мс. Устаревшие вытесняются |
| `tx_queue_depth` | `usize` | `4096` | Размер канала исходящих фреймов на сессию. При переполнении фреймы отбрасываются |
| `outbox_depth` | `usize` | `256` | Размер RPC outbox на сессию. При полном канале `send_request()` возвращает `None` |
| `max_frame_body_bytes` | `u32` | 1 МиБ | Наибольший допустимый размер тела фрейма; фреймы крупнее отвергаются. Жёсткий потолок: 16 МиБ |
| `qos_weights` | `[u8; 4]` | `[8, 4, 2, 1]` | WRR-веса для классов `[RealTime, Interactive, Bulk, Background]` внутри сессии |
| `rt_queue_len` | `usize` | `64` | Глубина очереди REALTIME на сессию. При переполнении фреймы отбрасываются |
| `bg_queue_len` | `usize` | `256` | Глубина очереди BACKGROUND на сессию. При переполнении фреймы отбрасываются |
| `rekey_bytes_threshold` | `u64` | 128 ГиБ (`137_438_953_472`) | Сменить ключи, когда через сессию прошло столько байт |
| `rekey_time_threshold_secs` | `u64` | 32 дня (`2_764_800`) | Сменить ключи, когда столько прошло с прошлой смены ключей или старта сессии |
| `max_per_subnet` | `usize` | `64` | Сколько входящих сессий допускать из одной подсети /24 (IPv4) или /48 (IPv6) |
| `battery_threshold_low` | `u8` | `20` | Процент батареи, на/ниже которого включается «низкое» масштабирование keepalive |
| `battery_threshold_medium` | `u8` | `50` | Процент батареи, на/ниже которого включается «среднее» масштабирование keepalive |
| `battery_keepalive_scale_low` | `f32` | `4.0` | Множитель keepalive-интервала при батарее ≤ `battery_threshold_low` |
| `battery_keepalive_scale_medium` | `f32` | `2.0` | Множитель keepalive-интервала при батарее ≤ `battery_threshold_medium` |
| `battery_sync_threshold` | `u8` | `15` | Процент батареи, ниже которого фоновая синхронизация откладывается |
| `allowed_peer_algos` | `[enum]` | `[]` | Список разрешённых алгоритмов подписи пиров, принимаемых при рукопожатии (`"ed25519"`, `"falcon512"`, гибриды). Пусто = принимать все поддерживаемые |

**Пример:**

```toml
[session]
keepalive_interval_secs    = 15
idle_timeout_secs          = 60
max_concurrent             = 2048
max_per_ip                 = 64
tx_queue_depth             = 8192
max_frame_body_bytes       = 2097152        # 2 МиБ
rekey_bytes_threshold      = 68719476736    # 64 ГиБ — чаще ротировать ключи
rekey_time_threshold_secs  = 604800         # 7 дней
```

| `max_total_sessions` | `usize` | `0` | Сколько всего сессий любого рода (relay + прямые). `0` = без ограничений |
| `tx_queue_high_watermark` | `f64` | `0.8` | Доля заполнения TX-очереди, при которой узел считается перегруженным (0.0–1.0) |
| `congestion_high` | `f64` | `0.8` | Порог congestion-score, выше которого узел перестаёт принимать новые relay-сессии |
| `congestion_low` | `f64` | `0.6` | Порог congestion-score, ниже которого узел снова начинает принимать relay-сессии (гистерезис) |
| `max_inbound_bandwidth_kbps` | `i64` | `10000000` | Суммарный лимит входящей полосы узла в kbps (по умолчанию 10 Гбит/с). `-1` = без лимита |
| `max_outbound_bandwidth_kbps` | `i64` | `10000000` | Суммарный лимит исходящей полосы узла в kbps. `-1` = без лимита |

**Пример:**

```toml
[capacity]
max_relay_sessions      = 500
max_total_sessions      = 1000
tx_queue_high_watermark = 0.75
congestion_high         = 0.75
congestion_low          = 0.5
```

## `[abuse]`

Защита от злоупотреблений: ограничение скорости, квоты mailbox, PoW, баны.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `rate_limit_fps` | `f64` | `200000.0` | Установившаяся скорость фреймов на пира (фреймов/сек) |

## `[dht]`

Настройки DHT (Kademlia) — фонового поиска узлов и хранения значений.

| Ключ | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `republish_interval_secs` | `u64` | `1800` | Как часто переопубликовываются DHT-записи (30 минут) |
| `cleanup_interval_secs` | `u64` | `60` | Как часто чистятся истёкшие DHT-записи |
| `participate` | `bool` | `true` | Участвовать в DHT-хранилище (принимать STORE/DELETE). `false` = только маршрутизация (FIND_NODE/FIND_VALUE) |
| `k` | `u8` | `20` | Kademlia k-bucket size — число контактов в ответе на FIND_NODE |
| `alpha` | `u8` | `3` | Kademlia α — параллельных запросов на раунд итеративного поиска |
| `max_rounds` | `u8` | `20` | Сколько раундов итеративного поиска пробовать, прежде чем сдаться |
| `find_node_timeout_ms` | `u64` | `2000` | Тайм-аут одного FIND_NODE/FIND_VALUE RPC, в мс |
| `vivaldi_weight` | `f64` | `0.3` | Насколько фактор топологии Vivaldi учитывается в рейтинге DHT-узлов. `0.0` = чистый XOR-порядок |
| `routing_persist_path` | `string` или отсутствует | не задано | Куда сохранять DHT-таблицу маршрутизации (k-bucket) |
| `values_persist_path` | `string` или отсутствует | не задано | Куда сохранять хранимые DHT-значения (периодический JSON-снимок всего хранилища) |
| `cold_store_path` | `string` или отсутствует | не задано | Каталог для дискового **холодного слоя RocksDB**, где лежат вытесненные DHT-значения. Когда задан (и бинарник собран с feature `rocksdb-cold` — включён по умолчанию для `veil-cli`), значения, вытесненные из горячего слоя в памяти, попадают в этот RocksDB-стор на диске вместо ограниченной холодной карты в памяти. Это переносит ограничение по числу записей с RAM на диск (выделенный DHT-узел тогда обслуживает >1M записей), а холодные записи переживают рестарт. В отличие от `values_persist_path` (периодический JSON-снимок), холодный слой — живая БД, обновляемая непрерывно. Если feature отсутствует или RocksDB не открылся, он игнорируется со строкой в логе при старте, и узел откатывается на холодный слой в памяти |
| `allow_unsigned_store` | `bool` | `false` | Принимать устаревшие **неподписанные** raw STORE. По умолчанию `false` (отвергаются). Включить обратно — мина для развёртывания, см. [OPERATIONS](OPERATIONS.md); при первом приёме один раз срабатывает предупреждение об устаревании |
| `max_store_entries` | `usize` | `25000` | Жёсткий лимит записей в DHT-сторе. Повышайте для выделенных DHT-сидов (например `250000`); чтобы выйти за RAM, сбрасывайте на диск через RocksDB-слой `cold_store_path` |
| `max_store_bytes` | `u64` или отсутствует | не задано | Необязательный лимит DHT-стора по байтам, вдобавок к `max_store_entries` |
| `per_origin_max_bytes` | `u64` или отсутствует | не задано | Байтовый лимит на одного подписанта (Этап 11e) — ограничивает, сколько может хранить один origin, чтобы один подписант не забил стор |
| `shard_filtering` | `bool` | `false` | По выбору: принимать STORE, только когда его ключ попадает в шард этого узла. По умолчанию `false`; станет включённым по умолчанию, когда сеть перевалит за ~1M узлов |

**Пример:**

```toml
[dht]
republish_interval_secs  = 3600
participate              = true
k                        = 20
alpha                    = 5
vivaldi_weight           = 0.5
routing_persist_path     = "/var/lib/veil/dht-routing.bin"
values_persist_path      = "/var/lib/veil/dht-values.bin"
cold_store_path          = "/var/lib/veil/dht-cold"
```
