# Руководство пользователя

## Что такое Veil (OVL1)?

**Veil** — это одноранговая сеть: центрального сервера нет, есть множество равных компьютеров — **узлов**, — которые передают трафик друг за друга. У каждого узла есть адрес, вычисленный из его криптографического ключа; записывается он как `node_id = BLAKE3(public_key)`. (BLAKE3 — это быстрая хеш-функция, то есть односторонний рецепт, который превращает ключ в короткий отпечаток фиксированной длины.) Адрес занимает 32 байта и никогда не меняется: он не зависит ни от вашего IP, ни от местоположения, ни от способа подключения. Приложения обращаются к veil-адресу, а сеть сама придумывает, как доставить байты.

OVL1 — это название протокола Veil, то есть общего свода правил, на котором «говорят» все узлы. Вот что он даёт:

- **Доставка сообщений** между узлами: сообщение прыгает через промежуточные узлы, и каждый пересылающий узел называется **ретранслятором**
- **Почтовый ящик** — если тот, кому вы пишете, сейчас не в сети, его сообщения ждут на узле-шлюзе
- **DHT** — общий справочник, разложенный по всем узлам (по-научному — *распределённая хеш-таблица* по схеме Kademlia). По нему ищут узлы и публикуют то, что смогут найти другие.
- **Имена** — читаемые имена вроде `@alice` вместо длинных шестнадцатеричных адресов, привязанные к узлу через небольшую задачу-головоломку (доказательство работы)
- **Потоковая передача** — двусторонние потоки данных с управлением окном, почти как TCP, но поверх Veil
- **Сквозное шифрование** — прочитать сообщение можете только вы и получатель; ретрансляторы посередине видят лишь зашифрованные байты (запечатано с помощью ML-KEM + ChaCha20-Poly1305 — современного шифрования, устойчивого даже к будущим квантовым компьютерам)
- **Локальная сеть** — быстрое широковещание по UDP в вашей локальной сети, чтобы найти соседей в том же сегменте

## Установка

Быстрый путь: взять готовую сборку. Скрипт скачивает уже собранную программу, сверяет её отпечаток sha256 — так вы убеждаетесь, что её не подменили, — и устанавливает. Ни компилятор, ни набор инструментов сборки Rust не нужны.

**Linux / macOS:**

```bash
curl --proto '=https' --tlsv1.2 -sSf \
  https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.sh | sh
```

**Windows (PowerShell):**

```powershell
irm https://raw.githubusercontent.com/veilnetwork/veil/main/scripts/install.ps1 | iex
```

Кладёт `veil-cli` в `~/.veil/bin` (`%USERPROFILE%\.veil\bin` на Windows). Добавьте `--all`, чтобы поставить ещё и `ogate` и клиент с сервером `oproxy`. Полный список компонентов, опции, настройку сервера и удаление смотрите в **[Установке и первом узле](install.md)**.

> **Windows:** здесь узел общается с админ-CLI через локальное TCP-соединение. Пара удобств, доступных только в Unix, отсутствует (перезагрузка конфигурации по сигналу SIGHUP и сокеты Unix-доменов), но всё важное работает: админ-CLI, ваш обычный veil-трафик и запуск узла на переднем плане. Задайте `global.admin_socket = "tcp://127.0.0.1:0"`; узел попросит операционную систему выбрать свободный порт, запишет `admin.port` и `admin.token` в `runtime_dir`, а клиенты прочитают их при подключении.

### Сборка из исходников

Это нужно только на платформах, где готовой сборки нет (скажем, Intel macOS), или если вы дорабатываете сам Veil. Сборка по умолчанию подключает две библиотеки на C/C++ — BoringSSL (`tls-boring`) для шифрования и RocksDB для хранения, — поэтому понадобится установленный компилятор C/C++:

```bash
# Зависимости Debian/Ubuntu:
sudo apt-get install -y cmake golang-go nasm ninja-build build-essential

git clone https://github.com/veilnetwork/veil
cd veil
cargo build --release --features veil-bootstrap/production-seeds
# Бинарники: target/release/{veil-cli,ogate,oproxy-client,oproxy-server}
cp target/release/veil-cli ~/.local/bin/
```

Подробнее: [Установка → Сборка из исходников](install.md#сборка-из-исходников).

## Быстрый запуск

### 1. Создать конфигурацию

```bash
veil-cli config init
```

Команда создаёт файл настроек `~/.config/veil/config.toml` (точный путь зависит от операционной системы) и заводит вашу личность — пару ключей, которая и есть ваш узел. Попутно она решает небольшую задачу-головоломку (доказательство работы): подбирает особое число — **nonce**, — при котором задача сходится. На это уходит несколько секунд процессорного времени. Смысл в том, чтобы штамповать поддельные личности пачками было дорого, и тогда два узла не подерутся за один и тот же адрес.

Посмотреть, где лежит файл настроек:

```bash
veil-cli config locate
```

Показать текущую конфигурацию:

```bash
veil-cli config show
```

### 2. Запустить узел

```bash
veil-cli node run
```

Узел запускается и продолжает работать в фоне. Проверить, как у него дела:

```bash
veil-cli node show
veil-cli node health
```

### 3. Посмотреть свой node_id

`node_id` — это ваш адрес в сети. Поделитесь им, чтобы другие могли до вас достучаться.

```bash
veil-cli node show
```

Пример вывода:

```
node_id:  a1b2c3d4e5f60718293a4b5c6d7e8f90a1b2c3d4e5f60718293a4b5c6d7e8f9
role:     leaf
listeners: 0
peers:    1 connected
```

Здесь `role: leaf` означает, что ваш узел сам выходит в сеть, но снаружи недоступен, — то, что нужно для ноутбука или телефона за домашним роутером. (Узел с публичным адресом, к которому подключаются другие, — это *ретранслятор*.) `peers` — сколько других узлов подключено к вам напрямую.

### 4. Подключиться к известному соседу

**Сосед (пир)** — это другой узел, с которым ваш общается напрямую. Чтобы подключиться, добавьте его в файл настроек. Понадобятся его открытый ключ, его nonce от доказательства работы и адрес транспорта — он подсказывает Veil, как до соседа добраться:

```bash
# PUBLIC_KEY, NONCE и TRANSPORT — позиционные аргументы
veil-cli peers add \
  --algo ed25519 \
  BASE64_PUBKEY \
  BASE64_POW_NONCE \
  "tls://gateway.example.com:9443"
```

Перезапустить узел, чтобы он подхватил изменение:

```bash
veil-cli node restart
```

### 5. Остановить узел

```bash
veil-cli node stop
```

## CLI — справочник команд

### `veil-cli config`

| Команда | Описание |
|---------|----------|
| `config init [PATH]` | Создать конфигурацию с новой личностью и PoW-nonce |
| `config init --difficulty N` | Задать сложность PoW для создаваемой личности (по умолчанию `16`; для боевого использования и публичных узлов берите `24` или выше) |
| `config init --force` | Перезаписать существующую конфигурацию |
| `config show` | Вывести текущую конфигурацию (private_key скрыт) |
| `config validate` | Проверить конфигурацию на ошибки |
| `config validate --fix` | Попробовать исправить найденные ошибки |
| `config locate` | Показать путь к файлу настроек |
| `config get KEY` | Получить значение по ключу (например `identity.algo`) |
| `config set KEY VALUE` | Установить значение |

### `veil-cli key`

| Команда | Описание |
|---------|----------|
| `key gen` | Сгенерировать новую пару ключей (Ed25519 или Falcon512) |
| `key gen --algo falcon512` | Использовать постквантовый алгоритм |

### `veil-cli node`

| Команда | Описание |
|---------|----------|
| `node run` | Запустить узел |
| `-c PATH node run` | Запустить с нестандартной конфигурацией (глобальный флаг `-c`/`--config` идёт **перед** подкомандой) |
| `node stop` | Мягко остановить узел |
| `node restart` | Перезапустить (stop + run) |
| `node show` | Показать сводку (node_id, роль, сессии) |
| `node health` | Убедиться, что узел отвечает, и показать число открытых сессий |

### `veil-cli listen`

| Команда | Описание |
|---------|----------|
| `listen add tcp://0.0.0.0:9000` | Добавить слушатель в конфигурацию (TRANSPORT — позиционный URI; есть опции `--advertise`/`--relay`) |
| `listen del LISTEN_ID` | Удалить слушатель (ID — позиционный) |
| `listen list` | Список активных слушателей |

### `veil-cli peers`

| Команда | Описание |
|---------|----------|
| `peers add [--algo ALGO] PUBLIC_KEY NONCE TRANSPORT [--alt-uri URI]` | Добавить пир (PUBLIC_KEY, NONCE, TRANSPORT — позиционные) |
| `peers del PEER_ID` | Удалить пир из конфигурации (ID — позиционный) |
| `peers list` | Список сконфигурированных пиров |

### `veil-cli sessions`

| Команда | Описание |
|---------|----------|
| `sessions list` | Активные OVL1-сессии (peer node_id, роль, RTT) |
| `sessions stats` | Агрегированная статистика сессий |

### `veil-cli debug`

| Команда | Описание |
|---------|----------|
| `debug ping NODE_ID [--count N] [--interval MS] [--timeout MS]` | Veil ping до node_id (64 hex-символа) |
| `debug trace TARGET [--max-hops N]` | Traceroute по veil-сети |
| `debug capture [--node-id HEX] [--family N] [--limit N]` | Захват фреймов для отладки |

`debug ping` и `debug trace` принимают только сам `NODE_ID` (64 шестнадцатеричных символа). Если у вас вместо этого `@name`, сначала превратите его в адрес через `node resolve-name` (см. раздел «Имена» ниже).

## Имена (Name System)

`node_id` — это 64 шестнадцатеричных символа: компьютеру удобно, человеку мучительно. Поэтому Veil позволяет заявить читаемое имя вроде `@alice` и привязать его к вашей личности. Раздать имена за вас никто не может, как и тихо увести ваше: имя заперто на ваши ключи, а небольшая задача-головоломка (доказательство работы) делает захват имён пачками дорогим. Имена живут в команде `identity` (отдельной команды `name` нет), а превратить `@name` обратно в адрес можно через `node resolve-name`.

### Заявить имя

```bash
veil-cli identity claim-name alice
```

Имя — позиционный аргумент. Оно приводится к нижнему регистру в простом ASCII, и допустимы только символы `[a-z0-9#_-]`. Команда решает задачу-головоломку (доказательство работы) — тем сложнее, чем реже и желаннее имя, — затем подписывает запись `NameClaim` вашим активным ключом подписи (`identity_sk`) и сохраняет её в `<veil_dir>/name_claims/<name>.bin`. Если ваш узел запущен, он объявляет заявку в DHT, чтобы её нашли другие: либо при перезапуске, либо при ближайшем плановом повторном объявлении (раз в 6 часов).

Опция `--veil-dir PATH` указывает на другой каталог личности (по умолчанию `~/.config/veil` или путь из `$VEIL_IDENTITY_DIR`).

### Найти имя

```bash
veil-cli node resolve-name @alice
```

Принимает имя и возвращает адрес, который за ним стоит. Подходят и `alice`, и `@alice`. Команда проходит по цепочке от `NameClaim` к полному документу личности `IdentityDocument` и проверяет каждый шаг — что доказательство работы достаточно сильное, что запись свежая и что имя действительно подписано текущим ключом личности, — так что полученному адресу можно доверять. Опция `--timeout-ms N` ограничивает суммарное время поиска (по умолчанию 5000 мс).

### DHT-ключ имени

Вычислить DHT-ключ, под которым хранится `NameClaim` для заданного имени, — это чисто локальный расчёт, без обращения к сети:

```bash
veil-cli identity name-dht-key alice
```

## Использование приложением (IPC)

Хотите, чтобы ваше приложение само отправляло и принимало трафик через Veil? С локальным узлом оно общается по IPC (межпроцессному взаимодействию) — это небольшой сокет на вашей же машине, к которому подключается ваша программа. Как это сделать:

1. Включите IPC в файле настроек:

```toml
[ipc]
enabled = true
socket_uri = "unix:///home/user/.veil/app.sock"
```

Обратите внимание: ключ называется `socket_uri` и ждёт полный URI — `unix:///абс/путь` на Linux/macOS или `tcp://127.0.0.1:0?runtime_dir=/абс/путь` на Windows, — а не обычный путь к файлу. Тильда `~` в URI понимается буквально и не раскрывается в вашу домашнюю папку, поэтому пишите абсолютный путь целиком. Если `socket_uri` не задан, на Unix узел использует запасной путь `~/.veil/app.sock`.

2. Подключитесь к сокету и выполните начальное рукопожатие (короткий обмен «привет — подтверждение», который устанавливает соединение):

```python
# Пример на Python (упрощённо)
import socket, struct, json

sock = socket.socket(socket.AF_UNIX)
sock.connect("/home/user/.veil/app.sock")

def send_msg(s, obj):
    body = json.dumps(obj).encode()
    s.sendall(struct.pack(">I", len(body)) + body)

def recv_msg(s):
    n = struct.unpack(">I", s.recv(4))[0]
    return json.loads(s.recv(n))

send_msg(sock, {"command": "hello", "version": 1})
resp = recv_msg(sock)  # {"command": "hello_ok"}

send_msg(sock, {
    "command": "bind",
    "namespace": "myapp",
    "app_name": "main",
    "endpoint_id": 1
})
resp = recv_msg(sock)  # {"command": "bind_ok", "app_id": "...hex..."}
```

Подробнее об IPC-протоколе — в [Спецификации протокола](protocol-spec.md#ipc-протокол-localapp).

## Типичные сценарии использования

### Одноранговый мессенджер

1. Каждый узел знает node_id другого (или его имя — `@alice` или `@bob`)
2. Приложение отправляет `StreamOpen` и получает в ответ двусторонний поток для общения
3. Всё шифруется сквозным шифрованием (ML-KEM + ChaCha20-Poly1305), поэтому ретрансляторы посередине ничего не прочтут
4. Если получатель не в сети, сообщение ждёт в его почтовом ящике на шлюзе, пока он не вернётся

### Публикация ресурса в DHT

Через IPC или напрямую через admin API:
```bash
veil-cli node dht put HEX_KEY HEX_VALUE
```

### Мониторинг сети

```bash
veil-cli sessions list     # Активные соединения
veil-cli node routes       # Известные маршруты (route cache)
veil-cli node dht list     # Локальное хранилище DHT
veil-cli node metrics      # Счётчики фреймов, сессий, доставок
```
