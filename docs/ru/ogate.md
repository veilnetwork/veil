# ogate: мост для IP-трафика поверх Veil

`ogate` позволяет группе машин общаться через Veil так, будто они стоят
в одной офисной сети, — это **виртуальная локальная сеть** (LAN, Local
Area Network). Каждая машина получает свой приватный IP-адрес, и дальше
её можно пинговать, заходить по SSH, запускать любое обычное сетевое
приложение — хотя на деле машины разбросаны по всему интернету.

Работает это через **TUN-устройство** — виртуальный сетевой адаптер,
который операционная система отдаёт программе вместо настоящего «железа».
Всё, что машина отправляет в этот адаптер, `ogate` подхватывает и
пересылает нужному Veil-пиру; всё, что приходит от пира, `ogate`
записывает обратно в адаптер. Каждый хост открывает одно TUN-устройство
с заданным IPv4/IPv6-адресом, а `ogate` маршрутизирует IP-пакеты между
ним и остальными пирами по таблице пиров, которую вы задаёте.

Кто именно может слать пакеты, решает один из двух режимов:

* **`open`** — любой пир, знающий пару `(network, app)`, может слать
  пакеты внутрь.
* **`authorized`** — проходят только пиры, чей `node_id` есть в списке
  `peers[]`. Вдобавок IP-адрес отправителя в пакете должен совпадать
  с тем виртуальным IP, который этому пиру положен. Так пир не сможет
  выдать себя за чужой адрес (защита от подмены, anti-spoof).

`ogate` — обычное *приложение поверх* veil-демона, а не его часть. Один
демон может обслуживать сразу несколько экземпляров `ogate`: каждая пара
`(network, app)` получает свой отдельный канал к демону.

## Как это сочетается с P-Net

Режим доступа `ogate` выбирается отдельно от того, как настроен
[P-Net](p-net.md) — членство на уровне самой сети Veil. Вместе они дают
такие сочетания:

| Veil режим | ogate режим | Эффект |
|---|---|---|
| public | open | любой veil-пир заходит в LAN |
| public | authorized | LAN формируют только перечисленные `node_id` |
| private (P-Net) | open | любой P-Net-член заходит в LAN |
| private (P-Net) | authorized | двухуровневый список доступа (P-Net + ogate) |

Если в сети Veil могут оказаться недоверенные пиры, выбирайте
`authorized`.

## Платформы

| Платформа | Реализация TUN | Настройка адреса |
|---|---|---|
| Linux | crate `tun` (`/dev/net/tun`, IFF_NO_PI) | crate ставит ipv4; ipv6 через `ip -6` |
| macOS | crate `tun` (utun) | crate ставит ipv4; ipv6 через `ifconfig` |
| Windows | crate `tun` (WinTun) | crate ставит ipv4; ipv6 через `netsh` |
| FreeBSD | прямой `/dev/tunN` + `TUNSIFHEAD` ioctl | `ifconfig inet ... up` |

На FreeBSD `iface_name` должен начинаться с `tun` — номер выделяет ядро.
Хотите понятное имя? После запуска переименуйте:
`ifconfig tunN name myiface`.

## Конфигурация

`/etc/ogate/ogate.toml`:

```toml
network       = "homenet"
app           = "ogate"
mode          = "authorized"
socket_path   = "/var/lib/veil/app.sock"
iface_name    = "ogate0"
mtu           = 1280
local_addr_v4 = "10.99.0.1"
prefix_v4     = 24
local_addr_v6 = "fd00:1::1"
prefix_v6     = 64
endpoint_id   = 1

[[peers]]
node_id = "<64-hex node_id пира>"
addr_v4 = "10.99.0.2"
addr_v6 = "fd00:1::2"
name    = "host-b"

[[peers]]
node_id = "<ещё один 64-hex node_id>"
addr_v4 = "10.99.0.3"
name    = "host-c"
```

Обязателен хотя бы один из `local_addr_v4` / `local_addr_v6`, плюс
подходящий к нему список пиров.

### Как назначаются подсети и IP

Вручную, в этом же файле конфигурации. (**Подсеть** — это просто блок
адресов, который занимает ваша виртуальная сеть, например
`10.99.0.0/24`.) Вы выбираете подсеть, а затем по одному виртуальному IP
каждому пиру. Два правила держат всех в согласии:

* **Все пиры одной сети сидят в одной подсети.**
* **Таблицы пиров зеркалят друг друга.** `local_addr_v4` хоста A — это
  `peers[A].addr_v4` у хоста B, и наоборот.

В режиме `authorized` пакет, у которого IP отправителя не совпадает с
таблицей, отбрасывается как подменённый адрес.

Хотите, чтобы адреса раздавались автоматически — скажем, выводились из
`node_id`, — а не назначались руками? В заметках по дизайну
[`../../crates/ogate/README.md`](../../crates/ogate/README.md) намечено
несколько вариантов, но готового автоматического реестра пока нет.

## Как вычисляется app_id

У каждого канала ogate есть `app_id`, который его обозначает. Считается
он так: `app_id = BLAKE3(node_id || "ogate.<network>" || <app>)`.
Поскольку это чистая математика, каждый пир сам вычисляет `app_id`
другого — не нужно ниоткуда выкачивать список пиров.

```bash
$ ogate app-id --network homenet --node-id $(cat host-a.nodeid)
namespace = ogate.homenet
name      = ogate
app_id    = 3c4e9f...
```

## Быстрый старт (два хоста)

### 1. Узнайте `node_id` каждого хоста

```bash
sudo -u veil veil-cli --config /var/lib/veil/node.toml \
    node identity
```

### 2. `/etc/ogate/ogate.toml` на хосте A

```toml
network       = "homenet"
app           = "ogate"
mode          = "authorized"
socket_path   = "/var/lib/veil/app.sock"
iface_name    = "ogate0"
mtu           = 1280
local_addr_v4 = "10.99.0.1"
prefix_v4     = 24

[[peers]]
node_id = "<node_id хоста B в hex>"
addr_v4 = "10.99.0.2"
name    = "host-b"
```

### 3. Зеркально настройте хост B

Задайте `local_addr_v4 = "10.99.0.2"` и добавьте запись `peers` для
хоста A с `addr_v4 = "10.99.0.1"`.

### 4. Запустите на обоих хостах (от пользователя демона, с CAP_NET_ADMIN)

От какого пользователя запускать `ogate`, определяют две вещи.

Во-первых, `ogate` подключается к демону по локальному каналу, а демон
по проверке uid (U9) **отбрасывает любое подключение от пользователя,
отличного от его собственного, — включая root, без исключений.** Значит,
`ogate` обязан работать от *того же* пользователя, что и veil-демон
(например, `veil`), а не от root.

Во-вторых, для открытия TUN-устройства обычно нужны права root. Но
запускать от root не обязательно: выдайте тому же пользователю всего одно
право, которое это разрешает, — `CAP_NET_ADMIN` (право Linux управлять
сетевыми интерфейсами):

```bash
# Разово: разрешаем пользователю демона открывать TUN без root.
sudo setcap cap_net_admin+ep "$(command -v ogate)"

# Запуск от пользователя демона (совпадает с uid, под которым крутится демон):
sudo -u veil ogate up --config /etc/ogate/ogate.toml
```

### 5. Пропингуйте через виртуальный IP

```bash
ping 10.99.0.2   # с хоста A
```

## Перезагрузка без перезапуска (SIGHUP)

Часть настроек можно поменять на работающем мосте, не останавливая его.
Отредактируйте конфигурацию и пошлите процессу сигнал SIGHUP:

```bash
sudo kill -HUP "$(pidof ogate)"
# Или через helper:
sudo ogate reload --pid "$(pidof ogate)"
# Или через systemd:
sudo systemctl reload ogate
```

Мост перечитает конфигурацию и одним неделимым шагом подменит таблицу
маршрутизации. Пакеты, что уже в пути, не теряются.

**Можно перезагрузить:** `mode`, `peers[]`.

**Нужен полный перезапуск:** `network`, `app`, `endpoint_id`,
`iface_name`, `mtu`, `local_addr_v4`, `local_addr_v6`, `prefix_v4`,
`prefix_v6`, `socket_path`. Если попытаться сменить одно из этих полей
через SIGHUP, мост предупредит об этом и продолжит работать как был.

Если перезагрузка по любой причине не удалась — опечатка в файле,
значение не прошло проверку или попытка сменить поле, требующее
перезапуска, — это попадёт в лог, а прежняя конфигурация останется
активной. Момента, когда мост остался бы в сломанном состоянии, не
бывает.

## CLI

```
ogate up         --config <path>      Поднять TUN + bridge до SIGINT/SIGTERM.
ogate show       --config <path>      Распечатать конфиг + посчитать app_id
                                      (без открытия TUN / IPC).
ogate reload     --pid <pid>          Послать SIGHUP запущенному процессу.
ogate app-id     --network <net> --node-id <hex>
                                      Посчитать app_id одного пира
                                      (удобно при bootstrap'е конфигов).
ogate gen-config [-o <path>]          Вывести закомментированный шаблон конфига (TOML).
                                      Без -o ⇒ stdout (пайп в less / в твой редактор).
                                      С -o ⇒ пишет файл (отказывается перезаписывать
                                      существующий).
```

Флаги: `-v` — уровень debug, `-vv` — trace. Либо переопределите через
переменную окружения `RUST_LOG=ogate=debug`.

### Создание конфигурации с нуля

```bash
# 1. Сгенерировать шаблон (отказывается перезаписывать, если файл уже есть).
sudo -u veil ogate gen-config -o /etc/ogate/ogate.toml

# 2. Отредактировать — минимум: имя network, local_addr_v4, записи [[peers]].
#    Inline-комментарии `#` в шаблоне поясняют каждый параметр.
sudo vim /etc/ogate/ogate.toml

# 3. (Опционально) Проверить разобранный конфиг, не открывая ни одного устройства.
sudo -u veil ogate show --config /etc/ogate/ogate.toml

# 4. Поднять.
sudo -u veil ogate up --config /etc/ogate/ogate.toml
```

## Настройка среды выполнения и логов

И то, и другое настраивается в файле конфигурации. Та же схема работает
в `veil-cli` и `oproxy`.

### `[runtime]` — настройки среды выполнения

Здесь настраивается асинхронная среда выполнения (tokio), на которой
работает `ogate`, — в основном сколько потоков она использует.

```toml
[runtime]
flavor               = "multi_thread"   # | "current_thread"
worker_threads       = 4                # только для multi_thread
max_blocking_threads = 64
thread_keep_alive_ms = 10000
thread_name          = "ogate"
thread_stack_size    = 2097152
```

Все поля необязательные. `flavor` понимает и старое имя
`runtime_flavor`. Нулевое значение `worker_threads` или
`max_blocking_threads` означает «оставить незаданным» — это защищает от
паники tokio, которую вызвал бы буквальный `0`.

**Переменные окружения** — применяются после загрузки конфигурации,
поэтому имеют приоритет над файлом:

| Переменная окружения | Эффект |
|---|---|
| `OGATE_RUNTIME` | `current_thread` или `multi_thread` |
| `OGATE_WORKERS` | число рабочих потоков |
| `OGATE_MAX_BLOCKING_THREADS` | предел пула блокирующих потоков |

Они нужны в основном ради совместимости: чтобы старые systemd-юниты,
написанные ещё до появления секции `[runtime]`, продолжали работать,
передавая настройки через окружение.

### `[logging]` — настройки вывода логов

Здесь задаётся, что логировать и куда это отправлять.

```toml
[logging]
level  = "info"                    # off | error | warn | info | debug | trace
format = "text"                    # text | json
file   = "/var/log/ogate.log"      # optional — по умолчанию stderr
```

| Поле | По умолчанию | Описание |
|---|---|---|
| `level` | `info` | Самый низкий уровень, который попадает в лог. `off` выключает логирование полностью — ничего не регистрируется, так что оно ничего не стоит и хранит полную тишину. |
| `format` | `text` | `text` = удобно читать глазами; `json` = одно структурированное событие в строке, для машинной обработки. |
| `file` | (stderr) | Необязательный путь к файлу. Новые строки **дописываются** (файл создаётся, если его нет), а родительская директория должна уже существовать. Запись неблокирующая, поэтому логирование никогда не тормозит из-за медленного диска. |

**Что имеет приоритет** (от высшего к низшему):

1. переменная окружения `RUST_LOG` (если задана, всегда побеждает)
2. флаги `-v` / `-vv` (когда > 0)
3. `[logging] level` в конфигурации
4. значение по умолчанию (`info`)

**Примеры:**

```toml
# Полностью тихий режим (никаких логов)
[logging]
level = "off"

# JSON-логи в файл для log-shipping (Promtail / Vector / Fluent Bit)
[logging]
level  = "info"
format = "json"
file   = "/var/log/ogate.json"

# Verbose в stderr (по умолчанию systemd → journald)
[logging]
level  = "debug"
format = "text"
```

```bash
# Override уровня через env при разовом запуске:
RUST_LOG="ogate=debug,veilclient=info" sudo -u veil ogate up --config /etc/ogate/ogate.toml
```

## Раскатка через Ansible

В репозитории лежат два плейбука: `ansible/deploy-ogate.yml` —
развернуть `ogate`, и `ansible/remove-{chat,chaos-ban}.yml` — убрать
старые тестовые нагрузки. Развёртывание идёт по одному хосту за раз
(`serial: 1`), а конфигурация для каждого хоста собирается из
`manifest.json` плюс карты `host_to_ogate_addr`, заданной внутри
плейбука.

```bash
# Раскатить:
ansible-playbook -i inventory.yml deploy-ogate.yml

# Обновить peer-таблицу без рестарта (после правки /etc/ogate/ogate.toml):
ansible all -i inventory.yml -m systemd \
    -a "name=ogate state=reloaded" --become

# Остановить везде:
ansible all -i inventory.yml -m systemd \
    -a "name=ogate state=stopped enabled=no" --become
```

Текущая тестовая сеть работает в `mode=authorized` поверх
`192.168.0.0/16`:
* bootstrap-ноды: `192.168.0.1`–`.3`
* leaf-ноды: `192.168.0.11`–`.15`

## Как это устроено внутри

* **Транспорт**: каждый IP-пакет, пришедший с TUN-устройства,
  заворачивается в сообщение `AppIpcSend`, передаётся демону и уходит
  наружу как обычная veil-датаграмма. `ogate` **не** использует старое,
  объявленное устаревшим семейство сообщений `Tunnel` на стороне демона.
* **Во что обходится один пакет**: чтение из TUN → запись в локальный
  сокет → демон шифрует пакет сквозным шифрованием → в сеть → демон
  расшифровывает → обратно через сокет → запись в TUN. Этот локальный
  сокет на Linux держит больше 100 000 мелких сообщений в секунду, так
  что если что-то и станет узким местом, то скорее криптография демона
  или сама сеть, а не `ogate`.
* **Авторизация** выполняется в самом `ogate`, когда пакеты приходят. В
  режиме `authorized` каждый входящий пакет проверяется дважды: что его
  `src_node_id` есть в таблице пиров И что его `src_ip` совпадает с
  `peers[src_node_id].addr_*`.
* **Защита от подмены**: благодаря этой второй проверке в режиме
  `authorized` пир не может протолкнуть пакеты, выдавая их за чужой
  виртуальный IP.

## Ограничения и открытые вопросы

* Маршрутами `ogate` не управляет — только настраивает один маршрут до
  своей подсети. Если нужен маршрут до конкретного хоста через пира,
  добавьте его вручную через `ip route` / `route add`.
* Ни NAT, ни пересылки трафика. `ogate` — это оконечный узел, а не
  **шлюз** (машина, которая передаёт трафик дальше, в другие сети).
  Хотите, чтобы он работал как шлюз, — включите пересылку сами
  (`net.ipv4.ip_forward=1` на Linux) и добавьте свои правила NAT.
* Производительность пока не замеряли. Перед тем как браться за
  оптимизацию, измерьте её с помощью `iperf3`.
