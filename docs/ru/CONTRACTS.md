# Сетевые контракты OVL1

Этот документ описывает три вещи: что veil-сеть **гарантирует** приложениям,
что она явно **не гарантирует** и как **версионируется** протокол. Это
авторитетный справочник, если вы строите приложение поверх OVL1.

Под **контрактом** (договором интерфейса) здесь понимается обещание сети о её
поведении — то, на что можно опереться, не читая реализацию. Если у гарантии
есть условие, то условие и есть оговорка: обещание выполняется, только когда
это условие соблюдено.

---

## 1. Доставка

**Протокол:** плоскость `IPC_SEND` / `DELIVERY_FORWARD`.

### Что гарантируется

| Гарантия | Условие |
|----------|---------|
| **Доставка по принципу «как получится» (best-effort)** | Всегда. Сеть делает одну попытку переслать пакет по единственному пути. Повторных передач на сетевом уровне нет. |
| **Подтверждение «хотя бы раз» (at-least-once)** | Когда отправитель ставит флаг `IPC_SEND_FLAG_REQUIRE_ACK`. Тогда узел повторяет отправку, пока почтовый ящик получателя не вернёт `DELIVERY_ACK`. |
| **Подавление дубликатов у получателя** | Узел-получатель отбрасывает дубликаты по `content_id` (32-байтному идентификатору). Приложения, которые обходят слой почтовых ящиков, дубликаты всё же могут увидеть. |

### Что НЕ гарантируется

- **Порядок** между независимыми датаграммами. Два сообщения, отправленные одному соседу, могут прийти в любом порядке.
- **Доставка** без `IPC_SEND_FLAG_REQUIRE_ACK`. Кадр может быть отброшен на любом прыжке, без всякого предупреждения.
- **Своевременность**. Гарантий по задержке нет. Перегрузка сети, смена маршрутов или ушедший в оффлайн сосед могут задержать доставку на сколько угодно — вплоть до `ttl_secs`.

### Ключевые константы

- `MAX_CLOCK_SKEW_SECS = 300` — поле `created_at` в конверте не может быть больше чем на 5 минут в будущем.
- `MAX_RELAY_HOPS = 16` — конверты, превысившие этот счётчик прыжков, отбрасываются ретрансляторами.

## 2. Потоки

**Протокол:** плоскость `APP_OPEN` / `APP_DATA` / `APP_CLOSE` (прикладной слой OVL1).

### Что гарантируется

| Гарантия | Условие |
|----------|---------|
| **Доставка по порядку** | Все байты внутри одного `stream_id` приходят в том порядке, в котором были отправлены. |
| **Надёжная доставка** | Транспорт нижележащей сессии сам переотправляет потерянные кадры. Слой потоков своей логики переотправки не добавляет. |
| **Полузакрытие (half-close)** | Любая сторона может послать `APP_CLOSE` — это сигнал, что слать данные она больше не будет. Другая сторона шлёт дальше, пока не закроется тоже. (Это и есть *полузакрытие*: одно направление закрыто, второе остаётся открытым.) |
| **Управление потоком** | Отправитель не выходит за `initial_window`, объявленное в `APP_RECEIPT`. Получатель обязан слать обновления окна, иначе отправитель встанет. |

### Что НЕ гарантируется

- **Порядок между разными потоками**. Байты в `stream_id = 1` и `stream_id = 2` могут перемежаться как угодно.
- **Доставка при разрыве сессии**. Если нижележащая сессия обрывается, открытые потоки прекращаются без уведомления — доставка `APP_CLOSE` соседу не гарантируется.

### Ключевые константы

- `MAX_STREAM_SEND_WINDOW = 16 MiB` — предельный объём байт «в полёте» (отправленных, но ещё не подтверждённых) на один поток. Отправитель, упёршийся в этот предел, притормаживается, пока получатель не нагонит.
- `MAX_STREAM_INITIAL_WINDOW = 16 MiB` — верхняя граница для `initial_window`, которое может объявить сосед; применяется и при открытии потока, и при каждом `window_update`. Без неё сосед мог бы объявить огромное окно и заставить локальную сторону буферизовать без предела (U3).

## 3. Безопасность

**Протокол:** рукопожатие сессии + слой сквозного (E2E) шифрования.

### Личность

- **Личность узла**: `node_id = BLAKE3(public_key_bytes)`. Ключ подписи — Ed25519 **или** Falcon-512, выбирается для каждого узла. Рукопожатие сессии (`SESS_IDENTITY` / `SESS_CONFIRM`) доказывает, что у узла есть соответствующий закрытый ключ. Без этого ключа ретранслятор не может выдать себя за соседа.
- **Конфиденциальность сессии**: как только рукопожатие завершено, каждый кадр сессии шифруется общим ключом. Ключ выводится из эфемерного обмена X25519 по Диффи — Хеллману (через HKDF-SHA256). Содержимое кадров ретрансляторам недоступно.

### Сквозное шифрование

- **Непрозрачность содержимого**: полезная нагрузка приложения зашифрована сквозным шифрованием (`ChaCha20-Poly1305`) между узлом-отправителем и узлом-получателем. Ретрансляторы видят только `dst_node_id` и `content_id`.
- **Аутентификация отправителя**: по умолчанию `node_id` отправителя заверен внутри шифртекста. Проверить его может только получатель.
- **Защита от повторов (replay)**: узел-получатель держит 32-секундное окно защиты от повторов по `content_id`. Соседа, который повторяет сообщение вне этого окна, банят. (*Повтор*, replay, — это перехваченное сообщение, отправленное заново, чтобы спровоцировать повторную обработку.)

### Что НЕ гарантируется

- **Прямая секретность на уровне датаграммы**. Текущий слой E2E шифрует на долговременный открытый ключ получателя, поэтому будущая компрометация ключа может раскрыть прошлые датаграммы. Ключи сессии дают прямую секретность только на транспортном уровне. (*Прямая секретность* (forward secrecy) означает: украв сегодняшние ключи, нельзя расшифровать вчерашний трафик.)
- **Анонимность от получателя**. Получатель всегда узнаёт `node_id` отправителя из заверенного шифртекста — если только не выставлен `IPC_SEND_FLAG_ANONYMOUS` (см. §4).

## 4. Приватность

**Протокол:** `IPC_SEND_FLAG_ANONYMOUS` + мета-E2E-шифрование.

### Анонимная отправка

Когда клиент выставляет `IPC_SEND_FLAG_ANONYMOUS` в `IPC_SEND`:

1. Узел вызывает `meta_encrypt` вместо обычного `encrypt`. Внешний `DeliveryEnvelope` тогда несёт `sender_node_id = [0u8; 32]` — все нули.
2. **Ретрансляторы** видят только `dst_node_id`. Личность отправителя им не видна.
3. **Получатель** расшифровывает мета-E2E-шифртекст и восстанавливает эфемерный ключ отправителя — но **не** устойчивый `node_id`, если только приложение само не вложило его в полезную нагрузку.

### Что НЕ гарантируется

- **Анонимность отправителя от получателя**. Мета-E2E-схема берёт на каждое сообщение новую эфемерную пару ключей, так что получатель не может связать два анонимных сообщения с одним отправителем. Но отправитель может выдать себя сам: если содержимое опознаваемо, анализ трафика анонимность вскроет.
- **Анонимность на уровне сети**. Ретрансляторы пересылают кадры по `dst_node_id`. Противник, наблюдающий несколько прыжков, может сопоставить отправителя и получателя.
- **Анонимность прокси**. Выходной SOCKS5-прокси (`IPC_PROXY_*`) знает, какой узел открыл соединение.

## 5. DHT

**Протокол:** распределённая хеш-таблица на базе Kademlia поверх плоскости обнаружения.

### Что гарантируется

| Гарантия | Условие |
|----------|---------|
| **k-кратное хранение** | Каждое значение DHT хранится на не более чем `k = 20` узлах, ближайших к его ключу. |
| **Хранение с ограничением по TTL** | Значения не живут вечно. Каждое истекает по TTL (времени жизни), заданному опубликовавшим его узлом. |
| **Поиск за O(log N)** | Поиск `FIND_VALUE` сходится за `O(log N)` раундов, пока меньше `k/2` узлов в каждом k-бакете ведут себя недобросовестно. (*k-бакет* — это набор известных соседей в одном срезе адресного пространства; *недобросовестные*, или византийские, узлы могут лгать и отбрасывать запросы, а не просто уходить в оффлайн.) |
| **Итеративная маршрутизация** | Инициатор ведёт поиск сам, обращаясь к каждому узлу напрямую, а не поручая одному узлу разыскать ответ за него. Поэтому один вредоносный узел способен испортить максимум один шаг поиска, а не весь поиск целиком. |

### Что НЕ гарантируется

- **Согласованность**. DHT согласована лишь в пределе. Пока узлы входят и выходят (*отток*, churn), поиск может вернуть устаревшее значение или вовсе не найти его.
- **Устойчивость к недобросовестным узлам сверх `k/2` на бакет**. Если больше половины узлов в конкретном k-бакете враждебны, поиски через этот бакет могут возвращать неверные ответы.
- **Сохранность при полном разделении сети**. Значение поддерживает живым только опубликовавший его узел — повторными публикациями. Если этот узел уйдёт в оффлайн до очередной публикации, значение пропадёт по истечении TTL.

### Ключевые константы

- `K = 20` — число ближайших узлов на бакет; коэффициент репликации.
- `MAX_NODES_PER_RESPONSE = 32` — максимум контактов, возвращаемых в одном ответе на `FIND_NODE`.

## 6. Маршрутизация

**Протокол:** путь ретрансляции `DELIVERY_FORWARD` + рассылка слухов `RouteAnnounce`.

### Что гарантируется

| Гарантия | Условие |
|----------|---------|
| **Несколько путей по принципу «как получится»** | `RouteCache` хранит до `MAX_ROUTES_PER_DST = 4` кандидатов на следующий прыжок к каждому получателю. При отправке узел берёт лучшего по оценке. |
| **Защита от петель** | Каждый прыжок через ретранслятор увеличивает `relay_hops`. Как только `relay_hops >= MAX_RELAY_HOPS`, кадр отбрасывается, так что бесконечно кружить он не может. |
| **Расщеплённый горизонт (split-horizon)** | Кадр, пришедший от соседа P, никогда не пересылается прямо обратно к P (ретранслятор проверяет `via_node_id != src_peer`). Это не даёт кадру скакать туда-сюда между двумя ретрансляторами. |
| **Истечение TTL** | Кадр отбрасывается перед пересылкой, как только `created_at + ttl_secs < now`. Это ограничивает, сколько устаревший трафик может задержаться в сети. |

### Что НЕ гарантируется

- **Гарантированная доставка**. Маршрутизация работает по принципу «как получится». Кадр может быть отброшен, если маршрута нет или маршрут устарел между отправкой и доставкой.
- **Границы по задержке**. Гарантий нет. Каждый лишний прыжок добавляет задержку, а насколько — зависит от формы сети и её перегруженности.
- **Стабильные пути**. Выбранный маршрут может меняться от кадра к кадру даже для одного и того же получателя.

### Ключевые константы

- `MAX_RELAY_HOPS = 16` — сколько прыжков через ретрансляторы кадр может пройти до отбрасывания.
- `MAX_ROUTES_PER_DST = 4` — сколько кандидатов на следующий прыжок `RouteCache` держит на каждого получателя.
- `MAX_ROUTES_PER_VIA = 256` — для скольких получателей один ретранслятор может быть следующим прыжком.
- `MAX_ROUTE_ANNOUNCE_AGE_SECS = 300` — объявления в рассылке слухов старше 5 минут отбрасываются.
- `MAX_ROUTE_ANNOUNCE_SKEW_SECS = 30` — объявления, датированные будущим, отбрасываются.

## Версионирование протокола

### IPC-протокол (client ↔ node)

Версия IPC-протокола задана в `proto/ipc.rs`:

```rust
pub const IPC_PROTOCOL_VERSION: u16 = 1;  // текущая wire-версия
pub const CLIENT_MIN_VERSION: u16   = 1;  // самая старая принимаемая версия клиента
pub const CLIENT_MAX_VERSION: u16   = 1;  // самая новая принимаемая версия клиента
```

Если `version` клиента вне `[CLIENT_MIN_VERSION, CLIENT_MAX_VERSION]`, узел отклоняет клиента с `ipc_hello_err::VERSION_MISMATCH`.

### Сетевой протокол OVL1

Версия протокола на уровне сессии хранится в `HelloPayload::ovl1_version`. Ломающее изменение увеличивает это поле на единицу. Если старый узел и новый не могут сойтись на общей версии, они разрывают соединение.
