# Руководство оператора — резервная копия и восстановление гибридной личности

Это руководство для оператора по постквантовой *гибридной* личности
(`ed25519+falcon512`). Оно проведёт вас через создание гибридной
личности, выбор того, что хранить в резерве, и восстановление на
свежей машине. *Гибридная* означает, что личность собрана сразу из
двух разных систем ключей — об этом ниже. Прочитайте это **до** того,
как запустите `veil-cli identity create --algo hybrid` в продакшене.

> **Коротко**: гибридной личности нужны **две** отдельные резервные
> копии — бумажная сид-фраза BIP-39 **и** файл `master_falcon.bin`.
> Сид-фраза (фраза-ключ) — это стандартный способ записать ключ
> двадцатью четырьмя обычными словами; файл хранит второй ключ.
> Потеряете любую из копий — личность не восстановить как гибридную,
> а ваш `node_id` (адрес, по которому вас все знают) изменится.

## Зачем две копии

Гибридный мастер-ключ — это две пары ключей, склеенные вместе. Пара
ключей — это согласованные открытый ключ (его можно показывать) и
закрытый (его держат в тайне):

| Половина | Алгоритм | Откуда восстанавливается |
|---|---|---|
| Классическая | Ed25519 | Бумажная сид-фраза BIP-39 (24 слова) |
| Постквантовая | Falcon-512 | Только файл `master_falcon.bin` |

Сид-фраза каждый раз в точности воссоздаёт половину Ed25519 — для
этого все 24 слова и нужны. Ed25519 — это классическая схема подписи;
Falcon-512 — постквантовая, она остаётся надёжной даже против
будущего квантового компьютера. Загвоздка вот в чём: половина
Falcon-512 рождается заново из системного генератора случайных чисел
(`OsRng`) в момент создания личности. Восстановить её из фразы
нельзя — текущий крейт pqcrypto-falcon такого пути не даёт. Поэтому
единственная копия половины Falcon — это файл на диске. Потеряете
файл — потеряете половину.

Вот почему это важно. Ваш `node_id` — это хеш *обоих* открытых
ключей: `BLAKE3(ed_pk(32) || falcon_pk(897))`, 929 байт, хешированных
вместе. Без половины Falcon вы не воспроизведёте этот 929-байтный
открытый ключ, поэтому при восстановлении не останется выбора, кроме
как создать свежую пару ключей Falcon, — а это даст **другой**
`node_id`. Регистрация вашего @name, контакты и репутация — всё
держится на старом `node_id`, так что всё это будет потеряно.

## Шаг 1: создать гибридную личность

```bash
veil-cli identity create \
    --algo hybrid \
    --label my-laptop \
    --veil-dir /var/lib/veil
```

CLI выводит сид-фразу BIP-39 из 24 слов в stdout и записывает:

```
/var/lib/veil/
├── identity_document.bin     # 1860 B — подписанный sovereign-документ
├── device_identity_sk.bin    # 32 B   — per-device Ed25519 subkey
├── instance_id               # 16 B   — per-device label binding
└── master_falcon.bin         # 2191 B — гибридный master Falcon SK + PK
                              #          (framed bundle "OFAM")
```

Проверить через:

```bash
veil-cli identity show --veil-dir /var/lib/veil
# master_algo:        ed25519+falcon512
# (master_pubkey — 929 байт, подразумевается по master_algo)
```

## Шаг 2: сохранить ОБА артефакта

### 2a. Сид-фраза BIP-39 (на бумаге)

Фразу показывают один раз при создании и больше никогда. Запишите её
на бумагу, выбейте на металле или иначе сохраните без подключения к
сети:

```
1.  large       2.  decline     3.  palace      4.  grunt
5.  tired       6.  track       7.  tent        8.  sphere
9.  test        10. era         11. clinic      12. fortune
13. require     14. unfold      15. cluster     16. flat
17. robot       18. eagle       19. scale       20. step
21. decorate    22. banner      23. sausage     24. label
```

НЕ сохраняйте фразу в файл на работающей машине, если не шифруете её
дополнительно. (`--password-file` запишет за вас зашифрованный
`master.enc` — но тогда сам пароль становится единственным, что может
вас потопить. Выбирайте это, только если вам понятен такой
компромисс.)

### 2b. `master_falcon.bin` (цифровая копия)

Этот файл — **2191 байт** непрозрачных двоичных данных. Он начинается
с четырёх ASCII-байт `OFAM` — это «магический» маркер, по которому
инструменты сразу узнают формат файла. Проверьте его:

```bash
xxd /var/lib/veil/master_falcon.bin | head -1
# 00000000: 4f46 414d 0100 0005 01...
#           O F A M  ver 1   sk_len=1281 (0x501)
```

Как хранить:
- **Держите хотя бы 2 копии из 3** на независимых носителях —
  скажем, зашифрованная USB-флешка, вторая машина без подключения к
  сети (та, что никогда не выходит в сеть) и зашифрованное облачное
  хранилище. Достаточно, чтобы уцелели любые две.
- **Берегите его так же тщательно, как фразу.** Любой, у кого есть и
  этот файл, *и* фраза, получает полный постквантовый контроль над
  вашей личностью.
- **Время от времени проверяйте копии.** Убедитесь, что каждая ещё
  читается *и* ещё разбирается, — запустите `veil-cli identity show`
  на копии.

> Файл Falcon легко счесть «менее важным», чем фразу: ведь фраза
> сама по себе что-то да восстанавливает. **Не делайте так.**
> Гибридная личность, восстановленная без файла Falcon, в сети уже не
> та же самая — `node_id` другой, и каждая заявка на имя, контакт и
> запись маршрутизации, привязанные к старому `node_id`, остаются
> ни к чему не привязанными.

## Шаг 3: восстановить на свежей машине

После потери устройства, на чистой машине:

```bash
veil-cli identity restore \
    --algo hybrid \
    --phrase-file /path/to/recovered_phrase.txt \
    --master-falcon-file /path/to/recovered_master_falcon.bin \
    --label new-laptop \
    --veil-dir /var/lib/veil
```

Проверить, что восстановленный `node_id` совпадает с исходным:

```bash
veil-cli identity show --veil-dir /var/lib/veil
# node_id: <должно совпадать со значением, выданным `identity create`
#          на исходной машине>
# master_algo: ed25519+falcon512
```

Если при гибридном восстановлении пропустить `--master-falcon-file`,
CLI сразу же останавливается:

```
restore: --algo=hybrid requires --master-falcon-file pointing at the
preserved master_falcon.bin (the BIP-39 phrase alone cannot recover
the post-quantum half — see docs/identity-hybrid-backup.md)
```

Это сделано намеренно. Тихого запасного пути к Ed25519-only нет,
потому что он сменил бы ваш `node_id` без всякого предупреждения.

## Шаг 4: смена алгоритма (`identity migrate`)

Допустим, вы начали с классической личности Ed25519 и теперь хотите
перейти на гибридную (или наоборот). Это **миграция**, а не
восстановление, — используйте `veil-cli identity migrate`.

### 4a. Создать новую личность

На любой машине:

```bash
veil-cli identity create --algo hybrid \
    --veil-dir /var/lib/veil-new \
    --label new-master
```

Она печатает свежий `node_id`. Сохраните новую сид-фразу BIP-39 **и**
новый `master_falcon.bin` — с той же тщательностью, что и в шаге 2.

### 4b. Выпустить migration cert

Migration cert (сертификат миграции) — это небольшая подписанная
записка, которая говорит: «старая личность теперь живёт по новому
адресу». Выпустите её на машине, которая дотягивается И до СТАРОГО
veil_dir, И до СТАРЫХ мастер-секретов — это сид-фраза BIP-39 (или
пароль от `master.enc`) плюс СТАРЫЙ `master_falcon.bin`, если СТАРАЯ
личность была гибридной или только Falcon:

```bash
veil-cli identity migrate \
    --from /var/lib/veil-old \
    --to /var/lib/veil-new \
    --from-phrase-file /path/to/old_phrase.txt
```

(Добавьте `--from-master-falcon-file /path/to/old_master_falcon.bin`,
если СТАРАЯ личность была гибридной или отдельной Falcon.)

Вывод:

```
migration cert minted: 1024 bytes
  old_node_id:     <hex>  (algo=ed25519)
  new_node_id:     <hex>  (algo=ed25519+falcon512)
  issued_at_unix:  ...
  valid_until_unix:...   (окно 604800 с)
  cert written к: /var/lib/veil-new/migration_cert.bin
  dht_key:         <hex>

Next step: a running daemon serving --to will publish this cert на
its next maintenance tick.  Or run `node dht put <dht_key> <cert_path>`
against an admin socket для immediate propagation.
```

### 4c. Публикация

Два варианта:

1. **Через демон** (рекомендуется) — запустите демон, указав
   `--to /var/lib/veil-new`. На своём следующем такте перепубликации
   в DHT он подберёт `migration_cert.bin` и опубликует и новый
   IdentityDocument, И MigrationCert.
2. **Вручную** — `veil-cli node dht put <dht_key> <cert_path>`
   через работающий admin-сокет для немедленного распространения.

После публикации каждый, кто ищет вас, сам пройдёт по цепочке:
`resolve_identity_verified(old_node_id)` возвращает НОВУЮ личность.
У поиска есть встроенная защита от петель, слишком длинных цепочек и
понижений — всё это описано в
`crates/veil-identity/src/resolver.rs`.

### 4d. На слабый алгоритм перейти нельзя

Команда `migrate` — и вызов `migration::sign_migration_cert` под
ней — **отклоняет** любой переход, который опускает личность на более
слабый уровень безопасности:

| OLD → NEW | Статус |
|---|---|
| ed25519 → ed25519 | OK (только обновление) |
| ed25519 → falcon512 | OK (переход на постквантовый) |
| ed25519 → hybrid | OK (переход на постквантовый, рекомендовано) |
| falcon512 → hybrid | OK (вернуть путь BIP-39) |
| falcon512 → ed25519 | **REJECTED** (постквантовый → классика = понижение) |
| hybrid → ed25519 | **REJECTED** (теряем Falcon-компонент) |
| hybrid → falcon512 | **REJECTED** (теряем Ed25519-компонент) |

Уровни — от самого слабого к самому сильному:
`ed25519 (1) < falcon512 (2) < ed25519+falcon512 (3)`.

Попробуете понизить — и шаг подписи завершится ошибкой:

```
migrate: sign_migration_cert: security downgrade rejected
(old_algo=3, new_algo=1)
```

Это проверка по принципу «двойной страховки». Даже если кто-то
протащит сертификат мимо защиты поиска другим путём, сам `sign` всё
равно откажется его выпустить.

## Шаг 5: что делать, если файл Falcon потерян

Фраза уцелела, а `master_falcon.bin` пропал. У вас два варианта, и
ни один из них не безболезненный:

1. **Выпустить новую гибридную личность** (`identity create --algo
   hybrid`) и опубликовать MigrationCert со старого гибридного
   `node_id` на новый. Загвоздка: сертификат должен быть подписан
   *старым* мастер-ключом, а для этого нужен потерянный файл Falcon.
   Значит, этот путь сработает, только если вы потеряли файл недавно
   И до сих пор работает живой процесс, держащий мастер-ключ в
   памяти. Как только этот процесс завершится, цепочку уже не
   построить.

2. **Выпустить новую личность только на Ed25519** (`identity create`,
   без `--algo`), восстановив одну половину по фразе. Смиритесь с
   тем, что новый `node_id` будет другим — теперь это хеш 32 байт, а
   не 929, так что `BLAKE3(32 B) ≠ BLAKE3(929 B)`, — и заново
   отстройте ваше @name и контакты с нуля. Это путь «потеряно всё», и
   он остаётся единственным, когда материал Falcon утрачен
   безвозвратно.

Значит, одной фразы **недостаточно**, чтобы восстановить гибридную
личность. Относитесь к двум копиям как к единой паре: нужны обе.

## Краткая шпаргалка

| Нужно | Файл(ы) |
|---|---|
| Подцепить второе устройство под той же личностью | `master_falcon.bin` + сид-фраза BIP-39 |
| Восстановиться после потери устройства | `master_falcon.bin` + сид-фраза BIP-39 |
| Расшифровать `master.enc`, если при создании был указан `--password-file` | пароль (хранится отдельно) |
| Найти локальный `instance_id` для диагностики | `<veil_dir>/instance_id` |

## Приложение A: отдельный `--algo=falcon512`

Отдельный Falcon-512 (`--algo=falcon512`) поддерживается, но **не
рекомендуется для продакшена**. Прочитайте этот раздел, прежде чем за
него браться.

### Что это такое

Чистая постквантовая пара ключей и ничего больше. **Никакой**
классической половины Ed25519, **никакой** бумажной резервной копии
BIP-39 и **никакого** способа восстановления, кроме
`master_falcon.bin`.

```
node_id   = BLAKE3(falcon_pk(897 B))    // отличается от hybrid (BLAKE3(929 B))
master_pubkey = falcon_pk(897 B)
master_sk     = OsRng-выведенный Falcon-512 SK, живёт ТОЛЬКО в master_falcon.bin
```

### Зачем это может понадобиться

- Чисто постквантовое развёртывание, где нигде нет классического
  ключа: нет Ed25519 — значит, нет ничего, что будущий квантовый
  компьютер мог бы вскрыть, даже теоретически.
- Вы предпочитаете, чтобы сид-фразы BIP-39 не было вовсе. Фраза
  способна восстановить личность, но ровно поэтому она же и цель,
  до которой может добраться злоумышленник.
- Исследования или эксперименты с чисто постквантовыми личностями.

### Чем это опасно

В гибридном пути сид-фраза BIP-39 нужна **ради одного: чтобы был
бумажный запасной путь, к которому можно прибегнуть**. Уберите её — и
каждый из этих случаев становится роковым:

- Вы теряете `master_falcon.bin` — и это вся личность целиком, без
  второй копии, к которой можно было бы прибегнуть.
- Вы ошибаетесь в пути при создании копии — личность потеряна.
- Диск тихо отказывает посреди копирования, а вы не замечаете —
  личность потеряна.
- Кто-то крадёт устройство с `master_falcon.bin`, а оповещения вы так
  и не получаете — личность полностью скомпрометирована, и хода
  «перейти со старой фразы» уже не осталось.

### Обязательное подтверждение

CLI не станет выпускать отдельную личность Falcon-512, пока вы не
передадите `--accept-no-recovery` — этим вы подтверждаете, что
понимаете: пути назад не будет:

```bash
veil-cli identity create --algo falcon512 --label foo \
    --accept-no-recovery
```

Без этого флага команда завершается с ошибкой:

```
create: --algo=falcon512 has NO recovery path — the master Falcon
SK is generated from OsRng и lives ONLY in <veil_dir>/master_falcon.bin.
Loss of that file = TOTAL identity loss с no paper backup.  Pass
--accept-no-recovery to acknowledge, или use --algo=hybrid which
retains BIP-39-recoverable Ed25519 half.  See docs/identity-hybrid-backup.md.
```

### Что выводит `create` для Falcon-512

- Блок сид-фразы BIP-39 из 24 слов **не печатается** — здесь он
  ничего не восстанавливает, поэтому его показ только сбил бы вас с
  толку.
- В поток оператора печатается громкий блок `!!! WARNING ...`.
- `master_falcon.bin` создаётся в `<veil_dir>` и печатается с
  пометкой `(PRESERVE — operator-side recovery medium)`.

### Восстановление

```bash
veil-cli identity restore --algo falcon512 \
    --master-falcon-file /path/to/preserved_master_falcon.bin \
    --label new-host \
    --veil-dir /var/lib/veil
```

`--phrase-file` здесь **не требуется** (а если передадите, получите
громкое предупреждение — файл декодируется только для отлова опечаток,
его байты не используются). Связка воспроизводит `node_id` байт в
байт.

### Рекомендация по резервным копиям

Держите **3 копии из 3**. У отдельного Falcon-512 нет запаса на
ошибку: одна плохая копия — это уже на одну меньше, чем нужно. Для
каждой копии:

1. Проверьте магический заголовок `OFAM` файла (`xxd | head -1`).
2. Проверьте, что файл разбирается (`veil-cli identity show` против
   временной цели восстановления).
3. Обновляйте по заданному расписанию — как минимум раз в квартал.

Если на все три копии вы не готовы, **используйте `--algo=hybrid`.**
Классическая половина гибридного пути закрывает ровно те сценарии
отказа, что перечислены в этом разделе, и стоит вам всего лишь лишних
64 байт подписи на каждом сертификате.
