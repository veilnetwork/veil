# oproxy: прокси-мост для veil-сети

oproxy позволяет приложению на вашей машине выходить в интернет *через*
Veil. **Прокси** — это просто посредник: приложение передаёт ему запрос,
а прокси получает ответ за вас. Здесь прокси — не один сервер, а две
небольшие программы, соединённые сессией Veil, так что ваш трафик
проходит сеть «от двери до двери» и только потом выходит наружу.

Приложение говорит на одном из трёх привычных языков прокси —
**SOCKS5**, **HTTP** или **прозрачный режим** (о них — в разделе
[Входящие режимы](#входящие-режимы)). oproxy несёт этот трафик через
сессию Veil к удалённому **узлу выхода** (exit) — точке, где ваш трафик
покидает Veil и выходит в открытый интернет, — а оттуда к настоящему
сайту.

```
[локальное приложение]
    │  SOCKS5  /  HTTP CONNECT  /  transparent (TPROXY)
    ▼
[oproxy-client]  ◀── подключается к локальному veil-daemon через IPC
    │
    │  veil-сессия (E2E-шифрование)
    │
    ▼
[veil-daemon на сервере]
    │
    │  bound endpoint → роутится к oproxy-server
    ▼
[oproxy-server]
    │ TCP CONNECT к host:port
    ▼
[настоящий интернет]
```

Две половины — это `oproxy-client` (рядом с вашим приложением) и
`oproxy-server` (на узле выхода). Каждая общается со **своим локальным
демоном Veil** — фоновой программой, которая держит ваш узел в сети, —
на своём хосте. Прокси-трафик идёт через Veil между этими двумя
демонами, так что ни одна из программ не открывает соединение напрямую
через открытый интернет. English: [oproxy.md](../en/oproxy.md).

## Почему Veil, а не обычный HTTPS-прокси

Обычный прокси слушает публичный порт и доказывает, кто он, с помощью
TLS-сертификатов. Если пустить его через Veil, привычные хлопоты
отпадают:

- **Сквозное шифрование Veil.** Подменить собеседника нельзя, даже если
  взломан удостоверяющий центр (компания, которая ручается за
  подлинность сайта). Каждая личность — это собственная пара ключей,
  которой не владеет никто другой.
- **Никаких открытых портов на сервере.** Сервер вообще не слушает на
  публичном IP для прокси-трафика — всё идёт внутри туннеля Veil.
  Снаружи виден только сам демон Veil.
- **Свои имена приложений.** Несколько прокси-сервисов могут жить на
  одном демоне, не споря за порты: каждый получает свой `app_id`,
  вычисленный из его имени.
- **Список разрешённых `node_id` на сервере.** Вы сами решаете, каким
  собеседникам можно, а остальных разворачиваете прямо на уровне
  приложения — без правил межсетевого экрана, без сертификатов, без
  возни с их ротацией.

## Входящие режимы

**Входящий режим** — это просто то, каким способом локальное приложение
передаёт свой трафик в `oproxy-client`. Выберите тот, на котором
приложение уже умеет говорить:

- **SOCKS5** — широко поддерживаемый протокол прокси. Приложение
  говорит: «соедини меня с этим хостом и портом», и прокси это делает.
  Большинство браузеров и инструментов вроде `curl` понимают его
  сразу, без настройки.
- **HTTP** — режим прокси, встроенный в браузеры и в переменную
  окружения `HTTP_PROXY`. Удобен, когда приложение умеет обращаться
  только к HTTP-прокси.
- **Прозрачный режим** (TProxy) — вообще без настройки в каждом
  приложении. Вы направляете через oproxy целую машину или сеть, и он
  незаметно перехватывает трафик. Удобно как шлюз, но нужен Linux и
  пара правил межсетевого экрана (см. [ниже](#настройка-tproxy-на-linux--keenetic)).

| Режим   | Назначение                                  | Платформы |
|---------|---------------------------------------------|-----------|
| SOCKS5  | SOCKS-прокси в браузере, `curl --socks5`    | Все (Linux / macOS / Windows / FreeBSD / Keenetic) |
| HTTP    | HTTP/HTTPS-прокси в браузере (`HTTP_PROXY=...`)| Все |
| TProxy  | Прозрачный шлюз (`iptables -j TPROXY`)       | Linux / Keenetic |

На одном клиенте можно держать несколько входящих режимов сразу —
скажем, SOCKS5 и HTTP рядом, — каждый на своём порту.

## `oproxy-client`

Это половина, которая работает рядом с вашим приложением, — одна
самостоятельная программа. При запуске она подключается к локальному
демону Veil, занимает на нём точку подключения и слушает локально в
одном или нескольких входящих режимах из списка выше. Каждое
принятое соединение туннелируется к настроенному **вышестоящему**
серверу — то есть к следующему узлу на пути в интернет, который здесь
задаётся парой `(server_node_id, server_app_name)`.

```bash
oproxy-client --config /etc/oproxy/client.toml
```

### Создание конфига с нуля

```bash
# 1. Сгенерировать шаблон (пишет в stdout — перенаправьте в файл).
sudo mkdir -p /etc/oproxy
sudo oproxy-client --gen-config | sudo tee /etc/oproxy/client.toml >/dev/null
sudo chmod 0640 /etc/oproxy/client.toml
sudo chown root:veil /etc/oproxy/client.toml

# 2. Отредактируйте — минимум: server_node_id, server_app_name, [[inbound]]-listener'ы.
sudo vim /etc/oproxy/client.toml

# 3. Запустите ОТ ПОЛЬЗОВАТЕЛЯ DAEMON'А (не root). Daemon отбрасывает
#    любое IPC-подключение, чей peer uid != его собственного (audit U9,
#    без исключения для root), поэтому root-клиент молча отбрасывается
#    на app-сокете.
sudo -u veil oproxy-client --config /etc/oproxy/client.toml
```

> **Запускайте от пользователя демона, никогда не от root.** Демон Veil
> на уровне ядра проверяет, что у того, кто подключается к его
> IPC-сокету, тот же идентификатор пользователя (uid), что и у самого
> демона (`SO_PEERCRED` / `getpeereid`), и отбрасывает любое подключение
> от другого uid — включая root. Поэтому запускайте `oproxy-client` от
> того же пользователя, под которым работает демон (здесь `veil`).
> Прозрачному режиму по-прежнему нужно право `CAP_NET_ADMIN`; выдайте
> пользователю именно его (например, `setcap cap_net_admin+ep`), а не
> запускайте всё от root.

### Минимальный конфиг

```toml
socket_path     = "/var/lib/veil/app.sock"
server_node_id  = "00112233445566778899001122334455667788990011223344556677889900aa"
server_app_name = "my-proxy"

[[inbound]]
kind   = "socks5"
listen = "127.0.0.1:1080"
```

### Полный конфиг

```toml
socket_path     = "/var/lib/veil/app.sock"
server_node_id  = "<64-hex node_id хоста, где запущен oproxy-server>"
server_app_name = "my-proxy"

[[inbound]]
kind   = "socks5"
listen = "127.0.0.1:1080"

[[inbound]]
kind   = "http"
listen = "127.0.0.1:8080"

[[inbound]]
kind   = "tproxy"          # только Linux / Keenetic
listen = "0.0.0.0:12345"

# Опционально. Per-target маршрутизация — proxy / direct / block.
# По умолчанию: весь трафик через veil, без fallback (fail при недоступности veil).
[routing]
default  = "veil"   # veil | direct | block
fallback = "fail"      # глобальный default — может быть override'нут per-rule

[[routing.rules]]      # порядок имеет значение, первое совпадение выигрывает
host_suffix = ".internal"
action      = "direct"

[[routing.rules]]
cidr     = "10.0.0.0/8"
action   = "veil"
fallback = "direct"    # override: пробуем veil, при провале — direct

[[routing.rules]]
cidr     = "172.16.0.0/12"
action   = "veil"
fallback = "fail"      # override: пробуем veil, при провале — error

[[routing.rules]]
cidr   = "192.168.0.0/16"
action = "direct"      # никогда не через veil

# Опционально. Tokio-настройки — общая схема с veil-cli + ogate.
[runtime]
flavor               = "multi_thread"
worker_threads       = 4
max_blocking_threads = 64

# Опционально. Куда писать логи + уровень.
[logging]
level = "info"                       # off | error | warn | info | debug | trace
file  = "/var/log/oproxy-client.log" # опционально — по умолчанию stderr
```

## `oproxy-server`

Это половина на узле выхода — программа, которая собственно и выходит в
интернет за вас. Тоже одна самостоятельная программа. Она занимает точку
подключения приложения Veil через локальный демон, принимает входящие
потоки Veil от клиентов и для каждого открывает обычное TCP-соединение к
тому хосту и порту, которые запросил клиент, а дальше перегоняет байты
между ними.

```bash
oproxy-server --config /etc/oproxy/server.toml
```

### Создание конфига с нуля

```bash
# 1. Сгенерировать шаблон.
sudo mkdir -p /etc/oproxy
sudo oproxy-server --gen-config | sudo tee /etc/oproxy/server.toml >/dev/null
sudo chmod 0640 /etc/oproxy/server.toml
sudo chown root:veil /etc/oproxy/server.toml

# 2. Отредактируйте — минимум: app_name и либо список allowed_node_ids,
#    либо allow_all=true (явный opt-in на open-proxy).
sudo vim /etc/oproxy/server.toml

# 3. Запустите ОТ ПОЛЬЗОВАТЕЛЯ DAEMON'А (не root). Daemon отбрасывает
#    любое IPC-подключение, чей peer uid != его собственного (audit U9,
#    без исключения для root), поэтому root-сервер молча отбрасывается
#    на app-сокете.
sudo -u veil oproxy-server --config /etc/oproxy/server.toml
```

### Конфиг

```toml
socket_path = "/var/lib/veil/app.sock"
app_name    = "my-proxy"

# Опциональный allowlist по source node_id (hex). Пустой = open proxy.
allowed_node_ids = [
  "0011223344556677889900112233445566778899001122334455667788990011",
]

# При false (по умолчанию, рекомендуется) exit отказывает в outbound TCP к
# RFC1918 / loopback / multicast / link-local + cloud-metadata (169.254/16) /
# CGNAT (100.64/10) — включая их IPv4-mapped/-compatible IPv6-формы
# (`::ffff:a.b.c.d`, `::a.b.c.d`). Ставьте true только для намеренно
# LAN-обращённого exit.
allow_private = false

# Опционально. Общая схема runtime + logging.
[runtime]
flavor         = "multi_thread"
worker_threads = 2

[logging]
level = "info"
file  = "/var/log/oproxy-server.log"
```

Никакого общего секрета передавать не нужно. `app_id` вычисляется прямо
из `node_id` сервера и его `app_name`, и одни и те же исходные данные
всегда дают один и тот же результат. Клиент, который знает `node_id`
сервера и задаёт совпадающий `server_app_name`, сам выводит тот же
`app_id` и подключается ровно к этой точке подключения.

## Режимы маршрутизации (клиент)

По умолчанию каждое соединение идёт через Veil. Но так нужно не всегда:
до локальных адресов, например, быстрее достучаться напрямую.
**Маршрутизация** — это когда клиент для каждой цели решает, *каким
путём* пустить соединение. Задаются такие решения в необязательной
секции `[routing]` конфига.

Работает это как короткий список проверок. Для каждой пары
`(host, port)`, которую запросило приложение, клиент идёт по вашим
правилам `rules` сверху вниз и останавливается на первом подходящем, а
затем делает то, что велит его `action`. Если не подошло ни одно —
применяется глобальный `default`.

### Действия

С соединением можно сделать одно из трёх — это и задаёт правило (или
значение по умолчанию):

| Action    | Поведение |
|-----------|-----------|
| `veil` | Открыть поток Veil к `(server_node_id, server_app_name)` |
| `direct`  | Минуя Veil, соединиться по TCP напрямую с локального хоста |
| `block`   | Отказать ответом-ошибкой SOCKS5/HTTP |

Проще говоря: `veil` отправляет соединение через туннель, `direct`
соединяет напрямую с этой машины, минуя Veil, а `block` сразу
отказывает.

### Поля правила

Правило — это набор условий плюс действие. Все поля ниже, кроме
`action`, необязательны, и пропущенное поле совпадает с чем угодно.
Если правило перечисляет несколько условий, для совпадения должны
выполниться **все** — они объединяются по «И», а не по «ИЛИ».

| Поле | Совпадение |
|---|---|
| `host_suffix` | имя хоста заканчивается на эту строку (без учёта регистра); `.internal` совпадёт с `db.internal` |
| `host_exact`  | имя хоста в точности равно этому (без учёта регистра) |
| `cidr`        | хост записан как IPv4/IPv6-адрес И попадает в этот диапазон CIDR |
| `port_range`  | `"443"` (один порт) или `"1024-65535"` (диапазон, границы включительно) |
| `action`      | (обязательно) одно из `veil` / `direct` / `block` |
| `fallback`    | переопределение для этого правила; `direct` или `fail` |

**CIDR** — это способ записать сразу целый блок IP-адресов: например,
`10.0.0.0/8` охватывает все адреса, начинающиеся с `10.`.

**Примечание.** `cidr` срабатывает, только когда хост записан как
буквальный IP-адрес. Клиент никогда не ищет имена хостов в DNS — это
«телефонная книга» интернета, которая превращает имя вроде
`example.com` в IP, — поэтому правило `cidr` не поймает цель, заданную
по имени. Для правил по имени берите `host_suffix`.

### Что происходит, когда Veil недоступен (запасной путь)

**Запасной путь** (`fallback`) — это план Б на случай, когда действие
`veil` не может пройти: сервер недоступен, истекло время ожидания или
запрос отклонён. Он работает только для `action = "veil"`; у `direct`
и `block` отступать некуда.

| `fallback` | При сбое veil |
|---|---|
| `fail` | Вернуть приложению ошибку CONNECT (без попыток восстановления) |
| `direct` | Тихо соединиться по TCP напрямую с этой машины и связать стороны |

То есть `fail` сдаётся и сообщает об ошибке вашему приложению, а
`direct` молча пробует соединиться заново — уже прямо с этой машины.

Есть тонкость со временем. Запасной путь доступен, только пока Veil ещё
устанавливает соединение, — на первых трёх шагах рукопожатия (открыть
поток, отправить заголовок подключения, прочитать ответ со статусом).
Как только начинается четвёртый шаг и настоящие данные пошли в обе
стороны, соединение уже зафиксировано: повторить ничего нельзя, и любой
последующий сбой передаётся прямо вашему приложению.

`fallback`, записанный в правиле, переопределяет глобальный `[routing]
fallback` для этого правила. Если его не указать — правило берёт
глобальное значение.

### Пример: трафик локальной сети — напрямую, остальное — через Veil

«RFC1918» — это стандарт, который выделил приватные диапазоны адресов
для домашних и офисных сетей: `10.x`, `172.16–31.x` и `192.168.x`. В
этом примере с каждым из них поступают чуть по-разному, а всё остальное
идёт через Veil.

```toml
[routing]
default  = "veil"   # всё остальное через veil
fallback = "fail"      # глобальный default — fail при недоступности veil

[[routing.rules]]
cidr     = "10.0.0.0/8"
action   = "veil"
fallback = "direct"    # 10/8 — пробуем veil, при провале → direct

[[routing.rules]]
cidr     = "172.16.0.0/12"
action   = "veil"
fallback = "fail"      # 172.16/12 — пробуем veil, fail-closed

[[routing.rules]]
cidr   = "192.168.0.0/16"
action = "direct"      # 192.168/16 — никогда не через veil
```

Если прочитать это обратно, для каждой цели получается вот что:

| Цель            | путь    | при сбое veil       |
|-----------------|---------|---------------------|
| `10.x.x.x:*`    | veil | direct              |
| `172.16-31.x:*` | veil | fail                |
| `192.168.x.x:*` | direct  | n/a                 |
| всё остальное   | veil | fail (default)      |

## Настройка среды выполнения и логов

Эти две секции настраивают «движок» и журналы. Формат у них тот же, что
у `veil-cli` и `ogate`, поэтому настройки переносятся, если вы уже
настраивали те. Обе необязательны: пропустите их — и подставятся
разумные значения по умолчанию.

### `[runtime]`

```toml
[runtime]
flavor               = "multi_thread"   # | "current_thread"
worker_threads       = 4
max_blocking_threads = 64
thread_keep_alive_ms = 10000
thread_name          = "oproxy-client"
thread_stack_size    = 2097152
```

То же можно задать через окружение. Переменная окружения читается после
файла конфига и берёт верх над ним — удобно для разовой подмены
значения без правки файла:

| Переменная окружения | Что задаёт |
|---|---|
| `OPROXY_RUNTIME` | `current_thread` или `multi_thread` |
| `OPROXY_WORKERS` | число рабочих потоков |
| `OPROXY_MAX_BLOCKING_THREADS` | предел для пула блокирующих потоков |

### `[logging]`

```toml
[logging]
level = "info"                  # off | error | warn | info | debug | trace
file  = "/var/log/oproxy.log"   # опционально — по умолчанию stderr
```

| Поле | По умолчанию | Описание |
|---|---|---|
| `level` | `info` | Минимальный уровень. `off` полностью пропускает запуск логгера — журнала нет вообще, даже предупреждений и ошибок. |
| `file` | (stderr) | Необязательный путь к файлу. Записи **дописываются** в конец (файл создаётся, если его нет). Родительский каталог должен существовать. |

Переменная окружения `RUST_LOG` переопределяет `level`, когда для
одного запуска нужно больше подробностей:

```bash
RUST_LOG=oproxy=debug oproxy-client --config client.toml
```

## Настройка TProxy на Linux / Keenetic

Прозрачному режиму нужна помощь ядра: пара правил межсетевого экрана,
которые перенаправят проходящий трафик на слушатель oproxy. `iptables`
— это инструмент Linux для таких правил. Набор ниже помечает
TCP-трафик к порту 80 и отправляет его на слушатель на порту 12345:

```bash
# Маркировка + routing трафика на listener:
iptables -t mangle -A PREROUTING -p tcp \
    --dport 80 -j TPROXY --tproxy-mark 0x1/0x1 --on-port 12345
ip rule add fwmark 0x1 lookup 100
ip route add local 0.0.0.0/0 dev lo table 100

oproxy-client --config client.toml
```

После этого слушатель принимает соединения, нацеленные на любой адрес,
и узнаёт, куда каждое из них шло на самом деле, через `SO_ORIGINAL_DST`
(это возможность ядра, которая запоминает исходную цель после того, как
межсетевой экран перенаправил соединение). Для этого нужно право
`CAP_NET_ADMIN`.

## Протокол обмена (oproxy-client ↔ oproxy-server)

Этот раздел — для любопытных, для работы с oproxy он не нужен. Здесь
описано небольшое рукопожатие, на котором две половины общаются после
открытия потока Veil, до того как пойдут данные вашего приложения.
Клиент отправляет короткий заголовок с нужным хостом и портом, а сервер
отвечает однобайтовым статусом:

```text
client → server:   [host_len u16 BE][host UTF-8][port u16 BE]
server → client:   [status u8]
```

| Статус | Значение |
|---|---|
| `0x00` | Соединено; продолжаем перегонять байты |
| `0x01` | Отказано (`node_id` нет в списке разрешённых ИЛИ запрещённая цель) |
| `0x02` | Не удалось соединиться (ошибка DNS или TCP) |
| `0x03` | Неверный запрос (испорченный заголовок) |

Если статус не `0x00`, сервер отправляет его и закрывает поток.
Именно от этого статуса зависит решение клиента о запасном пути:
«Отказано», «Не удалось соединиться» и «Неверный запрос» — все
считаются поправимыми сбоями на первых трёх шагах, так что запасной
путь `direct` ещё может вступить в дело.

## Известные ограничения

Чего oproxy пока не умеет — стоит знать заранее, прежде чем на это
полагаться:

- **Только TCP.** Передачи по UDP нет, поэтому всё, что опирается на UDP
  (некоторые игры и видеозвонки), не пройдёт.
- **Клиент не разрешает имена.** Имена хостов он передаёт серверу как
  есть, а тот сам ищет их в DNS. Поэтому правило `cidr` никогда не
  поймает цель, заданную по имени, — для таких используйте
  `host_suffix`.
- **Нет прозрачного режима на FreeBSD.** TProxy там пока заглушка: при
  запуске он завершается с ошибкой. На FreeBSD используйте SOCKS5.
